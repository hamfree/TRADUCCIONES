Contenedores Seguros

En Java EE, los contenedores de componentes son responsables de proporcionar seguridad a la aplicación. Un contenedor proporciona dos tipos de seguridad: declarativa y programática.

Aquí se tratan los siguientes temas:

  • Uso de Anotaciones para Especificar Información de Seguridad

  • Uso de Descriptores de Implementación para Seguridad Declarativa

  • Uso de Seguridad Programática

Uso de Anotaciones para Especificar Información de Seguridad

Las anotaciones permiten un estilo declarativo de programación y, por lo tanto, abarcan los conceptos de seguridad tanto declarativos como programáticos. Los usuarios pueden especificar información sobre seguridad dentro de un archivo de clase mediante el uso de anotaciones. El servidor GlassFish utiliza esta información cuando se implementa la aplicación. Sin embargo, no toda la información de seguridad se puede especificar mediante anotaciones. Se debe especificar cierta información en los descriptores de implementación de la aplicación.

Las anotaciones específicas que se pueden usar para especificar información de seguridad dentro de un archivo de clase de enterprise bean se describen en Protección de un Enterprise Bean mediante Seguridad Declarativa . El Capítulo 51, "Introducción a la Protección de Aplicaciones Web" , describe cómo usar anotaciones para proteger las aplicaciones web cuando sea posible. Los descriptores de implementación se describen solo cuando es necesario.

Para obtener más información sobre las anotaciones, consulte Más Información sobre Seguridad.

Uso de Descriptores de Implementación para Seguridad Declarativa

La seguridad declarativa puede expresar los requisitos de seguridad de un componente de aplicación mediante el uso de descriptores de implementación. Debido a que la información del descriptor de implementación es declarativa, se puede cambiar sin necesidad de modificar el código fuente. En tiempo de ejecución, el servidor Java EE lee el descriptor de implementación y actúa sobre la aplicación, el módulo o el componente correspondiente en consecuencia. Los descriptores de implementación deben proporcionar cierta información estructural para cada componente si esta información no se ha proporcionado en las anotaciones o no se va a establecer por defecto.

Esta parte del tutorial no documenta cómo crear descriptores de implementación; describe solo los elementos del descriptor de implementación relevantes para la seguridad. El EID de NetBeans proporciona herramientas para crear y modificar descriptores de implementación.

Los diferentes tipos de componentes usan diferentes formatos o esquemas para sus descriptores de implementación. Los elementos de seguridad de los descriptores de implementación analizados en este tutorial incluyen lo siguiente.

  • Los componentes web pueden usar un descriptor de implementación de aplicaciones web denominado web.xml.

    El esquema para los descriptores de implementación de componentes web se proporciona en el Capítulo 14 de la especificación Java Servlet 4.0 (JSR 369), que se puede descargar desde http://jcp.org/en/jsr/detail?id=369 .

  • Los componentes de Enterprise JavaBeans pueden utilizar un descriptor de despliegue EJB denominado META-INF/ejb-jar.xml, contenido en el archivo EJB JAR.

    El esquema para los descriptores de implementación de beans empresariales se proporciona en el Capítulo 14 de la Especificación de requisitos y contratos básicos de EJB 3.2 (JSR 345), que se puede descargar desde http://jcp.org/en/jsr/detail?id=345 .

Uso de Seguridad Programática

La seguridad programática está integrada en una aplicación y se utiliza para tomar decisiones de seguridad. La seguridad programática es útil cuando la seguridad declarativa por sí sola no es suficiente para expresar el modelo de seguridad de una aplicación. La IPA para seguridad programática consta de métodos de la interfaz SecurityContext de la IPA de seguridad de Java EE, y métodos de la interfaz EJB EJBContext y la interfaz de servlet HttpServletRequest. Estos métodos permiten que los componentes tomen decisiones de lógica de negocio basadas en el rol de seguridad de la persona que llama o del usuario remoto.
