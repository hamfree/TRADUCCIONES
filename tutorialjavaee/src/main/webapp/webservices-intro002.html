Tipos de Servicios Web

En el nivel conceptual, un servicio es un componente de software proporcionado a través de un punto final accesible a la red. El consumidor y el proveedor del servicio utilizan mensajes para intercambiar información de solicitud y respuesta de invocación en forma de documentos independientes que hacen muy pocas suposiciones sobre las capacidades tecnológicas del receptor.

A nivel técnico, los servicios web se pueden implementar de varias maneras. Los dos tipos de servicios web discutidos en esta sección se pueden distinguir como servicios web "grandes" y servicios web "RESTful".

Aquí se tratan los siguientes temas:

  • "Grandes" Servicios Web

  • Servicios Web RESTful

"Grandes" Servicios Web

JAX-WS proporciona la funcionalidad para servicios web "grandes", que se describen en el Capítulo 31, "Construyendo Servicios Web con JAX-WS". Los grandes servicios web utilizan mensajes XML que siguen el estándar del Protocolo Simple de Acceso a Objetos (SOAP, en inglés), un lenguaje XML que define una arquitectura de mensajes y formatos de mensajes. Dichos sistemas a menudo contienen una descripción legible por máquina de las operaciones ofrecidas por el servicio, escrita en el Lenguaje de Descripción de Servicios Web (WSDL, en inglés), un lenguaje XML para definir interfaces sintácticamente.

Nota: JAX-WS se trasladó a Java SE 8 y no se incluye en la plataforma Java EE 8. Sin embargo, la API JAX-WS todavía se puede usar en aplicaciones EE.

El formato de mensaje SOAP y el lenguaje de definición de interfaz WSDL han obtenido una adopción generalizada. Muchas herramientas de desarrollo, como EID de NetBeans, pueden reducir la complejidad del desarrollo de aplicaciones de servicios web.

Un diseño basado en SOAP debe incluir los siguientes elementos.

  • Se debe establecer un contrato formal para describir la interfaz que ofrece el servicio web. WSDL se puede utilizar para describir los detalles del contrato, que pueden incluir mensajes, operaciones, enlaces y la ubicación del servicio web. También puede procesar mensajes SOAP en un servicio JAX-WS sin publicar un WSDL.

  • La arquitectura debe abordar requisitos no funcionales complejos. Muchas especificaciones de servicios web abordan dichos requisitos y establecen un vocabulario común para ellos. Los ejemplos incluyen transacciones, seguridad, direccionamiento, confianza, coordinación, etc.

  • La arquitectura necesita manejar el procesamiento asíncrono y la invocación. En tales casos, la infraestructura proporcionada por los estándares, como Web Services Reliable Messaging (WSRM) y las IPA, como JAX-WS, con su soporte de invocación asíncrona del lado del cliente, se pueden aprovechar de forma inmediata.

Servicios Web RESTful

En Java EE, JAX-RS proporciona la funcionalidad para servicios web de transferencia de estado representacional (RESTful). REST es ideal para escenarios de integración ad hoc básicos. Los servicios web RESTful, a menudo mejor integrados con HTTP que los servicios basados en SOAP, no requieren mensajes XML ni definiciones de API de servicio WSDL.

Project Jersey es la implementación de referencia lista para producción para la especificación JAX-RS. Jersey implementa la compatibilidad con las anotaciones definidas en la especificación JAX-RS, lo que facilita a los desarrolladores la creación de servicios web RESTful con Java y Java Virtual Machine (JVM).

Debido a que los servicios web RESTful utilizan estándares bien conocidos del W3C y del Grupo de trabajo de ingeniería de Internet (IETF) (HTTP, XML, URI, MIME) y tienen una infraestructura liviana que permite construir servicios con herramientas mínimas, el desarrollo de servicios web RESTful es económico y por lo tanto, tiene una barrera muy baja para la adopción. Puede utilizar una herramienta de desarrollo como el EID NetBeans para reducir aún más la complejidad del desarrollo de servicios web RESTful.

Un diseño RESTful puede ser apropiado cuando se cumplen las siguientes condiciones.

  • Los servicios web son completamente sin estado. Una buena prueba es considerar si la interacción puede sobrevivir a un reinicio del servidor.

  • Una infraestructura de almacenamiento en caché se puede aprovechar para el rendimiento. Si los datos que devuelve el servicio web no se generan dinámicamente y se pueden almacenar en caché, la infraestructura de almacenamiento en caché que proporcionan los servidores web y otros intermediarios de forma inherente se puede aprovechar para mejorar el rendimiento. Sin embargo, el desarrollador debe tener cuidado porque tales cachés están limitadas al método HTTP GET para la mayoría de los servidores.

  • El productor del servicio y el consumidor del servicio tienen una comprensión mutua del contexto y el contenido que se transmite. Debido a que no existe una forma formal de describir la interfaz de servicios web, ambas partes deben acordar fuera de banda los esquemas que describen los datos que se intercambian y las formas de procesarlos de manera significativa. En el mundo real, la mayoría de las aplicaciones comerciales que exponen los servicios como implementaciones RESTful también distribuyen los llamados kits de herramientas de valor agregado que describen las interfaces para los desarrolladores en lenguajes de programación populares.

  • El ancho de banda es especialmente importante y debe limitarse. REST es particularmente útil para dispositivos de perfil limitado, como PDA y teléfonos móviles, para los cuales se debe restringir la sobrecarga de encabezados y capas adicionales de elementos SOAP en la carga XML.

  • La entrega o agregación de servicios web en sitios web existentes se puede habilitar fácilmente con un estilo RESTful. Los desarrolladores pueden usar tecnologías como JAX-RS y JavaScript asincrónico con XML (Ajax) y kits de herramientas como Direct Web Remoting (DWR) para consumir los servicios en sus aplicaciones web. En lugar de comenzar desde cero, los servicios pueden exponerse con XML y consumirse mediante páginas HTML sin refactorizar significativamente la arquitectura del sitio web existente. Los desarrolladores existentes serán más productivos porque están agregando algo con lo que ya están familiarizados en lugar de tener que empezar desde cero con la nueva tecnología.

Los servicios web RESTful se analizan en el Capítulo 32, "Construyendo Servicios Web RESTful con JAX-RS". Este capítulo contiene información sobre cómo generar el esqueleto de un servicio web RESTful utilizando el EID NetBeans y la herramienta de administración de proyectos Maven.
