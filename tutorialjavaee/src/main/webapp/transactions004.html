Transacciones Gestionadas por Contenedores

En un enterprise bean con demarcación de transacciones administradas por contenedor, el contenedor EJB establece los límites de las transacciones. Puede utilizar transacciones gestionadas por contenedor con cualquier tipo de enterprise bean: sesión o mensaje. Las transacciones administradas por contenedores simplifican el desarrollo porque el código del enterprise bean no marca explícitamente los límites de la transacción. El código no incluye declaraciones que comiencen y finalicen la transacción. De forma predeterminada, si no se especifica ninguna demarcación de transacciones, los enterprise beans utilizan la demarcación de transacciones administrada por contenedor.

Normalmente, el contenedor inicia una transacción inmediatamente antes de que se inicie un método de enterprise bean y confirma la transacción justo antes de que finalice el método. Cada método se puede asociar con una sola transacción. No se permiten transacciones anidadas o múltiples dentro de un método.

Las transacciones administradas por contenedor no requieren que todos los métodos estén asociados con transacciones. Al desarrollar un bean, puede configurar los atributos de transacción para especificar cuáles de los métodos del bean están asociados con las transacciones.

Los enterprise beans que utilizan la demarcación de transacciones gestionadas por contenedor no deben utilizar ningún método de gestión de transacciones que interfiera con los límites de demarcación de transacciones del contenedor. Ejemplos de tales métodos son los métodos commit, setAutoCommit y rollback de java.sql.Connection o métodos commit y rollback de javax.jms.Session. Si necesita control sobre la demarcación de transacciones, debe usar la demarcación de transacciones administrada por la aplicación.

Los enterprise beans que utilizan la demarcación de transacciones gestionadas por contenedores tampoco deben utilizar la interfaz javax.transaction.UserTransaction.

Atributos de Transacción

Un atributo de transacción controla el alcance de una transacción. La Figura 54-1 ilustra por qué es importante controlar el alcance. En el diagrama, method-A comienza una transacción y luego invoca method-B de Bean-2. Cuando se ejecuta method-B, ¿se ejecuta dentro del alcance de la transacción iniciada por method-A o se ejecuta con una nueva transacción? La respuesta depende del atributo de transacción de method-B.

Figura 54-1 Alcance de la transacción

Un atributo de transacción puede tener uno de los siguientes valores:

  • Required

  • RequiresNew

  • Mandatory

  • NotSupported

  • Supports

  • Never

Atributo Required

Si el cliente se ejecuta dentro de una transacción e invoca el método del enterprise bean, el método se ejecuta dentro de la transacción del cliente. Si el cliente no está asociado con una transacción, el contenedor inicia una nueva transacción antes de ejecutar el método.

El atributo Required es el atributo de transacción implícito para todos los métodos de enterprise beans que se ejecutan con demarcación de transacción administrada por contenedor. Por lo general, no establece el atributo Required a menos que necesite anular otro atributo de transacción. Debido a que los atributos de transacción son declarativos, puede cambiarlos fácilmente más adelante.

Atributo RequiresNew

Si el cliente se ejecuta dentro de una transacción e invoca el método del enterprise bean, el contenedor realiza los siguientes pasos:

  1. Suspende la transacción del cliente

  2. Comienza una nueva transacción

  3. Delega la llamada al método.

  4. Reanuda la transacción del cliente después de que se completa el método

Si el cliente no está asociado con una transacción, el contenedor inicia una nueva transacción antes de ejecutar el método.

Debe usar el atributo RequiresNew cuando desee asegurarse de que el método siempre se ejecute dentro de una nueva transacción.

Atributo Mandatory

Si el cliente se ejecuta dentro de una transacción e invoca el método del enterprise bean, el método se ejecuta dentro de la transacción del cliente. Si el cliente no está asociado con una transacción, el contenedor lanza una TransactionRequiredException.

Utilice el atributo Mandatory si el método del enterprise bean debe utilizar la transacción del cliente.

Atributo NotSupported

Si el cliente se está ejecutando dentro de una transacción e invoca el método del enterprise bean, el contenedor suspende la transacción del cliente antes de invocar el método. Una vez que se ha completado el método, el contenedor reanuda la transacción del cliente.

Si el cliente no está asociado con una transacción, el contenedor no inicia una nueva transacción antes de ejecutar el método.

Utilice el atributo NotSupported para los métodos que no necesitan transacciones. Debido a que las transacciones implican sobrecargas, este atributo puede mejorar el rendimiento.

Atributo Supports

Si el cliente se ejecuta dentro de una transacción e invoca el método del enterprise bean, el método se ejecuta dentro de la transacción del cliente. Si el cliente no está asociado con una transacción, el contenedor no inicia una nueva transacción antes de ejecutar el método.

Debido a que el comportamiento transaccional del método puede variar, debe usar el atributo Supports con precaución.

Atributo Never

Si el cliente se ejecuta dentro de una transacción e invoca el método del enterprise bean, el contenedor genera una RemoteException. Si el cliente no está asociado con una transacción, el contenedor no inicia una nueva transacción antes de ejecutar el método.

Configuración de Atributos de Transacción

Los atributos de transacción se especifican decorando la clase o método del enterprise bean con una anotación javax.ejb.TransactionAttribute y estableciéndola en una de las constantes javax.ejb.TransactionAttributeType.

Si decora la clase de enterprise bean con @TransactionAttribute, el TransactionAttributeType especificado se aplica a todos los métodos de negocio de la clase. Decorar un método de negocio con @TransactionAttribute aplica el TransactionAttributeType solo a ese método. Si una anotación @TransactionAttribute decora tanto la clase como el método, el método TransactionAttributeType anula la clase TransactionAttributeType.

Las constantes TransactionAttributeType que se muestran en la Tabla 54-2 encapsulan los atributos de transacción descritos anteriormente en esta sección.

El siguiente fragmento de código demuestra cómo usar la anotación @TransactionAttribute: 

                  
@TransactionAttribute(NOT_SUPPORTED)
@Stateful
public class TransactionBean implements Transaction {
...
    @TransactionAttribute(REQUIRES_NEW)
    public void firstMethod() {...}
    @TransactionAttribute(REQUIRED)
    public void secondMethod() {...}
    public void thirdMethod() {...}
    public void fourthMethod() {...}
}

En este ejemplo, el atributo de transacción de la clase TransactionBean se ha establecido en NotSupported, firstMethod se ha establecido en RequiresNew, y secondMethod se ha establecido en Required. Debido a que un @TransactionAttribute establecido en un método anula la clase @TransactionAttribute, las llamadas a firstMethod crearán una nueva transacción y las llamadas a secondMethod se ejecutará en la transacción actual o iniciará una nueva transacción. Las llamadas a thirdMethod o fourthMethod no tienen lugar dentro de una transacción.

Revertir una Transacción Gestionada por Contenedor

Hay dos formas de revertir una transacción administrada por contenedor. Primero, si se lanza una excepción del sistema, el contenedor revertirá automáticamente la transacción. En segundo lugar, al invocar el método setRollbackOnly de la interfaz EJBContext, el método bean instruye al contenedor para revertir la transacción. Si el bean arroja una excepción de aplicación, la reversión no es automática, pero puede iniciarse mediante una llamada a setRollbackOnly.

Sincronización de las Variables de Instancia de un Bean de Sesión

La interfaz SessionSynchronization, que es opcional, permite que las instancias de beans de sesión con estado reciban notificaciones de sincronización de transacciones. Por ejemplo, podría sincronizar las variables de instancia de un enterprise bean con sus valores correspondientes en la base de datos. El contenedor invoca los métodos SessionSynchronization (afterBegin, beforeCompletion y afterCompletion) en cada una de las etapas principales de un transacción.

El método afterBegin informa a la instancia que ha comenzado una nueva transacción. El contenedor invoca afterBegin inmediatamente antes de invocar el método de negocio.

El contenedor invoca el método beforeCompletion después de que el método de negocio haya finalizado, pero justo antes de que se confirme la transacción. El método beforeCompletion es la última oportunidad para que el bean de sesión revierta la transacción (llamando a setRollbackOnly).

El método afterCompletion indica que la transacción se ha completado. Este método tiene un único parámetro boolean cuyo valor es true si la transacción se confirmó y false si se revirtió.

Métodos no Permitidos en Transacciones Gestionadas por Contenedor

No debe invocar ningún método que pueda interferir con los límites de transacción establecidos por el contenedor. Los siguientes métodos están prohibidos:

  • Los métodos commit, setAutoCommit y rollback de java.sql.Connection

  • El método getUserTransaction de javax.ejb.EJBContext

  • Cualquier método de javax.transaction.UserTransaction

Sin embargo, puede usar estos métodos para establecer límites en las transacciones administradas por la aplicación.
