Mecanismos de Seguridad

Las características de una aplicación deben tenerse en cuenta al decidir la capa y el tipo de seguridad que se proporcionará a las aplicaciones. Las siguientes secciones analizan las características de los mecanismos comunes que se pueden utilizar para proteger las aplicaciones Java EE. Cada uno de estos mecanismos se puede usar individualmente o con otros para proporcionar capas de protección basadas en las necesidades específicas de su implementación.

Aquí se tratan los siguientes temas:

  • Mecanismos de Seguridad de Java SE

  • Mecanismos de Seguridad de Java EE

Mecanismos de Seguridad de Java SE

Java SE proporciona soporte para una variedad de características y mecanismos de seguridad.

  • El Servicio de Autenticación y Autorización de Java (JAAS, en inglés) es un conjunto de IPA que permite que los servicios autentiquen y apliquen controles de acceso a los usuarios. JAAS proporciona un marco conectable y extensible para la autenticación y autorización de usuarios mediante programación. JAAS es una IPA central de Java SE y es una tecnología subyacente para los mecanismos de seguridad de Java EE.

  • Los Servicios de Seguridad Genéricos de Java (Java GSS-API, en inglés) es una IPA basada en token que se utiliza para intercambiar mensajes de forma segura entre aplicaciones que se comunican. GSS-API ofrece a los programadores de aplicaciones un acceso uniforme a los servicios de seguridad sobre una variedad de mecanismos de seguridad subyacentes, incluido Kerberos.

  • La Extensión de Criptografía de Java (JCE, en inglés) proporciona un marco e implementaciones para algoritmos de cifrado, generación de claves y acuerdo de claves, y Código de Autenticación de Mensajes (MAC, en inglés). La compatibilidad con el cifrado incluye cifrados simétricos, asimétricos, de bloque y de flujo. Los cifrados en bloque operan en grupos de bytes; Los cifrados de flujo operan en un byte a la vez. El software también admite transmisiones seguras y objetos sellados.

  • Extensión de Sockets Seguros de Java (JSSE, en inglés) proporciona un marco y una implementación para una versión de Java de los protocolos Capa de Sockets Seguros (SSL, en inglés) y Capa de Transporte Segura (TLS, en inglés) e incluye funcionalidad para el cifrado de datos, autenticación de servidor, integridad de mensajes y autenticación de cliente opcional para permitir comunicaciones seguras por Internet.

  • Capa de Seguridad y Autenticación Simple (SASL, en inglés) es un estándar de Internet (RFC 2222) que especifica un protocolo para la autenticación y el establecimiento opcional de una capa de seguridad entre las aplicaciones cliente y servidor. SASL define cómo se van a intercambiar los datos de autenticación, pero no especifica el contenido de esos datos. SASL es un marco en el que pueden encajar mecanismos de autenticación específicos que especifican el contenido y la semántica de los datos de autenticación.

Java SE también proporciona un conjunto de herramientas para administrar almacenes de claves, certificados y archivos de políticas; generar y verificar firmas JAR; y obtener, listar y administrar tickets de Kerberos.

Para obtener más información sobre la seguridad de Java SE, visite http://docs.oracle.com/javase/7/docs/technotes/guides/security/.

Mecanismos de Seguridad de Java EE

Los servicios de seguridad de Java EE los proporciona el contenedor de componentes y se pueden implementar mediante técnicas declarativas o programáticas (consulte Contenedores Seguros). Los servicios de seguridad de Java EE proporcionan un mecanismo de seguridad sólido y fácil de configurar para autenticar a los usuarios y autorizar el acceso a las funciones de la aplicación y los datos asociados en muchas capas diferentes. Los servicios de seguridad de Java EE están separados de los mecanismos de seguridad del sistema operativo.

Seguridad de la Capa de Aplicación

En Java EE, los contenedores de componentes son responsables de proporcionar seguridad en la capa de aplicación, servicios de seguridad para un tipo de aplicación específico adaptado a las necesidades de la aplicación. En la capa de aplicación, los cortafuegos de aplicaciones se pueden utilizar para mejorar la protección de las aplicaciones al proteger el flujo de comunicación y todos los recursos de aplicaciones asociados de los ataques.

La seguridad de Java EE es fácil de implementar y configurar y puede ofrecer un control de acceso detallado a las funciones y datos de la aplicación. Sin embargo, como es inherente a la seguridad aplicada en la capa de aplicación, las propiedades de seguridad no son transferibles a las aplicaciones que se ejecutan en otros entornos y protegen los datos solo mientras residen en el entorno de la aplicación. En el contexto de una aplicación empresarial tradicional, esto no es necesariamente un problema, pero cuando se aplica a una aplicación de servicios web, en la que los datos a menudo viajan a través de varios intermediarios, es necesario utilizar los mecanismos de seguridad de Java EE junto con la seguridad de la capa de transporte y la seguridad de capa de mensaje para una solución de seguridad completa.

Las ventajas de usar la seguridad de la capa de aplicación incluyen lo siguiente.

  • La seguridad se adapta de manera única a las necesidades de la aplicación.

  • La seguridad es detallada, con configuraciones específicas de la aplicación.

Las desventajas de usar la seguridad de la capa de aplicación incluyen lo siguiente.

  • La aplicación depende de atributos de seguridad que no son transferibles entre tipos de aplicaciones.

  • La compatibilidad con varios protocolos hace que este tipo de seguridad sea vulnerable.

  • Los datos están cerca o contenidos dentro del punto de vulnerabilidad.

Para obtener más información sobre cómo proporcionar seguridad en la capa de aplicación, consulte Contenedores Seguros.

Seguridad de la Capa de Transporte

La seguridad de la capa de transporte la proporcionan los mecanismos de transporte utilizados para transmitir información por cable entre clientes y proveedores; por lo tanto, la seguridad de la capa de transporte se basa en el transporte HTTP seguro (HTTPS) mediante la capa de sockets seguros (SSL). La seguridad del transporte es un mecanismo de seguridad de punto a punto que se puede utilizar para la autenticación, la integridad de los mensajes y la confidencialidad. Cuando se ejecuta una sesión protegida por SSL, el servidor y el cliente pueden autenticarse mutuamente y negociar un algoritmo de cifrado y claves criptográficas antes de que el protocolo de la aplicación transmita o reciba su primer byte de datos. La seguridad está activa desde que los datos salen del cliente hasta que llegan a su destino, o viceversa, incluso a través de intermediarios. El problema es que los datos no están protegidos una vez que llegan al destino. Una solución es cifrar el mensaje antes de enviarlo.

La seguridad de la capa de transporte se realiza en una serie de fases, como se indica a continuación.

  • El cliente y el servidor acuerdan un algoritmo apropiado.

  • Se intercambia una clave mediante el cifrado de clave pública y la autenticación basada en certificados.

  • Se utiliza un cifrado simétrico durante el intercambio de información.

Los certificados digitales son necesarios cuando se ejecuta HTTPS mediante SSL. El servicio HTTPS de la mayoría de los servidores web no se ejecutará a menos que se haya instalado un certificado digital. Ya se han creado certificados digitales para el servidor GlassFish.

Las ventajas de utilizar la seguridad de la capa de transporte incluyen las siguientes.

  • Es una tecnología estándar relativamente simple y bien entendida.

  • Se aplica tanto al cuerpo del mensaje como a sus archivos adjuntos.

Las desventajas de utilizar la seguridad de la capa de transporte incluyen las siguientes.

  • Está estrechamente acoplado con el protocolo de la capa de transporte.

  • Representa un enfoque de todo o nada para la seguridad. Esto implica que el mecanismo de seguridad desconoce el contenido del mensaje, por lo que no puede aplicar seguridad de manera selectiva a partes del mensaje como puede hacerlo con la seguridad de la capa del mensaje.

  • La protección es transitoria. El mensaje está protegido solo mientras está en tránsito. El punto final elimina automáticamente la protección cuando recibe el mensaje.

  • No es una solución de extremo a extremo, simplemente punto a punto.

Para obtener más información sobre la seguridad de la capa de transporte, consulte Establecimiento de una conexión segura mediante SSL .

Seguridad de la Capa de Mensaje

En la seguridad de la capa de mensaje, la información de seguridad está contenida dentro del mensaje SOAP y/o el archivo adjunto del mensaje SOAP, lo que permite que la información de seguridad viaje junto con el mensaje o el archivo adjunto. Por ejemplo, una parte del mensaje puede estar firmada por un remitente y cifrada para un receptor en particular. Cuando se envía desde el remitente inicial, el mensaje puede pasar a través de nodos intermedios antes de llegar a su destinatario. En este escenario, las partes cifradas continúan siendo opacas para los nodos intermedios y solo el receptor previsto puede descifrarlas. Por esta razón, la seguridad de la capa de mensaje también se denomina a veces seguridad de extremo a extremo.

Las ventajas de la seguridad de la capa de mensaje incluyen lo siguiente.

  • La seguridad permanece con el mensaje en todos los saltos y después de que el mensaje llega a su destino.

  • La seguridad se puede aplicar de forma selectiva a diferentes partes de un mensaje y, si se utiliza la seguridad de servicios web XML, a los archivos adjuntos.

  • La seguridad de mensajes se puede utilizar con intermediarios en múltiples saltos.

  • La seguridad de los mensajes es independiente del entorno de la aplicación o del protocolo de transporte.

La desventaja de usar la seguridad de la capa de mensajes es que es relativamente compleja y agrega algunas sobrecargas al procesamiento.

El servidor GlassFish admite la seguridad de los mensajes mediante Metro, una pila de servicios web que utiliza Seguridad de Servicios Web (WSS, en inglés) para proteger los mensajes. Debido a que la seguridad de este mensaje es específica de Metro y no forma parte de la plataforma Java EE, este tutorial no analiza el uso de WSS para proteger los mensajes. Consulte la Guía del usuario de Metro en https://javaee.github.io/metro/.
