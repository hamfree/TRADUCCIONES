Consideraciones de Seguridad de los WebSocket

Si el WebSocket se declara en una página que está restringida a usuarios registrados solo con un rol específico, es posible que desee agregar la URL de solicitud de protocolo de enlace push al conjunto de URL restringidas.

La URL de solicitud de protocolo de enlace push se compone del prefijo URI, /javax.faces.push/, seguido del nombre del canal. En el ejemplo de seguridad administrada por contenedor, que ya ha restringido una página de ejemplo, /user/foo.xhtml, a usuarios registrados con el rol de ejemplo, USER, en el patrón de URL de ejemplo, /user/*, en web.xml, consulte a continuación: 

              
<security-constraint>
  <web-resource-collection>
    <web-resource-name>Restrict access to role USER.
    </web-resource-name>
    <url-pattern>/user/*</url-pattern>
  </web-resource-collection>
  <auth-constraint>
    <role-name>USER</role-name>
  </auth-constraint>
</security-constraint>

Si la página, /user/foo.xhtml, contiene <f:websocket channel="foo">, entonces debe agregar una restricción en el patrón de URL de la solicitud de protocolo de enlace push de /javax.faces.push/foo, como se muestra a continuación: 

              
<security-constraint>
  <web-resource-collection>
    <web-resource-name>Restrict access to role USER.
    </web-resource-name>
    <url-pattern>/user/*</url-pattern>
    <url-pattern>/javax.faces.push/foo</url-pattern>
  </web-resource-collection>
  <auth-constraint>
    <role-name>USER</role-name>
  </auth-constraint>
</security-constraint>

Como seguridad adicional, particularmente para aquellos canales públicos que no pueden ser restringidos por restricciones de seguridad, la etiqueta f:websocket registrará todos los canales declarados previamente en la sesión HTTP actual, y se verificará cualquier solicitud de apertura de WebSocket entrante. para saber si coincide con estos canales en la sesión HTTP actual. Si el canal es desconocido (por ejemplo, adivinado o suplantado al azar por los usuarios finales o reconectado manualmente después de que caduque la sesión), el WebSocket se cerrará inmediatamente con el código de motivo de cierre, CloseCodes.VIOLATED_POLICY (1008). Además, cuando se destruye la sesión HTTP, todas las sesiones y los canales de ámbito de vista que todavía están abiertos se cerrarán explícitamente desde el lado del servidor con el código de motivo de cierre, CloseCodes.NORMAL_CLOSURE (1000). Solo los sockets del ámbito de la aplicación permanecen abiertos y aún se puede acceder a ellos desde el servidor, incluso cuando la sesión o la vista asociada con la página en el lado del cliente haya expirado.
