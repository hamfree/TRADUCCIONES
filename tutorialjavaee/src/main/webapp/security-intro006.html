Establecimiento de una Conexión Segura Mediante SSL

La tecnología Secure Sockets Layer (SSL) es seguridad que se implementa en la capa de transporte (consulte Seguridad de la Capa de Transporte para obtener más información sobre la seguridad de la capa de transporte). SSL permite que los navegadores web y los servidores web se comuniquen a través de una conexión segura. En esta conexión segura, los datos se cifran antes de enviarse y luego se descifran al recibirlos y antes de procesarlos. Tanto el navegador como el servidor encriptan todo el tráfico antes de enviar cualquier dato.

SSL aborda las siguientes consideraciones de seguridad importantes.

  • Autenticación: durante su intento inicial de comunicarse con un servidor web a través de una conexión segura, ese servidor le presentará a su navegador web un conjunto de credenciales en forma de un certificado de servidor (también llamado certificado de clave pública). El propósito del certificado es verificar que el sitio es quién y qué dice ser. En algunos casos, el servidor puede solicitar un certificado que demuestre que el cliente es quién y lo que dice ser; este mecanismo se conoce como autenticación de cliente.

  • Confidencialidad: cuando los datos se transmiten entre el cliente y el servidor en una red, terceros pueden ver e interceptar estos datos. Las respuestas SSL están cifradas para que los datos no puedan ser descifrados por terceros y los datos permanezcan confidenciales.

  • Integridad: cuando los datos se transmiten entre el cliente y el servidor en una red, terceros pueden ver e interceptar estos datos. SSL ayuda a garantizar que los datos no serán modificados en tránsito por ese tercero.

El protocolo SSL está diseñado para ser lo más eficiente posible de forma segura. Sin embargo, el cifrado y el descifrado son procesos computacionalmente costosos desde el punto de vista del rendimiento. No es estrictamente necesario ejecutar una aplicación web completa sobre SSL, y es habitual que un desarrollador decida qué páginas requieren una conexión segura y cuáles no. Las páginas que pueden requerir una conexión segura incluyen las de inicio de sesión, información personal, pagos del carrito de compras o transmisión de información de tarjetas de crédito. Cualquier página dentro de una aplicación se puede solicitar a través de un socket seguro simplemente anteponiendo la dirección con https: en lugar de http:. Cualquier página que requiera absolutamente una conexión segura debe verificar el tipo de protocolo asociado con la solicitud de página y tomar la acción adecuada si no se especifica https:.

El uso de hosts virtuales basados ​​en nombres en una conexión segura puede ser problemático. Esta es una limitación de diseño del propio protocolo SSL. El protocolo de enlace SSL, mediante el cual el navegador del cliente acepta el certificado del servidor, debe ocurrir antes de que se acceda a la solicitud HTTP. Como resultado, la información de solicitud que contiene el nombre de host virtual no se puede determinar antes de la autenticación y, por lo tanto, no es posible asignar varios certificados a una sola dirección IP. Si todos los hosts virtuales en una sola dirección IP necesitan autenticarse con el mismo certificado, la adición de múltiples hosts virtuales no debería interferir con las operaciones normales de SSL en el servidor. Sin embargo, tenga en cuenta que la mayoría de los navegadores de los clientes compararán el nombre de dominio del servidor con el nombre de dominio que figura en el certificado, si corresponde; esto se aplica principalmente a los certificados oficiales firmados por una autoridad certificadora (CA). Si los nombres de dominio no coinciden, estos navegadores mostrarán una advertencia al cliente. En general, solo los hosts virtuales basados ​​en direcciones se usan comúnmente con SSL en un entorno de producción.

Verificación y Configuración del Soporte SSL

Como regla general, debe abordar los siguientes problemas para habilitar SSL para un servidor.

  • Debe haber un elemento Connector para un conector SSL en el descriptor de implementación del servidor.

  • Debe haber un almacén de claves y archivos de certificado válidos.

  • La ubicación del archivo de almacén de claves y su contraseña deben especificarse en el descriptor de implementación del servidor.

Un conector SSL HTTPS ya está habilitado en el servidor GlassFish.

Con fines de prueba y para verificar que la compatibilidad con SSL se ha instalado correctamente, cargue la página de introducción predeterminada con una URL que se conecta al puerto definido en el descriptor de implementación del servidor: 

                
https://localhost:8181/

El https en esta URL indica que el navegador debe usar el protocolo SSL. El localhost en este ejemplo asume que está ejecutando el ejemplo en su máquina local como parte del proceso de desarrollo. El 8181 en este ejemplo es el puerto seguro que se especificó donde se creó el conector SSL. Si está utilizando un servidor o puerto diferente, modifique este valor en consecuencia.
