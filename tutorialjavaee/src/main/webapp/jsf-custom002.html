Determinar Si Necesita un Componente o Renderizador Personalizado

La implementación de JavaServer Faces admite un conjunto muy básico de componentes y renderizadores asociados. Esta sección le ayuda a decidir si puede usar componentes y renderizadores estándar en su aplicación o si necesita un componente personalizado o un renderizador personalizado.

Aquí se tratan los siguientes temas:

  • Cuándo Usar un Componente Personalizado

  • Cuándo Usar un Renderizador Personalizado

  • Combinaciones de Componente, Renderizador, y Etiqueta

Cuándo Usar un Componente Personalizado

Una clase de componente define el estado y el comportamiento de un componente de la interfaz de usuario. Este comportamiento incluye convertir el valor de un componente en el marcado apropiado, poner en cola los eventos de los componentes, realizar la validación y cualquier otro comportamiento relacionado con la forma en que el componente interactúa con el navegador y el ciclo de vida de procesamiento de solicitudes.

Necesita crear un componente personalizado en las siguientes situaciones.

  • Debe agregar un nuevo comportamiento a un componente estándar, como generar un tipo adicional de evento (por ejemplo, notificar a otra parte de la página que algo cambió en este componente como resultado de la interacción del usuario).

  • Debe realizar una acción diferente en el procesamiento de solicitudes del valor de un componente de lo que está disponible en cualquiera de los componentes estándar existentes.

  • Desea aprovechar una capacidad HTML ofrecida por su navegador de destino, pero ninguno de los componentes estándar de JavaServer Faces aprovecha la capacidad de la manera que desea, si es que lo hace. La versión actual no contiene componentes estándar para componentes HTML complejos, como marcos; sin embargo, debido a la extensibilidad de la arquitectura del componente, puede utilizar la tecnología JavaServer Faces para crear componentes como estos. El estudio de caso de la librería de Duke crea componentes personalizados que corresponden a las etiquetas HTML map y area.

  • Necesita renderizar a un cliente que no sea HTML que requiera componentes adicionales que no sean compatibles con HTML. Eventualmente, el kit de renderizado HTML estándar brindará soporte para todos los componentes HTML estándar. Sin embargo, si está renderizando a un cliente diferente, como un teléfono, es posible que deba crear componentes personalizados para representar los controles admitidos exclusivamente por el cliente. Por ejemplo, algunas arquitecturas de componentes para clientes inalámbricos incluyen soporte para tickers y barras de progreso, que no están disponibles en un cliente HTML. En este caso, es posible que también necesite un renderizador personalizado junto con el componente, o puede que solo necesite un renderizador personalizado.

No es necesario crear un componente personalizado en los siguientes casos.

  • Necesita agregar componentes para crear un nuevo componente que tenga su propio comportamiento único. En esta situación, puede utilizar un componente compuesto para combinar componentes estándar existentes. Para obtener más información sobre componentes compuestos, consulte Componentes Compuestos y Capítulo 14, "Componentes Compuestos: Temas Avanzados y un Ejemplo" .

  • Simplemente necesita manipular los datos del componente o agregarle funcionalidad específica de la aplicación. En esta situación, debe crear un bean administrado para este propósito y vincularlo al componente estándar en lugar de crear un componente personalizado. Ver Beans Gestionados en la Tecnología JavaServer Faces para obtener más información sobre beans gestionados.

  • Debe convertir los datos de un componente a un tipo no compatible con su renderizador. Ver Usando los Convertidores Estándar para obtener más información sobre cómo convertir los datos de un componente.

  • Debe realizar la validación de los datos del componente. Se pueden agregar validadores estándar y validadores personalizados a un componente utilizando las etiquetas de validación de la página. Ver Usando los Validadores Estándar y Creando y Usando un Validador Personalizado para obtener más información sobre cómo validar los datos de un componente.

  • Debe registrar los detectores de eventos en los componentes. Puede registrar oyentes de eventos en componentes usando las etiquetas f:valueChangeListener y f:actionListener, o puede apuntar a un método de procesamiento de eventos en un bean gestionado usando los Atributos actionListener o valueChangeListener del componente. Ver Implementando un Oyente de Evento y Escribiendo Métodos de Bean Gestionado para más información.

Cuándo Usar un Renderizador Personalizado

Un renderizador, que genera el marcado para mostrar un componente en una página web, le permite separar la semántica de un componente de su apariencia. Al mantener esta separación, puede admitir diferentes tipos de dispositivos cliente con el mismo tipo de experiencia de creación. Puede pensar en un renderizador como un "adaptador de cliente". Produce una salida adecuada para el consumo y visualización por parte del cliente y acepta la entrada del cliente cuando el usuario interactúa con ese componente.

Si está creando un componente personalizado, debe asegurarse, entre otras cosas, de que su clase de componente realice estas operaciones que son fundamentales para representar el componente:

  • Decodificación: conversión de los parámetros de la solicitud entrante al valor local del componente

  • Codificación: conversión del valor local actual del componente en el marcado correspondiente que lo representa en la respuesta

La especificación JavaServer Faces admite dos modelos de programación para manejar la codificación y decodificación.

  • Implementación directa: la clase de componente en sí implementa la decodificación y codificación.

  • Implementación delegada: la clase de componente delega la implementación de codificación y decodificación a un renderizador separado.

Al delegar las operaciones al renderizador, tiene la opción de asociar su componente personalizado con diferentes renderizadores para que pueda renderizar el componente en diferentes clientes. Si no planea renderizar un componente en particular en diferentes clientes, puede ser más sencillo dejar que la clase del componente maneje la renderización. Sin embargo, un renderizador independiente le permite preservar la separación de la semántica de la apariencia. La aplicación Duke's Bookstore separa los procesadores de los componentes, aunque solo se procesa en navegadores web HTML 4.

Si no está seguro de si necesitará la flexibilidad que ofrecen los renderizadores separados, pero desea utilizar el enfoque de implementación directa más simple, puede utilizar ambos modelos. Su clase de componente puede incluir algún código de renderizado predeterminado, pero puede delegar el renderizado a un renderizador si lo hay.

Combinaciones de Componente, Renderizador, y Etiqueta

Cuando crea un componente personalizado, puede crear un renderizador personalizado que lo acompañe. Para asociar el componente con el renderizador y hacer referencia al componente desde la página, también necesitará una etiqueta personalizada.

Aunque necesita escribir el componente personalizado y el renderizador, no es necesario escribir código para una etiqueta personalizada (llamada manejador de etiquetas). Si especifica la combinación de componente y renderizador, Facelets crea el controlador de etiquetas automáticamente.

En raras ocasiones, puede utilizar un renderizador personalizado con un componente estándar en lugar de un componente personalizado. O puede usar una etiqueta personalizada sin un renderizador o un componente. Esta sección ofrece ejemplos de estas situaciones y resume lo que se requiere para un componente, un renderizador y una etiqueta personalizados.

Usaría un renderizador personalizado sin un componente personalizado si quisiera agregar alguna validación del lado del cliente en un componente estándar. Implementaría el código de validación con un lenguaje de scripting del lado del cliente, como JavaScript, y luego renderizaría JavaScript con el renderizador personalizado. En esta situación, necesita una etiqueta personalizada que vaya con el renderizador para que su controlador de etiquetas pueda registrar el renderizador en el componente estándar.

Los componentes personalizados, así como los renderizadores personalizados, necesitan etiquetas personalizadas asociadas a ellos. Sin embargo, puede tener una etiqueta personalizada sin un renderizador personalizado o un componente personalizado. Por ejemplo, suponga que necesita crear un validador personalizado que requiera atributos adicionales en la etiqueta del validador. En este caso, la etiqueta personalizada corresponde a un validador personalizado y no a un componente personalizado o un renderizador personalizado. En cualquier caso, aún debe asociar la etiqueta personalizada con un objeto del lado del servidor.
