Usar finally para desbloquear

Algunas veces los objetos Lock se usan para controlar el acceso a datos mutables en un entorno multihilo. Cuando un hilo termina con los datos, debe asegurarse de que se libera el bloqueo.

Brian Goetz en Concurrencia de Java en la Práctica recomienda firmemente la técnica siguiente para asegurar que se libera un bloqueo. Usa un estilo try..finally; esto asegura que sin importar lo que ocurra durante el tiempo que se mantenía el bloqueo, este será eventualmente desbloqueado. Incluso si se lanza una RuntimeException inesperada, el bloqueo aún será liberado. 

Lock myLock = ...
myLock.lock();
try {
  //interactuar con datos mutables
}
finally{
  myLock.unlock();
}

Aquí tiene un ejemplo que sigue el patrón de arriba, con un bloque de lectura-escritura.

import java.util.LinkedHashMap;
import java.util.Map;
import java.util.concurrent.locks.Lock;
import java.util.concurrent.locks.ReentrantReadWriteLock;

/**
  Preferencias de usuario, usando un bloque de lectura-escritura.
  
 <P>El contexto: la información de preferencias se lee al inicio.
  Los datos de configuración son 'en su mayoría de lectura': por lo general, una persona que llama simplemente lee el
  información. Se actualiza solo ocasionalmente.
 
 <P>El uso de un bloqueo de lectura y escritura significa que varios lectores pueden acceder a los mismos datos 
  simultáneamente. Sin embargo, si un solo escritor obtiene el bloqueo, todas las demás personas que llaman (ya sea lector o 
  escritor) se bloquearán hasta que el escritor libere el bloqueo.
 
  <P>(En la práctica, Brian Goetz informa de que la implementación de ConcurrentHashMap 
  es tan buena que probablemente sería suficiente en muchos casos, en lugar de un bloqueo de lectura y escritura).
*/
public final class Preferences {
  
  /** Obtiene una configuración - esta es la operación más común.  */
  public String fetch(String aName){
    String result = "";
    fReadLock.lock();
    try {
      result = fPreferences.get(aName);
    }
    finally {
      fReadLock.unlock();
    }
    return result;
  }
 
  /** Cambia una configuración - esta es la operación menos común. */
  public void update(String aName, String aValue){
    fWriteLock.lock();
    try {
      fPreferences.put(aName, aValue);
    }
    finally {
      fWriteLock.unlock();
    }
  }
 
  //...omitido
  
  // PRIVADO
  
  /** Contiene las preferencias como simples pares de nombre-valor de Strings. */
  private final Map<String, String> fPreferences = new LinkedHashMap<>();
  private final ReentrantReadWriteLock fLock = new ReentrantReadWriteLock();
  private final Lock fReadLock = fLock.readLock();
  private final Lock fWriteLock = fLock.writeLock();
}
