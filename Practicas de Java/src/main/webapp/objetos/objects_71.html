Evite clone

Evite implementar clone.
  • clone es muy difícil de implementar correctamente en todas las circunstancias, casi hasta el punto de ser patológico
  • la importancia de copiar objetos siempre permanecerá, ya que los campos de los objetos a menudo necesitan ser copiados defensivamente
  • los constructores de copia y los métodos de factoría estática ofrecen una alternativa a clone, y son mucho más fáciles de implementar
Si necesita extender una superclase que implementa clone, entonces su subclase debe implementar clone también. La solución más rápida para su subclase es simplemente lanzar una excepción.

Ejemplo

Este ejemplo muestra una superclase con una implementación típica de clone, y una subclase la cual ha deshabilitado su método clone. 

import java.util.Date;

public abstract class Fruit implements Cloneable {

  public Fruit(String aColour, Date aBestBeforeDate) {
    super();
    fColour = aColour;
    //copia defensiva necesitada para este objeto mutable
    fBestBeforeDate = new Date(aBestBeforeDate.getTime());
  }

  public abstract void ripen();

  public String getColour() {
    return fColour;
  }

  public Date getBestBeforeDate() {
    //devuelve la copia defensiva de este objeto mutable
    return new Date(fBestBeforeDate.getTime());
  }

  /**
  * Implementa clone como sigue
  * <ul>
  * <li>la declaración de la clase "implements Cloneable" (no necesitada si ya está
  * declarada en la superclase)
  * <li>declara al método clone como público
  * <li>si la clase es final, clone no necesita lanzar CloneNotSupportedException
  * <li>llama a super.clone y moldea a esta clase
  * <li>como en la copia defensiva, se asegura que cada campo mutable tenga construida 
  * una copia independiente, para evitar compartir el estado interno entre objetos
  * </ul>
  */
  @Override public Object clone() throws CloneNotSupportedException {
    //obtiene una copia inicial bit-por-bit, la cual maneja todos los campos inmutables
    Fruit result = (Fruit)super.clone();

    //los campos mutables deben hacerse independientes de este objeto, por razones
    //similares a esas por copias defensivas - para prevenir  accesos no queridos al 
    //estado interno de este objeto
    result.fBestBeforeDate = new Date(this.fBestBeforeDate.getTime());

    return result;
  }

  // PRIVADO

  /** Los Strings son siempre inmutables. */
  private String fColour;

  /**
  * En código nuevo, debería usar LocalDate, no java.util.Date. 
  * Aquí, java.util.Date se usa solo para hacer un punto con respecto a los campos mutables.
  * 
  * Date es un objeto mutable. En esta clase, este campo del objeto va ser tratado 
  * como perteneciente enteramente a esta clase, y ningún usuario de esta clase es 
  * capaz de acceder directamente y cambiar el estado de este campo.
  */
  private Date fBestBeforeDate;
}
Aquí está la subclase, con su método clone deshabilitado. 
import java.util.Date;

public final class Apple extends Fruit {

  public Apple(String aColour, Date aBestBeforeDate) {
    super(aColour, aBestBeforeDate);
  }

  public void ripen() {
    //implementación vacía del método abstracto
  }

  /**
  * La subclase Apple no soporta clone.
  */
  @Override public final Object clone() throws CloneNotSupportedException {
    throw new CloneNotSupportedException();
  }
}
