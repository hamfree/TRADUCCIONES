Encapsular colecciones

En general , las colecciones no son objetos inmutables. Como tales, a menudo se debe tener cuidado de que los campos de la colección no se expongan involuntariamente a la función llamante.

Una técnica es definir un conjunto de métodos relacionados los cuales impiden que el llamante use directamente la colección subyaciente, tales como:

  • agregaCosa(Cosa)
  • borraCosa(Cosa)
  • obtenCosas() - devuelve una inmodificable Collection
Ejemplo 1 
import java.util.*;

public final class SoccerTeam {

  public SoccerTeam(String aTeamName, String aHeadCoachName){
    //..no incluído
  }

  public void addPlayer(Player aPlayer){
    fPlayers.add(aPlayer);
  }

  public void removePlayer(Player aPlayer){
    fPlayers.remove(aPlayer);
  }

  public Set<Player> getPlayers(){
    return Collections.unmodifiableSet(fPlayers);
  }

  //..no incluído

  // PRIVADO
  private Set<Player> fPlayers = new LinkedHashSet<>();
  private String fTeamName;
  private String fHeadCoachName;
}
Ejemplo 2

BaseballTeam es un ejemplo de exponer la colección directamente a la función llamante. Esto no es necesariamente un diseño incorrecto, pero es más arriesgado, ya que los contenidos de la colección pueden ser directamente cambiados tanto por BaseballTeam y su función llamante: 

import java.util.*;

public final class BaseballTeam {

  public BaseballTeam(String aTeamName, String aHeadCoachName){
    //..no incluído
  }

  public void setPlayers(Set<Player> aPlayers){
    fPlayers = aPlayers;
  }

  public Set<Player> getPlayers(){
    return fPlayers;
  }

  //....no incluído

  // PRIVADO
  private Set<Player> fPlayers;
  private String fTeamName;
  private String fHeadCoachName;
}
