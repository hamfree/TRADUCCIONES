Operaciones de linea de comandos

La mayoría de los programadores Java usan un Entorno de Desarrollo Integrado (EID) tal como IntelliJ, NetBeans, o Eclipse. Los EID modernos son habitualmente agradables de usar y contribuyen en gran medida a simplificar las cosas para un programador típico.

Sin embargo, estar familiarizado con las herramientas de línea de comando sigue siendo muy útil. No solo le brinda una mejor comprensión de lo que está sucediendo, es esencial en situaciones en las que necesita usar las herramientas de línea de comandos directamente (scripts, herramientas, etc.).

Las herramientas más utilizadas son:

  • javac- el compilador de java
  • java- lanza programas en el Entorno de Ejecución de Java
  • javadoc- genera documentación para su proyecto

Aquí tiene algunos ejemplos de ejecuciones de herramientas de líneas de comandos. Asume un proyecto con la siguiente estructura de directorios: 

PROJECT_HOME
  -> lib (archivos  jar)
  -> src (código java)
    -> hirondelle (paquete nivel superior; sin archivos .java)
      -> ante (archivos .java)
        -> deluvian (archivos .java)

Compilar Con javac 

PROJECT_HOME>javac -cp lib\\* src\\hirondelle\\ante\\*.java src\\hirondelle\\ante\\deluvian\\*.java
Esto se compila en su lugar y crea archivos .class junto a los archivos .java. Si desea colocar los archivos de clase generados en otro lugar, use la opción -d para colocarlos en un directorio existente: 
PROJECT_HOME>javac -cp lib\\* -d build src\\hirondelle\\ante\\*.java src\\hirondelle\\ante\\deluvian\\*.java
Además, observe que todos los los archivos jars en un directorio pueden ser referenciados sucintamente usando 'lib\\*'. Sin embargo, hacer referencia a los archivos de origen que necesita compilar es una historia diferente: todos sus paquetes deben aparecer uno por uno. Es decir, no puede simplemente especificar un único directorio raíz para sus archivos de origen. Este es un defecto bastante grave de javac.

Si sus archivos jars están en varios directorios, entonces la ruta de clases es una lista delimitada por puntos y comas: 

-cp lib\\*;C:\\abc\\one.jar;C:\\xyz\\two.jar

Ejecutar Con java

Cualquier clase que tenga un método main puede ser ejecutada como un programa Java. El método main debe tener la forma: 
public static void main(String... aArgs){...}
Un aplicación se ejecuta referenciando un nombre de clase completamente cualificado, pero sin la extensión .class. Asumiendo que las clases se hayan compilado en su lugar, nuestra aplicación de ejemplo se puede ejecutar usando: 
PROJECT_HOME>java -cp lib\\*;src hirondelle.ante.Launcher
Pasar 3 argumentos al programa: 
PROJECT_HOME>java -cp lib\\*;src hirondelle.ante.Launcher arg1 arg2 "Arg Three"
Establecer un valor de propiedad del Sistema con -D: 
PROJECT_HOME>java -Dbla=loquesea -cp lib\\*;src hirondelle.ante.Launcher
Establecer el tamaño inicial y máximo del grupo de asignación de objetos en 5 MB y 100 MB, respectivamente: 
PROJECT_HOME>java -cp lib\\*;src -Xms5m -Xmx100m hirondelle.ante.Launcher
Ejecutar un archivo jar cuyo manifiesto especifica un atributo Main-Class: 
C:\\@build\\dist>java -jar example-app-1.1.jar
El comando javaw es el mismo que el comando java, con la única diferencia de que javaw no tiene asociada una ventana de consola.

Usar javadoc

Para los archivos bajo el directorio src, el comando siguiente generará la documentación javadoc de todos los paquets que empiezan con 'hirondelle.ante', y genera el resultado en C:\\@build\\javadoc: 

javadoc -sourcepath src -subpackages hirondelle.ante -classpath lib\\* -d C:\\@build\\javadoc
La herramienta javado tiene muchas opciones. Puede encontrar estas especialmente útiles:
  • -linksource: vincula su javadoc a una instantánea estática del código fuente subyacente (muy bueno).
  • -windowtitle, -header y -footer: a menudo es útil para mostrar el nombre y la versión de su aplicación.
  • -link: enlace al javadoc del KDJ también
  • -noqualifier: elimina los tediosos prefijos de paquetes de las clases del KDJ
  • define el alcance mínimo de clase a ser procesado, usando uno de: -private, -package, -protected, -public.
Aquí tiene un segundo ejemplo, con más opciones: 
javadoc 
 -sourcepath src 
 -subpackages hirondelle.ante 
 -package
 -classpath lib\\* 
 -d C:\\@build\\javadoc
 -linksource
 -link http://docs.oracle.com/javase/7/docs/api/
 -noqualifier java.*:javax.*
 -windowtitle "My App 1.0"
 -header "<b>My App 1.0</b>"
 -footer "<a href='http://www.blah.com'>My App</a>"
Javadoc puede también ser controlaa usando los elementos siguientes, colocados al lado de su código fuente:
  • package-info.java - Cada paquete puede incluir un archivo package-info.java. Contiene la documentación de nivel de paquete. A pesar de su extensión, un archivo package-info.java no es un archivo de clase Java.
  • subdirectorios archivos-doc - cada paquete puede incluir un subdirectorio 'archivos-doc', contiene elementos que pueden ser enlazados del javadoc (imágenes, archivos html, etcetera) usando enlaces de hipertexto ordinarios.
  • overview.html - un único archivo que describe los aspectos generales de su proyecto. (Este archivo no reside necesariamente en su árbol de fuentes, pero tiene sentido colocarlo ahí, en analogía con los archivos package-info.java).
