Ejecutar en modo depuración

Cuando desarrolla una aplicación, es a menudo deseable ejecutarla en modo depuración:
  • puede suspender la ejecución en un punto específico en su codigo (puntos de ruptura), y después recorrer el código, una línea cada vez.
  • Además de establecer los puntos de corte para líneas específicas de código, puede también establecer puntos de ruptura para atrapar excepciones específicas.
  • Cuando se suspende la ejecución puede tanto examinar los datos y cambiarla al vuelo.
  • Puede habitualmente drop stack frames para moverse hacia atrás en el tiempo, a métodos invocados anterioremente, sin tener que reiniciar la aplicación entera desde el principio. Esto puede salvar un montón de tiempo.
  • la depuración suele ser mucho más eficaz que utilizar declaraciones de registro para analizar problemas.
Los desarrolladores usualmente usan el depurador construído dentro de su EID (Eclipse, IntelliJ, etcetera.)

Tenga en cuenta que el Kit de Desarrollo de Java (KDJ) viene con su propio depurador integrado llamado jdb. Tiene una interfaz de línea de comando,.Hay diferentes formas de que usarlo:

  • lanze jdb con la clase principal de su aplicación.
  • dígale a jdb que conecte un segundo Entorno de Tiempo de Ejecución de Java (EEJ). El segundo EEJ puede estar ejecutándose localmente o de forma remota.
Por ejemplo:

Estilo 1: 

>jdb MyMainClass

Estilo 2: 

>java -agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=8000 MyMainClass >jdb -attach 8000
Si el EEJ de destino es remoto, entonces la sintaxis es usualmmente como esta: 
>jdb -attach my-remote-host-name:8000

Su EID usa una técnica similar en segundo plano. Los EIDs tienen una interfaz gráfica de usuario con la misma clase de de depuración de operaciones y datos que están disponibles en jdb.

En raras ocasiones, jdb podría posiblemente usarse para investigar problemas en una aplicación implementada, donde un EID no está disponible.

La herramienta javac tiene una opción -g que incrementa la cantidad de información de depuración colocada en los ficheros class compilados. Usted no necesita habitualmente la opción -g cuando desarrolla, ya que puede habitualmente apuntar su depurador al código fuente subyacente.
