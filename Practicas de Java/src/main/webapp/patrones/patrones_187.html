Objetos de Modelo

El término Objeto de Modelo es un término informal, sin una definición ampliamente aceptada. Aqui, Los Objetos de Modelo (MOs, en inglés) se refieren a clases centradas en datos que encapsulan elementos estrechamente relacionados.

Objetos de Modelo:

  • son muy comunes, y se usan en casi todas las aplicaciones
  • a menudo son fundamentales para una aplicación, ya que generalmente modelan objetos de dominio de problemas
  • a menudo se asignan aproximadamente a los registros de una tabla de base de datos correspondiente
  • son a menudo usados con valores devueltos para los métodos de un Objeto de acceso a datos
  • se prueban fácilmente usando JUnit (o una herramienta similar)
  • se pueden usar para implementar el Modelo en un patrón Modelo-Vista-Controlador
¿Deben los objetos de modelo seguir las convenciones de JavaBeans?

No necesariamente. En realidad, algunos argumentan que el estilo de JavaBeans debe ser evitado como modelo general para los Objetos de Modelo.

¿Deberían los Objetos de Modelo ser inmutables?

Dada la profunda simplicidad de los objetos inmutables, algunos prefieren diseñar sus Objetos de Modelo como inmutables. Sin embargo, cuando los datos subyacentes cambian, se debe crear un nuevo objeto, en vez de llamar simplemente un método setXXX en un objeto existente. Algunos argumentan que esta penalización es demasiado alta, mientras que otros argumentan que es una microoptimización - Especialmente en casos donde los datos son "leídos mayormente", y el estado de los Objetos de Modelo correspondientes cambian solo raramente.

Implementar Objetos de Modelo como inmutables parece particularmente natural en las aplicaciones web. Ahí, los Objetos de Modelo son a menudo colocados habitualmente en el alcance de petición (request, en inglés), no en el alcance de sesión. En este caso, no hay objetos de larga duración que el usuario altere directamente, de forma que el Objeto de Modelo puede ser inmutable.

Ejemplo

Comment representa un comentario publicado en un tablón de anuncios. Su implementación sigue el patrón de Objeto Inmutable. Comment proporciona los métodos usuales getXXX. Tenga en cuenta que, en este caso, una copia defensiva se usa para el campo Date. También implementan los métodos toString, equals, y hashCode.

El constructor es responsable de establecer la clase invariante, y realizar validación del Objeto de Modelo.

package hirondelle.fish.main.discussion; 

import java.util.*;
import hirondelle.web4j.model.ModelCtorException;
import hirondelle.web4j.model.ModelUtil;
import hirondelle.web4j.model.Check;
import hirondelle.web4j.security.SafeText;
import static hirondelle.web4j.util.Consts.FAILS;

/** 
 Comentario publicado por un usuario anónimo posiblemente anónimo.
*/
public final class Comment { 

  /**
   Constructor. 
     
   @param aUserName identifica el usuario registrado que publica el comentario. 
   @param aBody el comentario, debe tener contenido.
   @param aDate fecha y tiempo de cuando el mensaje fue publicado.
  */
  public Comment (
    SafeText aUserName, SafeText aBody, Date aDate
  ) throws ModelCtorException {
    fUserName = aUserName;
    fBody = aBody;
    fDate = aDate.getTime();
    validateState();
  }

  /** Devuelve el nombre del usuario registrado pasado al constructor. */
  public SafeText getUserName() {
    return fUserName;
  }

  /** Devuelve el cuerpo del mensaje pasado al constructor.  */
  public SafeText getBody() {
    return fBody;
  }

  /**
   Devuelve una <a href="../practicascomunes/practicascomunes_15.html">copia defensiva</a> de la fecha pasada al 
   constructor.
   
   <P>El llamador puede cambiar el estado del valor devuelto, sin afectar
   los internos de este <tt>Comment</tt>. Dicha copia es necesaria dado que 
   una {@link Date} es un objeto mutable.
  */
  public Date getDate() {
    // el objeto devuelto es independiente de fDate
    return new Date(fDate);
  }

  /** Destinado solo para depuración. */
  @Override public String toString() {
    return ModelUtil.toStringFor(this);
  }

  @Override public boolean equals( Object aThat ) {
    Boolean result = ModelUtil.quickEquals(this, aThat);
    if ( result == null ){
      Comment that = (Comment) aThat;
      result = ModelUtil.equalsFor(
        this.getSignificantFields(), that.getSignificantFields()
      );
    }
    return result;    
  }

  @Override public int hashCode() {
    if ( fHashCode == 0 ) {
      fHashCode = ModelUtil.hashCodeFor(getSignificantFields());
    }
    return fHashCode;
  }

  // PRIVADO // 
  private final SafeText fUserName;
  private final SafeText fBody;
  /** Aquí se utiliza Long en lugar de Date para garantizar la inmutabilidad.*/
  private final long fDate;
  private int fHashCode;
  
  private Object[] getSignificantFields(){
    return new Object[] {fUserName, fBody, new Date(fDate)};
  }
  
  private void validateState() throws ModelCtorException {
    ModelCtorException ex = new ModelCtorException();
    if( FAILS ==  Check.required(fUserName) ) {
      ex.add("El nombre de usuario debe tener contenido.");
    }
    if ( FAILS == Check.required(fBody) ) {
      ex.add("El cuerpo del comentario debe tener contenido.");
    }
    if ( ! ex.isEmpty() ) throw ex;
  }
}
