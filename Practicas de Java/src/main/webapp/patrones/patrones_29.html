Objetos inmutables

Los objetos inmutables son objetos simples cuyo estado (los datos del objeto) no pueden cambiar después de la construcción. Ejemplos de objetos inmutables del KDJ incluyen String e Integer.

Los objetos inmutables simplifican enormemente su programa, ya que ellos:

  • son simples de construir, probar, y usar
  • son automáticamente seguros en programción multihilo y no tiene problemas de sincronización
  • no necesitan un constructor de copia
  • no necesitan una implementación de clone
  • Permiten que hashCode use la inicialización perezosa, y almacenar en caché su valor de retorno
  • no necesita ser copiado defensivamente cuando se usa como campo
  • hace buenas claves de Map y elementos de Set (estos objetos no deben cambiar de estado mientras están en la colección)
  • tienen sus invariantes de clase establecidos una vez en la construcción, y nunca necesitan ser revisados nuevamente
  • siempre tiene "atomicidad de falla" (un término utilizado por Joshua Bloch): si un objeto inmutable arroja una excepción, nunca se deja en un estado indeseable o indeterminado
Los objetos inmutables tienen una lista muy convincente de cualidades positivas. Sin lugar a dudas, se encuentran entre los tipos de clases más simples y robustas que puede crear. Cuando crea clases inmutables, categorías enteras de problemas simplemente desaparecen.

Haga una clase inmutable seguiendo estas pautas:

  • asegúrese de que la clase no puede ser sobreescrita - haga la clase final, o use factorías estáticas y mantenga los constructores privados
  • haga los campos private y final
  • fuerze a los llamadores a construir un objeto completamente en un único paso, en vez de usar un constructor sin argumentos combinado con llamadas subsecuentes a métodos setXXX (esto es, evite la convención de Java Beans)
  • no proporciona ningún método que pueda cambiar el estado del objeto de cualquier forma - no simplemente métodos setXXX, sino cualquier método que pueda cambiar el estado
  • si la clase tiene cualquier campo mutable de objeto, entonces deben ser copiados defensivamente cuando se pasen entre la clase y su llamante.
En Java Efectivo, Joshua Bloch hace esta convincente recomendación:

"Las clases deberían ser inmutables a menos que haya una muy buena razón para hacerlas mutables ... Si una clase no puede hacerse inmutable, limite su mutabilidad tanto como sea posible."

Es interesante tener en cuenta que BigDecimal es técnicamente no inmutable, ya que no es final.

Ejemplo

import java.util.Date;

/**
* Planet es una clase inmutable, ya que no hay forma de cambiar 
* su estado después de la construcción.
*/
public final class Planet {

  public Planet (double aMass, String aName, Date aDateOfDiscovery) {
     fMass = aMass;
     fName = aName;
     //hace una copia privada de aDateOfDiscovery
     //esta es la única forma de mantener el campo fDateOfDiscovery
     //privado, y blinda esta clase de cualquier cambio que  
     //el llamador pueda hacer al objeto aDateOfDiscovery original 
     fDateOfDiscovery = new Date(aDateOfDiscovery.getTime());
  }

  /**
  * Devuelve un valor primitivo.
  *
  * El llamador puede hacer lo que quiera con el valor devuelto, sin
  * afectar las partes internas de esta clase. ¿Porqué? Porqu este es un valor 
  * primitivo. El llamador vea su "propio" double que simplemente tiene el 
  * mismo valor que fMass.
  */
  public double getMass() {
    return fMass;
  }

  /**
  * Devuelve un objeto inmutable.
  *
  * El llamador obtiene una referencia directa al campo interno. Pero esto no es 
  * peligroso, ya que String es inmutable y no puede ser cambiada.
  */
  public String getName() {
    return fName;
  }

//  /**
//  * Devuelve un objeto mutable- probablemente un mal estilo.
//  *
//  * El llamador obtiene una referencia directa al campo interno. Esto es habitualmente peligroso, 
//  * ya que el estado del objeto Date puede cambiarse tanto por esta clase como por su llamador.
//  * Es decir, esta clase ya no tiene el control completo de fDate.
//  */
//  public Date getDateOfDiscovery() {
//    return fDateOfDiscovery;
//  }

  /**
  * Devuelve un objeto mutable - buen estilo.
  * 
  * Devuelve una copia defensiva del campo.
  * El llamador de este método puede hacer cualquier cosa que quiera con el 
  * objeto Date devuelto, sin afectar las partes internas de esta 
  * clase de ninguna forma. ¿Porqué? Porque no tinee una referencia a 
  * fDate. Mas bien, está jugando con una segunda Date que inicialmente tiene los 
  * fDate. Rather, they are playing with a second Date that initially has the 
  * mismos datos que fDate.
  */
  public Date getDateOfDiscovery() {
    return new Date(fDateOfDiscovery.getTime());
  }

  // PRIVADO

  /**
  * Los datos primitivos finales siempre son inmutables.
  */
  private final double fMass;

  /**
  * Un campo de objeto inmutable. (Los objetos String nunca cambian su estado).
  */
  private final String fName;

  /**
  * Un campo de objeto mutable. En este caso, el estado de campo mutable
  * va a ser cambiado solo por esta clase. (En otros casos, tiene perfecto 
  * sentido permitir qu el estado de un campo sea cambiado fuera de la clase 
  * nativa; este es el caso donde un campo actúa como un "puntero" a un objeto 
  * creado en otra parte).
  *
  * En código nuevo, debe usar las clases de java.time , no java.util.Date.
  */
  private final Date fDateOfDiscovery;
}

Tenga en cuenta que javadoc 1.4 incluye la opción -tag, donde las etiquetas simples personalizadas pueden definirse. Uno podría definir una etiqueta @is.Immutable, por ejemplo, para documentar una clase que está siendo inmutable.

Podría también considerar definir su propia interfaz de etiqueta para los objetos inmutables.
