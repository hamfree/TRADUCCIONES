La aserción es sólo para argumentos privados

La mayoría de las verificaciones de validez en un programa son verificaciones de parámetros pasados a métodos no privados. La palabra clave assert no está destinada a este tipo de validaciones. (Algunos pueden encontrar esto sorprendente. Aquí está la guía de Oracle).

Las aserciones pueden ser deshabilitadas. Dado que las verificaciones de los parámetros de los métodos no privados implementan los requisitos exigidos por la persona que llama, la desactivación de dichas verificaciones en tiempo de ejecución significaría que parte del diseño por contrato ya no se aplica.

A la inversa, comprobar argumentos en métodos privados pueden usar assert. Estas comprobaciones son realizadas para verificar suposiciones sobre los detalles internos de la implementación, y no para comprobar que la persona que llama ha seguido los requerimientos de un contrato del méetodo no privado.

Además, si una AssertionError puede ser lanzada por un método, entonces este hecho no debería aparecer en el comentario javadoc del método. La especificación del comportamiento de un método nunca debería mencionar las aserciones, ya que las aserciones pueden ser deshabilitadas.

Ejemplo

public final class SpaceShuttle {
  
    /**
    * La no nulidad de aName es parte del contrato del 
    * constructor.
    *
    * @param aName es no nulo.
    */
    public SpaceShuttle(String aName) {
      //forma INCORRECTA:
      assert aName != null : "Name no debe ser nulo";
      //forma CORRECTA:
      if (aName == null) {
        throw new IllegalArgumentException("Name no debe ser nulo.");
      }
      fName = aName;
    }
  
    public String getName() {
      return fName;
    }
  
    // PRIVADO 
    private final String fName;
  }
