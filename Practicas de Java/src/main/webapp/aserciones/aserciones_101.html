Aserciones en general

Aserciones en general:
  • cuando una aserción se dispara, pase el dato que lo provoca a AssertionError
  • las comprobaciones realizadas dentro de una asrción son usualmente simples condiciones booleanas - los efectos secundarios generalmente deben evitarse
  • ya que las aserciones pueden ser deshabilitadas, el comportamiento de un programa no debe depender de las aserciones que se ejecuten
  • si se desea, el idioma de compilación condicional puede ser aún usado
  • Usted puede casi siempre ignorar los métodos en ClassLoader con respecto a las aserciones - están destinadas a intérpretes como java, y no son usados en un programa típico
Recordatorios sobre uso:
  • en tiempo de ejecución, las aserciones se deshabilitan por defecto
  • los parámetros de ejecución -enableassertions  y -disableassertions (o sus sinónimos -ea y -da) controlan las aserciones en cualquier grado de granularidad
  • AssertionError se comporta como cualquier otro Error, y no se ve afectado por habilitar o deshabilitar las aserciones.
  • para compilar código que contiene aserciones, los JDKs más viejos necesitan agregar -source 1.4 como un argumento de línea de comandos a javac
Ejemplos de estilos típico, que afectan a clases no del sistema:
Linea de Comandos de Ejecución Clases con las aserciones habilitadas
java -ea  Main todas las clases
java -ea:com.Blah   Main una clase particular llamada Blah
java -ea:com.data... Main todas las clases en el paquete com.data, y todos sus subpaquetes
java -ea -da:com.Blah Main todas menos una clase llamada Blah
