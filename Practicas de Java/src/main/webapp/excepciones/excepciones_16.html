Evite los bloques catch vacíos

La mayoría sostiene que, por lo general, es una muy mala idea tener un bloque de captura vacío.

Cuando ocurre la excepción, no ocurre nada, y el programa falla por razones desconocidas.

Ejemplo

Los comentarios de "hicimos nuestro mejor esfuerzo" a continuación son un ejemplo de lo que no se debe hacer: 

import java.io.*;
import java.util.*;
import java.nio.file.*;

/** Cómo NO implementar una captura.  */
public final class BadCatch {

  public static void main(String... arguments) {
    List<String> quarks = Arrays.asList(
      "up", "down", "charm", "strange", "top", "bottom"
    );   

    //serializamos la Lista
    Path path = Paths.get("quarks.ser");
    try(
      ObjectOutputStream output = new ObjectOutputStream(Files.newOutputStream(path))
    ){
      output.writeObject(quarks);
    }
    catch (IOException ex) {
      //HICIMOS NUESTRO MEJOR ESFUERZO
    }
  }
}
En general, cuando ocurre una excepción, puede ser lanzada de vuelta al llamante, o puede ser capturada en un bloque catch. Cuando se captura una excepción, algunas opciones incluyen:

  • informar al usuario (muy recomendado)
  • registrar el problema, usando los servicios de Mensajes de trazado, o herramienta similar
  • enviar un correo describiendo el problema a un administrador

Decidir qué hacer exactamente parece depender de la naturalez del problema. Si hay un error real en el programa - un defecto que necesita ser corregidos - entonces uno podría hacer las tres de arriba. En este caso, es probable que al usuario final se le muestre un mensaje genérico "Lo sentimos, hemos cometido un error", no un seguimiento de la pila. Es usualmente considerado una mala forma el visualizar una traza de pila a un usuario final sin conocimientos técnicos, o si exponer una traza de pila puede ser un riesgo para la seguridad.

Si la excepción no representa un error, entonces puede ser apropiado un comportamiento diferente. Por ejemplo, si se detecta un problema con la entrada del usuario y, como resultado, se lanza una excepción, lo único que se requiere es informar al usuario del problema.
