Mantenga siempre HttpSessions

Si está usando sesiones, debe asegurarse que están siendo mantenidas apropiadamente.

Las cookies son el mecanismo usual para implementar sesiones. Sin embargo, el navegador del usuario puede tener las cookies deshabilitadas. En ese caso, es posible que desee recurrir a la reescritura de URL para asegurarse de que las sesiones aún se mantengan. (Vea este tema por una razón por la que puede no querer hacer esto). La reescritura de URL inserta un identificador de sesión dentro de las URLs encontradas en la página web.

Si decide usar reescritura de URL, entonces necesitará asegurarse de que se aplica a cada enlace (y atributo action del formulario) en cada página que pueda formar parte de una sesión. Si emite por error un enlace que no tiene reescritura de URL, entonces sus usuarios perderán su información de sesión si hacen click en ese enlace.

Para permitir esto, una aplicación debe definir una política para reescribir URLs. Por ejemplo, uno podría decidir usar siempre las etiquetas JSTL <c:url> y <c:redirect> (o herramientas similares) para emitir todos los enalces en una aplicación. Estas etiquetas automáticamente reescribirán las URLs cuando se necesite. (Otra opción podría ser un servlet filtro que automáticamente escanea la salida por enlaces).

La codificación de enlaces "sin procesar" siempre es arriesgada, ya que nunca puede permitir la reescritura de URL.
