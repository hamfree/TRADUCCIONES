Refactorize los Controladores grandes

Las implementaciones de HttpServlet son algunas veces informalmente llamadas clases Controller (o clases Action). Forman el punto de entrada de las peticiones HTTP del cliente en una aplicación web.

Un problema común con los Controladores es que se vuelven muy grandes. Ya que el flujo de control empieza en el Controlador, hay una fuerte tendencia a seguir agregando más y más código en el mismo lugar. Pero eso es usualmente un error. Realmente no debería pensar que el controlador tiene el control. Por el contrario, debería pensar en el Servicio como si tuviera el control. (Desde esta perspectiva, el término Controlador es un nombre poco apropiado).

Aquí tiene un resumen de la distinción entre Controlador y Servicio:

  • el Servicio tiene las cosas más importantes: acceso a la base de datos, y toda la lógica de datos asociada y las operaciones con los datos. (En este contexto, la palabra Servicio significa incluir Objetos de Acceso a Datos).
  • el Controlador tiene las cosas menos importantes: acceso a la petición HTTP entrante, y la respuesta HTTP saliente.
  • El trabajo del Controlador es ser un mecanismo de entrada-salida para el Servicio: traduce las peticiones de HTTP entrantes en datos utilizables, y reenvía los datos de salida a una JSP para una respuesta HTTP final.
  • el Servicio ignora por completo HTTP.
  • para comprobar, el Servicio solo necesita acceso a la base de datos; no necesita HTTP.
Como tal, las clases Controladores deberían ser bastante compactas. Si los métodos de su Controlador se vuelven grandes, o tienen lógica de servicio en ellos, entonces debe considerar mover ese código a una clase de Servicio de alguna clase.

(La distinción entre Controlador y Servicio puede ser más que simplemente un deseo para un diseño limpio. Cuando usa el marco de trabajo Spring, hay en realidad algunas importants diferencias funcionales entre las clases Controller y Service).
