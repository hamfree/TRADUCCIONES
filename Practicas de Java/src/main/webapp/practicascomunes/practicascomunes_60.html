Use javadoc generosamente

Javadoc es una gran herramienta, y debe usarse con sentimientos de alegría desenfrenada;).

Una cabecera de método combinado con su javadoc asociado forma la especificación, o el contrato, de un método. Si el llamador cumple con los requerimientos indicados, entonces el método se compromete a cumplir sus promesas declaradas.

El uso de Javadoc reconoce que hay dos preguntas distintas que un lector puede hacer sobre el código:

  • ¿Qué se supone que debe hacer esto? (contestado solo por el javadoc y cabecera de método)
  • ¿Cómo intenta hacerlo? (contestado solo por la implementación)

Si el javadoc está escrito correctamente, entonces:

  • Uno puede entender exactamente qué servicios son ofrecidos por un método ("¿Qué se supone que debe hacer esto?"), sin tener que mirar su implementación ("¿Cómo intenta hacerlo?"). Leer una implementación suele requerir mucho más esfuerzo que leer javadoc.
  • la implementación se puede comprobar para verificar que sea correcta en comparación con la especificación. Es decir, algunos errores pueden ser encontrados simplemente leyendo el código, en lugar de ejecutarlo.
Oracle ha publicado esta guía de estilo para escribir comentarios de javadoc.

Tenga en cuenta que:

  • si desea proporcionar descripciones de alto nivel, entonces puede usar los nombres de archivo overview.html (un nombre convencional) y package-info.java para hacer eso (mire abajo).
  • javadoc es "heredado" - javadoc escrito para un método de interfaz o un método abstracto se asocia automáticamente con las implementaciones concretas correspondientes
  • es un error común colocar a nivel de clase los comentarios de javadoc al comienzo de un archivo fuente, antes de las sentencias import. Esto es un error, ya que debe aparecer justo encima de la cabecera de la clase.
Javadoc 1.4 ha agregado muchas funcionalidades, incluyendo:
  • la opción -tag permite etiquetas personalizadas definidas por el usuario. Esta funcionalidad se puede usar para implementar elementos comunes tales como @to.do, @is.Mutable, o @no.Nulls. (Oracle recomienda colocar un punto en alguna parte en la etiqueta personalizada, para evitar potenciales conflictos futuros con etiquetas definidas por Oracle).
  • La opción -linksource genera una versión HTML de su código fuente, y enlaces en el javadoc al fuente. (La fuente se presenta sin ningún resaltado de sintaxis, pero esta sigue siendo una característica muy agradable).
  • existe un control más detallado sobre cómo se hereda javadoc. La etiqueta {@inheritDoc} es útil aquí (advertencia: esto no funciona en JDK 1.4.0)
Javadoc 1.5 tiene estas notables adiciones:
  • soporte para nuevas funcionalidades del lenguaje, tales como genéricos y anotaciones
  • las etiquetas @literal y @code para asegurar que el texto no se trata como marcado

Prefiera package-info.java a package.html

Originalmente, un archivo HTML llamado package.html fue usado para especificar comentario a nivel de paquete. Ese estilo está aún disponible, pero probablemente sea mejor usar package-info.java en su lugar:

  • es más corto, ya que no se necesita HTML repetitivo
  • el '*' inicial de cada línea se puede descartar, lo que facilita la tarea de pegar en HTML existente
  • package-info.java es el único lugar donde anotar un paquete, si se necesita
El archivo package-info.java es inusual desde que no contiene una clase. De hecho, el nombre package-info no es un nombre de clase válido. A continuación, se muestra un ejemplo sencillo de su contenido: 
/** Editar los roles adjuntos a un usuario. */
package hirondelle.fish.access.role;
Aquí tiene un ejemplo incluyendo una anotación para el paquete: 
/** Editar las configuraciones adjuntadas a un usuario. */
@Unpublished
package hirondelle.fish.access.user;
Aquí tiene un ejemplo final sin el '*' inicial en cada línea: 
/**
Mantiene una lista de Miembros en el Club Pescado y Patatas Fritas.
Un Miembro Activo será parte de la lista RSVP. Cuando un Miembro 
está inactivo, no aparecerá en el listado RSVP.
@author B. P. Hennessey
@version 1.0
*/
package hirondelle.fish.main.member;
