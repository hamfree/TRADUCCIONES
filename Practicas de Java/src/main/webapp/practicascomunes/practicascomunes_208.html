Intente alternativas a ResourceBundle

Hay algunos problemas bastantes inquietantes con ResourceBundle:
  • Si las traducciones se almacenan en una base de datos, no hay forma de que una aplicación cree una nuev ubicación sin cambios en el código (es decir, sin crear una nueva clase ResourceBundle respaldada por la base de datos para cada nueva Locale).
  • antes del KDJ 6, no había forma de refrescar los datos subyacentes en tiempo de ejecución.

Estos son defectos serios.

Además, hay muchas molestias asociadas con el uso de ResourceBundle y sus archivos de propiedades asociados:

  • antes del KDJ 9, la codificación utilizada en un fichero de propiedades era ISO-8859-1, no UTF-8. Si la aplicación estaba usando un idioma que no usaba ISO-8859-1, entonces se tenía que usar la herramienta native2ascii para procesar los ficheros. (Bastante descortés para un mecanismo de internacionalización, ¿no es así?)
  • claves de texto natural versus claves de codificador: desde el punto de vista del llamador, usar una clave de búsqueda por texto natural tiene la ventaja de que incrementa la legibilidad. Sin embargo, tal clave de búsqueda aparece como blah=blah en un archivo de propiedades. Tal duplicación hace que la mayoría de los programadores se pongan justificablemente nerviosos. (Si se usa una base de datos, un ResultSet teniendo la misma estructura ame blah=blah puede ser fácilmente creado, pero sin ningún dato real repetido en las tablas subyacentes).
  • en un archivo de propiedades, no hay una forma fácil de encontrar duplicados. En general, no hay una forma fácil de realizar ninguna clase de consulta sobre los datos (como sí la hay en una base de datos).
  • enviar archivos de propiedades sin formato a los traductores no parece deseable, pero crear una herramienta de interfaz para dichos archivos no es fácil, por lo que nunca se hace
  • si un elemento tiene más de una línea, entonces el caracter de continuación de línea "\\" debe ser agregado a cada línea
  • los caracteres de apóstrofe son un problema: la clase MessageFormat trata los apóstrofes como caracteres especiales. Si un apóstrofe aparece en un patrón pasado a un MessageFormat, este se debe escapar, o si no se lanza una pequeña y bonita RuntimeException. Si envía archivos de propiedades a los traductores, la probabilidad de errores relacionados con los apóstrofes is muy alta. Si se usa una base de datos en vez de archivos de propiedades, sin embargo, entonces es bastante fácil construir una herramienta de interfaz que pueda hacer las necesarias validaciones (Construir tal herramienta es especialmente atractivo cuando se puede usar con múltiples proyectos).
  • uso de nombres de módulo: el mecanismo ResourceBundle permite al llamador especificar un nombre de módulo, que es vinculado a su nombre de clase correspondiente o nombre del archivo de propiedades. Es posible cuestionar seriamente este diseño. En general, un método no debe saber la identidad de su llamador.

En resumen, parece que ResourceBundle se diseñó originalmene para aplicaciones de escritorio y archivos de propiedades, y que su diseño no es suficientemente robusto para un uso fácil en el contexto de una aplicación web respaldada por una base de datos.

Afortunadamente, la traducción de texto no es una tarea difícil, y crear una alternativa a ResourceBundle es en realidad bastante fácil. Para un ejemplo, por favor vea el paquete hirondelle.web4j.ui.translate de la herramienta WEB4J. Este usa una técnica llamada "localización de último segundo".
