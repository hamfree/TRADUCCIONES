Valide los argumentos del método

Las primeras líneas de un método generalmente se dedican a verificar la validez de los argumentos del método. La idea es fallar tan rápidamente como sea posible en el caso de un error. Esto es particularmente importante para los constructores.

Es una política razonable para una clase saltarse la validación de argumentos de los métodos private. La razón es que los métodos private pueden solo llamarse desde la misma clase. Así, el autor de una clase debería ser capaz de confirmar que todas las llamadas de un método private son válidas. Si se desea, la palabra clave assert se puede usar para verificar los argumentos de los métodos private, para comprobar la consistencia interna de la clase. (Usar assert para comprobar los argumentos de un método no-private no se recomienda, ya que deshabilitando tales aserciones significaría que el contrato del método no-private ya no se aplica).

Cuando se valida un argumento, se lanza una Exception si la prueba falla. Es a menudo una de estas excepciones no comprobadas las que se lanzan:

  • IllegalArgumentException
  • NullPointerException
  • IllegalStateException
Estas son todas las subclases de RuntimeException.

También se pueden lanzar excepciones comprobadas, como se mostró en el tema Validación del Modelo de Objetos. Muchos programadores documentan estas excepciones en la cláusula @throws del javadoc del método, ya que indican claramente los requisitos del método al llamador (las condiciones previas). Otros, sin embargo, sienten que documentar manualmente las excepciones debe ser evitado.

Si cada parámetro de objeto de cada método en una clase necesita ser no-nulo para evitar lanzar una NullPointerException, entonces es aceptable indicar esto una vez en el javadoc general de la clase, en vez de repetirlo para cada método. Otra alternativa es indicar en overview.html (descripción resumida de javadoc de una aplicación completa) que todos los parámetros van a ser considerados no nulos a menos que se indique explícitamente otra cosa.

Ejemplo 1

El constructor de esta clase valida sus argumentos antes de hacer cualquier otra cosa. Si una validación falla, entonces se lanza una IllegalArgumentException. 

public final class ElementaryParticle {

  /**
   @param aName tiene contenido.
   @param aSpeed esta en el rango 0 (inclusive) a 1 (inclusive), y 
   se expresa como una fracción de la velocidad de la luz. (El fotón es 
   un ejemplo de partícula elemental que viaja a esta velocidad).
   @exception IllegalArgumentException si un parámetro no cumple.
  */
  public ElementaryParticle (String aName, double aSpeed) {
    if (!textHasContent(aName)) {
      throw new IllegalArgumentException("Name no tiene contenido.");
    }
    if (aSpeed < 0.0 || aSpeed > 1.0) {
      throw new IllegalArgumentException("Speed no está en el rango [0..1]: " + aSpeed);
    }
    fName = aName;
    fSpeed = aSpeed;
  }

  //..otros métodos omitidos

  // PRIVADO
  private String fName;
  private double fSpeed;

  /**
    Devuelve verdadero si aText es no nulo y tiene contenido visible.
  
    Esto es una prueba que se realiza a menudo, y probablemente 
    debe ser colocada en una clase de utilidad general.
  */
  private boolean textHasContent(String aText){
    String EMPTY_STRING = "";
    return (aText != null) && (!aText.trim().equals(EMPTY_STRING));
  }
}

Ejemplo 2

Algunas validaciones son muy comunes:

  • comprobar que un objeto es no null
  • comprobar que el texto tiene contenido visible
  • comprobar que un número está dentro de un rango
Proporcionar una clase para tales validaciones puede a menudo ser útil. En este ejemplo, la clase Args lanza IllegalArgumentException si una comprobación de booleanos falla. Usando Args para validar argumentos incrementará la legibilidad noticiablemente, especialmente en el caso de validaciones múltiples. 
package hirondelle.web4j.util;

import java.util.regex.*;

/**
 Métodos de utilidad para validaciones comunes de argumentos.

<P>Reemplaza las sentencias <tt>if</tt> al inicio de un método con 
 llamadas a método más compactas.
 
 <P>Ejemplo de caso de uso.
 <P>En vez de :
 <PRE>
 public void doThis(String aText){
   if (!Util.textHasContent(aText)){
     throw new IllegalArgumentException();
   }
   //..cuerpo principal omitido
 }
 </PRE>
 <P>Uno puede en su lugar escribir :
 <PRE>
 public void doThis(String aText){
   Args.checkForContent(aText);
   //..cuerpo principal omitido
 }
 </PRE>
*/
public final class Args {
  
  /**
   Si <code>aText</code> no satisface {@link Util#textHasContent}, entonces 
   lanza una <code>IllegalArgumentException</code>.
  
   <P>La mayoría del texto usado en una aplicación tiene sentido solo si este tiene contenido visible.
  */
  public static void checkForContent(String aText){
    if( ! Util.textHasContent(aText) ){
      throw new IllegalArgumentException("El texto no tiene contenido visible");
    }
  }

  /**
   Si {@link Util#isInRange} devuelve <code>false</code>, entonces
   lanza una <code>IllegalArgumentException</code>. 
  
   @param aLow es menor que o igual a <code>aHigh</code>.
  */
  public static void checkForRange(int aNumber, int aLow, int aHigh) {
    if ( ! Util.isInRange(aNumber, aLow, aHigh) ) {
      throw new IllegalArgumentException(aNumber + " no está en el rango " + aLow + ".." + aHigh);
    }
  }

  /**
   Si <tt>aNumber</tt> es menor que <tt>1</tt>, entonces lanza una 
   <tt>IllegalArgumentException</tt>.
  */
  public static void checkForPositive(int aNumber) {
    if (aNumber < 1) {
      throw new IllegalArgumentException(aNumber + " es menor que 1");
    }
  }

  /**
   Si {@link Util#matches} devuelve <tt>false</tt>, entonces 
   lanza una <code>IllegalArgumentException</code>. 
  */
  public static void checkForMatch(Pattern aPattern, String aText){
    if (! Util.matches(aPattern, aText)){
      throw new IllegalArgumentException(
        "Text " + Util.quote(aText) + " no coincide con '" +aPattern.pattern()+ "'"
      );
    }
  }
  
  /**
   Si <code>aObject</code> es nulo, entonces lanza una <code>NullPointerException</code>.
  
   <P>Casos de uso :
  <pre>
   doSomething( Football aBall ){
     //1. llama algún método sobre el argumento : 
     //si aBall es nulo, entonces se lanza automáticamente la excepción, de forma que 
     //no hay necesidad de una comprobación explícita del nulo.
     aBall.inflate();
    
     //2. asignar a un campo correspondiente (común en constructores): 
     //si aBall es nulo, no se lanza inmediatamente la excepción, de forma que 
     //una comprobación explícita por nulo puede ser útil aquí
     Args.checkForNull( aBall );
     fBall = aBall;
     
     //3. pasar a algún otro método como parámetro : 
     //puede ser o no apropiado tener una comprobación explícita 
     //del nulo aquí, de acuerdo a las necesidades del problema
     Args.checkForNull( aBall ); //??
     fReferee.verify( aBall );
   }
   </pre>
  */
  public static void checkForNull(Object aObject) {
    if (aObject == null) {
      throw new NullPointerException();
    }
  }
  
  // PRIVADO
  private Args(){
    //vacío - previene la construcción
  }
}
