Los campos deberían ser usualmente privados

Los campos deben ser declarados private a menos que haya una buena razón para no hacer eso.

Uno de los principios guía de último valor en programación es "minimizar los efectos dominantes manteniendo secretos." Cuando un campo es private, el llamador normalmente no puede obtener un acceso directo inapropiado al campo.

Para aclarar, hay tres posibilidades para un campo private:

  • datos primitivos tales como int, boolean, y así.
  • objetos inmutables tales como String y BigDecimal, cuyos datos nunca cambian.
  • objetos mutables, cuyos datos pueden cambiar (es decir, ellos pueden cambiar su estado). Estos necesitan un cuidado especial, y a menudo necesitan una copia defensiva. Es fácil olvidar esto. Siempre recuerde que los campos de objetos mutables son 'puntos calientes' en su código, y necesitan especial cuidado.

Es solo el último caso - el campo private de un objeto mutable - donde el acceso directo inapropiado a un campo private es posible. Aquí tiene una ilustración:

Ejemplo 1

import java.util.Date;

/**
* Planet es una clase inmutable, ya que no hay forma de cambiar 
* su estado después de la construcción.
*/
public final class Planet {

  public Planet (double aMass, String aName, Date aDateOfDiscovery) {
     fMass = aMass;
     fName = aName;
     //hace una copia privada de aDateOfDiscovery
     //esta es la única forma de mantener el campo fDateOfDiscovery
     //privado, y blinda a esta clase de cualquier cambio que 
     //el llamante pueda hacer al objeto original aDateOfDiscovery
     fDateOfDiscovery = new Date(aDateOfDiscovery.getTime());
  }

  /**
  * Devuelve un valor primitivo.
  *
  * El llamador puede hacer lo que quiera con el valor devuelto, sin 
  * afectar las interioridades de esta clase. ¿Porqué? Porque este es un valor 
  * primitivo. El llamador ve su "propio" double que simplemente tiene el 
  * mismo valor que fMass.
  */
  public double getMass() {
    return fMass;
  }

  /**
  * Devuelve un objeto inmutable.
  *
  * El llamador obtiene una referencia directa del campo interno. Pero esto no es 
  * peligroso, ya que String es inmutable y no puede ser cambiada.
  */
  public String getName() {
    return fName;
  }

//  /**
//  * Devuelve un objeto mutable - probablemente mal estilo
//  *
//  * El llamador obtiene una referencia directa al campo interno. Esto es habitualmente peligroso, 
//  * ya que el estado del objeto Date puede ser cambiado tanto por esta clase como su llamador.
//  * Es decir, esta clase ya no tiene más el control completo de fDate.
//  */
//  public Date getDateOfDiscovery() {
//    return fDateOfDiscovery;
//  }

  /**
  * Devuelve un objeto mutable - buen estilo.
  *
  * Devuelve una copia defensiva del campo.
  * El llamador de este método puede hacer lo que quiere con el 
  * objeto devuelto Date, sin afectar las interioridades de esta 
  * clase de ninguna forma. ¿Porqué? Porque no tiene una referencia a 
  * fDate. En su lugar, está jugando con una segunda Date que inicialmente tiene los 
  * mismos datos que fDate.
  */
  public Date getDateOfDiscovery() {
    return new Date(fDateOfDiscovery.getTime());
  }

  // PRIVADO

  /**
  * Los datos primitivos finales son siempre inmutables.
  */
  private final double fMass;

  /**
  * Un campo de objeto inmutable. (Los objetos String nunca cambian su estado).
  */
  private final String fName;

  /**
  * Un campo de objeto mutable. En este caso, el estado de esta campo mutable 
  * va a ser cambiado solo por esta clase. (En otros casos, tiene perfecto 
  * sentido permitir que sea cambiado el estado de un campo fuera de la clase 
  * nativa; este es el caso cuando un campo actúa como un "puntero" a un objeto 
  * creado en otra parte).
  *
  * En código nuevo, debe usar clases de java.time, no de java.util.Date.
  */
  private final Date fDateOfDiscovery;
}
La clase de arriba usa una copia defensiva como parte de su diseño, pero hay casos en los que las copias defensivas no se desean. El punto es que para campos mutables, necesita saber la diferencia, y hacer la elección apropiada.
