Evite null si es posible

Algunos argumentan que permitir null dentro de la IPA de una clase, en la forma de parámetros posiblemente null o valores devueltos, debe ser evitado si es posible:
  • las referencias null solo pueden usarse en pruebas booleanas - cualquier otro uso lanzará una NullPointerException
  • las referencias null siempre representan casos especiales, y si se eliminan estos casos especiales, el código será más fácil de entender
  • los valores devueltos null son propensos a errores, ya que no hay forma de garantizar que el llamante siempre haga una verificación de null.
  • los métodos que aceptan una referencia explícita null como un argumento válido son menos intelegibles en el punto de llamada. El lector necesitará usualmente consultar la documentación para determinar qué significa el argumento null. (En vez de pasar un null literal, por lo general, es más útil para el lector pasar una variable de objeto null bien denominada en su lugar).
Si se permite null para un método, entonces esto debe ser claramente indicado en su javadoc. Algunos siguen la convención de que se asume que todos los elementos son no null a menos que se indique lo contrario. Tal convención puede ser indicada explícitamente una vez en el overview.html de javadoc.

Hay una excepción muy común a esta regla. Para el Modelo de Objetos, que se asignan aproximadamente a los registros de la base de datos, es a menudo apropiado usar null para representar campos opcionales almacenados como NULL en la base de datos.

La clase Objects tiene métodos amistosos con null diseñados para ayudarle a trabajar con objetos null.
