Evite los estilos de error básicos

Muchos principiantes en Java repiten los mismos errores básicos de estilo. Tales errores de estilo no hacen su programa incorrecto, pero hacen su programa menos mantenible. En esencia, muchos principiantes escriben código que sufre del mismo defecto básico subyacente o no está siendo escrito con compasión por el lector.

Hay en realidad dos plataformas objetivo para todo el código: el hardware de tiempo de ejecución, y el cortex cerebral humano. Todos los códigos se 'ejecutan' en el cerebro humano, en el sentido de que todo el código necesita ser entendido por un ser humano. El estilo de codificación siempre se preocupa por esta segunda plataforma de destino, no por la primera.

Algunos errores de estilo básicos comunes:

  • clases demasiado largas
  • métodos demasiado largos
  • poca o ninguna javadoc
  • sin convención para distinguir entre variables locales, argumentos, y campos
  • muchos bloques catch vacíos que suprimen las excepciones
  • uso inapropiado de multiple sentencias de retorno
  • usar excepciones para definir el flujo regular del programa
  • uso excesivo del operador instanceof
  • usar datos de punto flotante para representar dinero
  • preferir matrices sobre colecciones (especialmente en KDJ 5+)
  • no ordenar los miembros de la clase por alcance

Algunos errores de estilo básico son mucho más dañinos para su código que otros. El mal hábito más dañino es el de la longitud excesiva. Las clases que son demasiado largas son difíciles de entender. Como pauta, si una clase tiene más de 300 líneas de largo, debería considerar dividirla en piezas más pequeñas. De forma similar, los métodos que ocupan más de una única pantalla son muy probablemente demasiado largos y casi siempre se beneficiarían al dividirse en varios métodos. No es raro que un método útil consista solo de una línea de código.
