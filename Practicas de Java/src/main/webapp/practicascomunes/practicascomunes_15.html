Copia defensiva

Un objeto mutable es simplemente un objeto que puede cambiar su estado después de la construcción. Por ejemplo, StringBuilder y Date son objetos mutables, mientras que String y Integer son objetos inmutables.

Una clase puede tener un objeto mutable como un campo. Hay dos casos posibles para cómo el estado de un campo de objeto mutable puede cambiar:

  • su estado puede ser cambiado solo por la clase nativa - la clase nativa crea el campo del objeto mutable, y es solo la clase la que está enterada directamente de su existencia
  • su estado puede ser cambiado tanto por la clase nativa y por sus llamantes - la clase nativa simplemente apunta a un objeto mutable que fue creado en cualquier otra parte
Ambos casos son elecciones válidas de diseño, pero debe tener cuidado de cual de ellas es apropiada para cada caso.

Si el estado del campo del objeto mutable debe ser cambiado solo por la clase nativa, entonces se debe hacer una copia defensiva del objeto mutable cada vez que se pasa a (constructores y métodos set) o fuera de (métodos get) la clase. Si esto no se hace, entonces es simple para el llamador romper la encapsulación, cambiando el estado de un objeto que es visible simultáneamente a ambas, la clase y su llamador.

Ejemplo

Planet tiene un campo con un objeto mutable fDateOfDiscovery, que es copiado defensivamente en todos los constructores, y en getDateOfDiscovery. Planet representa una clase inmutable, y no tiene métodos set para sus campos. Tenga en cuenta que si no se hace la copia defensiva de DateOfDiscovery, ¡entonces Planet ya no es inmutable!

import java.util.Date;

/**
* Planet es una clase inmutable, ya que no hay forma de cambiar
* su estado después de la construcción.
*/
public final class Planet {

  public Planet (double aMass, String aName, Date aDateOfDiscovery) {
     fMass = aMass;
     fName = aName;
     //hace una copia privada de aDateOfDiscovery
     //esta es la única forma de mantener el campo fDateOfDiscovery
     //privado, y blinda esta clase de cualquier cambio que 
     //el llamador puede hacer al objeto original aDateOfDiscovery
     fDateOfDiscovery = new Date(aDateOfDiscovery.getTime());
  }

  /**
  * Devuelve un valor primitivo.
  *
  * El llamador puede hacer lo que quiera con el valor devuelto, sin 
  * afectar a las interioridades de esta clase. ¿Porqué? Porque esto es un valor 
  * primitivo. El llamador ve su "propio" double que simplemente tiene el 
  * mismo valor que fMass.
  */
  public double getMass() {
    return fMass;
  }

  /**
  * Devuelve un objeto inmutable.
  *
  * El llamador obtiene una referencia directa al campo interno. Pero esto no es 
  * peligroso, ya que String es inmutable y no puede ser cambiada.
  */
  public String getName() {
    return fName;
  }

//  /**
//  * Devuelve un objeto mutable - probablemente mal estilo.
//  *
//  * El llamante obtiene una referencia directa al campo interno. Esto es habitualmente peligroso, 
//  * ya que el estado del objeto Date puede ser cambiado tanto por esta clase y su llamador.
//  * Esto es, esta clase ya no controla completamente a fDate.
//  */
//  public Date getDateOfDiscovery() {
//    return fDateOfDiscovery;
//  }

  /**
  * Devuelve un objeto mutable - buen estilo.
  * 
  * Devuelve una copia defensiva del campo.
  * El llamador de este método puede hacer lo que quiera con el 
  * objeto Date devuelto, sin afectar las interioridades de esta 
  * clase de ninguna forma. ¿Porqué? Porque no tiene una referencia a 
  * fDate. En su lugar, está jugando con una segunda Date que inicialmente tiene los 
  * los mismos datos que fDate.
  */
  public Date getDateOfDiscovery() {
    return new Date(fDateOfDiscovery.getTime());
  }

  // PRIVADO

  /**
  * Los datos primitivos finales son siempre inmutables.
  */
  private final double fMass;

  /**
  * Un campo de objeto inmutable. (Los objetos String nunca cambian su estado).
  */
  private final String fName;

  /**
  * Un campo de objeto mutable. En este caso, el estado de este campo mutable
  * va a ser cambiado solo por esta clase. (En otros casos, tienen perfecto 
  * sentido permitir que se cambie el estado de un campo fuera de la clase 
  * nativa; este es el caso cuando un campo actúa como un "puntero" a un objeto 
  * creado en cualquier otra parte).
  *
  * En código nuevo, debe usar las clases de java.time, no java.util.Date.
  */
  private final Date fDateOfDiscovery;
}
