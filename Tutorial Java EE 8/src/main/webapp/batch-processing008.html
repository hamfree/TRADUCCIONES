La Aplicación de Ejemplo webserverlog

La aplicación de ejemplo webserverlog, ubicada en el directorio tut-install`/examples/batch/webserverlog/`, demuestra cómo usar el marco de trabajo por lotes en Java EE para analizar el archivo de registro de un servidor web. Esta aplicación de ejemplo lee un archivo de registro y encuentra qué porcentaje de páginas vistas desde tabletas son ventas de productos.

Aquí se tratan los siguientes temas:

  • Arquitectura de la Aplicación de Ejemplo webserverlog

  • Ejecución de la Aplicación de Ejemplo webserverlog

Arquitectura de la Aplicación de Ejemplo webserverlog

La aplicación de ejemplo webserverlog consta de los siguientes elementos:

  • Un archivo de definición de trabajo (webserverlog.xml) que usa el lenguaje de especificación de trabajo (JSL) para definir un trabajo por lotes con un paso de fragmento y un paso de tarea. El paso de fragmento actúa como un filtro y el paso de tarea calcula las estadísticas de las entradas restantes.

  • Un archivo de registro (log1.txt) que sirve como datos de entrada para el trabajo por lotes.

  • Dos clases de Java (LogLine y LogFilteredLine) que representan elementos de entrada y elementos de salida para el paso de fragmento.

  • Tres artefactos por lotes (LogLineReader, LogLineProcessor y LogFilteredLineWriter) que implementan el paso de fragmento de la aplicación. Este paso lee elementos del archivo de registro del servidor web, los filtra según el navegador web utilizado por el cliente y escribe los resultados en un archivo de texto.

  • Dos artefactos por lotes (InfoJobListener y InfoItemProcessListener) que implementan dos escuchas simples.

  • Un artefacto por lotes (MobileBatchlet.java) que calcula estadísticas sobre los elementos filtrados.

  • Dos páginas Facelets (index.xhtml y jobstarted.xhtml) que proporcionan el front-end de la aplicación por lotes. La primera página muestra el archivo de registro que procesará el trabajo por lotes y la segunda página permite al usuario comprobar el estado del trabajo y muestra los resultados.

  • Un bean administrado (JsfBean) al que se accede desde las páginas de Facelets. El bean envía el trabajo al tiempo de ejecución por lotes, verifica el estado del trabajo y lee los resultados de un archivo de texto.

El Archivo de Definición de Trabajo

El archivo de definición de trabajo webserverlog.xml se encuentra en el directorio WEB-INF/classes/META-INF/batch-jobs/. El archivo especifica siete propiedades a nivel de trabajo y dos pasos: 

                  
<?xml version="1.0" encoding="UTF-8"?>
<job id="webserverlog" xmlns="http://xmlns.jcp.org/xml/ns/javaee"
     version="1.0">
    <properties>
        <property name="log_file_name" value="log1.txt"/>
        <property name="filtered_file_name" value="filtered1.txt"/>
        <property name="num_browsers" value="2"/>
        <property name="browser_1" value="Tablet Browser D"/>
        <property name="browser_2" value="Tablet Browser E"/>
        <property name="buy_page" value="/auth/buy.html"/>
        <property name="out_file_name" value="result1.txt"/>
    </properties>
    <listeners>
        <listener ref="InfoJobListener"/>
    </listeners>
    <step id="mobilefilter" next="mobileanalyzer"> ... </step>
    <step id="mobileanalyzer"> ... </step>
</job>

El primer paso se define de la siguiente manera:

                  
<step id="mobilefilter" next="mobileanalyzer">
    <listeners>
        <listener ref="InfoItemProcessListeners"/>
    </listeners>
    <chunk checkpoint-policy="item" item-count="10">
        <reader ref="LogLineReader"></reader>
        <processor ref="LogLineProcessor"></processor>
        <writer ref="LogFilteredLineWriter"></writer>
    </chunk>
</step>

Este paso es un paso de fragmento normal que especifica los artefactos por lotes que implementan cada fase del paso. Los nombres de los artefactos por lotes no son nombres de clase completos, por lo que los artefactos por lotes son beans CDI anotados con @Named.

El segundo paso se define de la siguiente manera:

                  
<step id="mobileanalyzer">
    <batchlet ref="MobileBatchlet"></batchlet>
    <end on="COMPLETED"/>
</step>

Este paso es un paso de tarea que especifica el artefacto por lotes que lo implementa. Este es el último paso del trabajo.

Los Elementos LogLine y LogFilteredLine

La clase LogLine representa entradas en el archivo de registro del servidor web y se define de la siguiente manera: 

                  
public class LogLine {
    private final String datetime;
    private final String ipaddr;
    private final String browser;
    private final String url;
    /* ... Constructor, getters, and setters ... */
}

La clase LogFileteredLine es similar a esta clase pero solo tiene dos campos: la dirección IP del cliente y la URL.

Los Artefactos por Lotes del Paso de Fragmento

El primer paso se compone de los artefactos por lotes LogLineReader, LogLineProcessor y LogFilteredLineWriter.

El artefacto LogLineReader lee registros del archivo de registro del servidor web: 

                  
@Dependent
@Named("LogLineReader")
public class LogLineReader implements ItemReader {
    private ItemNumberCheckpoint checkpoint;
    private String fileName;
    private BufferedReader breader;
    @Inject
    private JobContext jobCtx;
    public LogLineReader() { }
    /* ... Override the open, close, readItem, and
     *     checkpointInfo methods ... */
}

El método open lee la propiedad log_file_name y abre el archivo de registro con un lector almacenado en búfer. En este ejemplo, el archivo de registro se ha incluido con la aplicación en webserverlog/WEB-INF/classes/log1.txt: 

                  
fileName = jobCtx.getProperties().getProperty("log_file_name");
ClassLoader classLoader = Thread.currentThread().getContextClassLoader();
InputStream iStream = classLoader.getResourceAsStream(fileName);
breader = new BufferedReader(new InputStreamReader(iStream));

Si se proporciona un objeto de punto de control, el método open hace avanzar al lector hasta el último punto de control. De lo contrario, este método crea un nuevo objeto de punto de control. El objeto de punto de control realiza un seguimiento del número de línea desde el último fragmento confirmado.

El método readItem devuelve un nuevo objeto LogLine o nulo al final del archivo de registro: 

                  
@Override
public Object readItem() throws Exception {
    String entry = breader.readLine();
    if (entry != null) {
        checkpoint.nextLine();
        return new LogLine(entry);
    } else {
        return null;
    }
}

El artefacto LogLineProcessor obtiene una lista de navegadores de las propiedades del trabajo y filtra las entradas de registro según la lista: 

                  
@Override
public Object processItem(Object item) {
    /* Obtain a list of browsers we are interested in */
    if (nbrowsers == 0) {
        Properties props = jobCtx.getProperties();
        nbrowsers = Integer.parseInt(props.getProperty("num_browsers"));
        browsers = new String[nbrowsers];
        for (int i = 1; i < nbrowsers + 1; i++)
            browsers[i - 1] = props.getProperty("browser_" + i);
    }
    LogLine logline = (LogLine) item;
    /* Filter for only the mobile/tablet browsers as specified */
    for (int i = 0; i < nbrowsers; i++) {
        if (logline.getBrowser().equals(browsers[i])) {
            return new LogFilteredLine(logline);
        }
    }
    return null;
}

El artefacto LogFilteredLineWriter lee el nombre del archivo de salida de las propiedades del trabajo. El método open abre el archivo para escritura. Si se proporciona un objeto de punto de control, el artefacto continúa escribiendo al final del archivo; de lo contrario, sobrescribe el archivo si existe. El método writeItems escribe elementos filtrados en el archivo de salida: 

                  
@Override
public void writeItems(List<Object> items) throws Exception {
    /* Write the filtered lines to the output file */
    for (int i = 0; i < items.size(); i++) {
        LogFilteredLine filtLine = (LogFilteredLine) items.get(i);
        bwriter.write(filtLine.toString());
        bwriter.newLine();
    }
}

El artefacto por lotes InfoItemProcessListener implementa la interfaz ItemProcessListener para pasos de bloques:

Las Páginas de JavaServer Faces

La página index.xhtml contiene un área de texto que muestra el registro del servidor web. La página proporciona un botón para que el usuario envíe el trabajo por lotes y navegue a la página siguiente: 

                  
<body>
    ...
    <textarea cols="90" rows="25"
              readonly="true">#{jsfBean.getInputLog()}</textarea>
    <p> </p>
    <h:form>
        <h:commandButton value="Start Batch Job"
                         action="#{jsfBean.startBatchJob()}" />
    </h:form>
</body>

Esta página llama a los métodos del bean administrado para mostrar el archivo de registro y enviar el trabajo por lotes.

La página jobstarted.xhtml proporciona un botón para verificar el estado actual del trabajo por lotes y muestra los resultados cuando finaliza el trabajo: 

                  
<p>Current Status of the Job: <b>#{jsfBean.jobStatus}</b></p>
<p>#{jsfBean.showResults()}</p>
<h:form>
    <h:commandButton value="Check Status"
                     action="jobstarted"
                     rendered="#{jsfBean.completed==false}" />
</h:form>

El Bean Gestionado

El bean administrado JsfBean envía el trabajo al tiempo de ejecución por lotes, verifica el estado del trabajo y lee los resultados de un archivo de texto.

El método startBatchJob envía el trabajo al tiempo de ejecución por lotes:

                  
/* Submit the batch job to the batch runtime.
 * JSF Navigation method (return the name of the next page) */
public String startBatchJob() {
    jobOperator = BatchRuntime.getJobOperator();
    execID = jobOperator.start("webserverlog", null);
    return "jobstarted";
}

El método getJobStatus comprueba el estado del trabajo:

                  
/* Get the status of the job from the batch runtime */
public String getJobStatus() {
    return jobOperator.getJobExecution(execID).getBatchStatus().toString();
}

El método showResults lee los resultados de un archivo de texto.

Ejecución de la Aplicación de Ejemplo webserverlog

Puede usar el EID de NetBeans o Maven para compilar, empaquetar, implementar y ejecutar la aplicación de ejemplo webserverlog.

Aquí se tratan los siguientes temas:

  • Para Ejecutar la Aplicación de Ejemplo webserverlog Usando el EID NetBeans

  • Para Ejecutar la Aplicación de Ejemplo webserverlog Usando Maven

Para Ejecutar la Aplicación de Ejemplo webserverlog Usando el EID NetBeans

  1. Asegúrese de que el servidor GlassFish se haya iniciado (consulte Arrancando y Parando el Servidor GlassFish).

  2. En el menú Archivo, elija Abrir proyecto.

  3. En el cuadro de diálogo Abrir proyecto, vaya a:

                            
tut-install/examples/batch

  4. Seleccione la carpeta webserverlog.

  5. Haga clic en Abrir proyecto.

  6. En la pestaña Proyectos, haga clic con el botón derecho en el proyecto webserverlog y seleccione Ejecutar.

    Este comando crea y empaqueta la aplicación en un archivo WAR, webserverlog.war, ubicado en el directorio target/; lo implementa en el servidor; y abre una ventana del navegador web en la siguiente URL: 

                            
http://localhost:8080/webserverlog/

Para Ejecutar la Aplicación de Ejemplo webserverlog Usando Maven

  1. Asegúrese de que el servidor GlassFish se haya iniciado (consulte Arrancando y Parando el Servidor GlassFish).

  2. En una ventana de terminal, vaya a:

                            
tut-install/examples/batch/webserverlog/

  3. Ingrese el siguiente comando para implementar la aplicación:

                            
mvn install

  4. Abra una ventana del navegador web en la siguiente URL:

                            
http://localhost:8080/webserverlog/
