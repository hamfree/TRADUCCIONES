IPAs de Java EE 8

Las siguientes secciones ofrecen un breve resumen de las tecnologías requeridas por la plataforma Java EE y las API utilizadas en las aplicaciones Java EE.

Tecnología Enterprise JavaBeans

Un componente Enterprise JavaBeans (EJB), o enterprise bean, es un cuerpo de código que tiene campos y métodos para implementar módulos de lógica empresarial. Puede pensar en un enterprise bean como un componente básico que se puede utilizar solo o con otros enterprise beans para ejecutar la lógica empresarial en el servidor Java EE.

Los enterprise beans son beans de sesión o beans controlados por mensajes.

  • Un bean de sesión representa una conversación transitoria con un cliente. Cuando el cliente termina de ejecutarse, el bean de sesión y sus datos desaparecen.

  • Un bean controlado por mensajes combina características de un bean de sesión y una escucha de mensajes, lo que permite que un componente empresarial reciba mensajes de forma asíncrona. Normalmente, estos son mensajes de Java Message Service (JMS).

La plataforma Java EE 8 requiere Enterprise JavaBeans 3.2 e Interceptors 1.2. La especificación de Interceptors es parte de la especificación EJB.

Tecnología de Servlets de Java

La tecnología Java Servlet le permite definir clases de servlet específicas de HTTP. Una clase de servlet amplía las capacidades de los servidores que albergan aplicaciones a las que se accede mediante un modelo de programación de solicitud-respuesta. Aunque los servlets pueden responder a cualquier tipo de solicitud, se usan comúnmente para extender las aplicaciones alojadas por servidores web.

En la plataforma Java EE 8, las nuevas características de la tecnología Java Servlet incluyen lo siguiente:

  • Server Push

  • Avances de HTTP

La plataforma Java EE 8 requiere Servlet 4.0

Tecnología JavaServer Faces

La tecnología JavaServer Faces es un marco de interfaz de usuario para crear aplicaciones web. Los principales componentes de la tecnología JavaServer Faces son los siguientes:

  • Un marco de componentes de IGU.

  • Un modelo flexible para renderizar componentes en diferentes tipos de HTML o diferentes lenguajes y tecnologías de marcado. Un objeto Renderer genera el marcado para representar el componente y convierte los datos almacenados en un objeto modelo en tipos que se pueden representar en una vista.

  • Un RenderKit estándar para generar marcado HTML 4.01.

Las siguientes funciones son compatibles con los componentes de la IGU:

  • Validación de entrada

  • Manejo de eventos

  • Conversión de datos entre componentes y objetos del modelo

  • Creación de objeto de modelo administrado

  • Configuración de navegación de página

  • Lenguaje de Expresión (EL)

Toda esta funcionalidad está disponible utilizando IPA estándar de Java y archivos de configuración basados en XML.

En la plataforma Java EE 8, las nuevas características de la tecnología JavaServer Faces incluyen las siguientes:

  • Soporte directo para WebSockets a través de la nueva etiqueta <f: websocket>

  • Validación de bean a nivel de clase a través de la nueva etiqueta <f:validateWholeBean>

  • Una anotación @ManagedProperty compatible con CDI

  • Marco de expresión de búsqueda de componentes mejorado

La plataforma Java EE 8 requiere JavaServer Faces 2.3 y Expression Language 3.0.

Para obtener un excelente resumen de las novedades de JSF 2.3, consulte https://javaserverfaces.github.io/users.html .

Tecnología JavaServer Pages

La tecnología JavaServer Pages (JSP) le permite colocar fragmentos de código de servlet directamente en un documento basado en texto. Una página JSP es un documento basado en texto que contiene dos tipos de texto:

  • Datos estáticos, que se pueden expresar en cualquier formato de texto, como HTML o XML

  • Elementos JSP, que determinan cómo la página construye contenido dinámico.

Para obtener información sobre la tecnología JSP, consulte el Tutorial de Java EE 5 en http://docs.oracle.com/javaee/5/tutorial/doc/ .

La plataforma Java EE 8 requiere JavaServer Pages 2.3 para ser compatible con versiones anteriores, pero recomienda el uso de Facelets como tecnología de visualización en nuevas aplicaciones.

Librería de Etiquetas Estándar de JavaServer Pages

La Librería de Etiquetas Estándar de JavaServer Pages (JSTL) encapsula la funcionalidad principal común a muchas aplicaciones JSP. En lugar de mezclar etiquetas de numerosos proveedores en sus aplicaciones JSP, utiliza un único conjunto estándar de etiquetas. Esta estandarización le permite implementar sus aplicaciones en cualquier contenedor JSP que admita JSTL y hace que sea más probable que la implementación de las etiquetas esté optimizada.

JSTL tiene etiquetas iteradoras y condicionales para manejar el control de flujo, etiquetas para manipular documentos XML, etiquetas de internacionalización, etiquetas para acceder a bases de datos usando SQL y etiquetas para funciones de uso común.

La plataforma Java EE 8 requiere JSTL 1.2.

IPA de Java Persistence

La IPA de Java Persistence (JPA) es una solución basada en estándares Java para la persistencia. La persistencia utiliza un enfoque de mapeo relacional/objeto para cerrar la brecha entre un modelo orientado a objetos y una base de datos relacional. La IPA de Java Persistence también se puede utilizar en aplicaciones Java SE fuera del entorno Java EE. La persistencia de Java consta de las siguientes áreas:

  • La IPA de Java Persistencia

  • El lenguaje de consulta

  • Metadatos de mapeo relacional/objeto

La plataforma Java EE 8 requiere la IPA de Java Persistence 2.1.

IPA de Java Transaction

La IPA de Java Transaction (JTA) proporciona una interfaz estándar para demarcar transacciones. La arquitectura Java EE proporciona una confirmación automática predeterminada para manejar las confirmaciones y reversiones de transacciones. Una confirmación automática significa que cualquier otra aplicación que esté viendo datos verá los datos actualizados después de cada operación de lectura o escritura de la base de datos. Sin embargo, si su aplicación realiza dos operaciones de acceso a la base de datos separadas que dependen una de la otra, querrá usar la API de JTA para demarcar dónde comienza, revierte y confirma toda la transacción, incluidas ambas operaciones.

La plataforma Java EE 8 requiere la IPA de Java Transaction 1.2.

IPA de Java para Servicios Web RESTful

La IPA de Java para Servicios Web RESTful (JAX-RS) define IPAs para el desarrollo de servicios web construidos según el estilo arquitectónico Representational State Transfer (REST). Una aplicación JAX-RS es una aplicación web que consta de clases empaquetadas como un servlet en un archivo WAR junto con las bibliotecas necesarias.

En la plataforma Java EE 8, las nuevas características de los servicios web RESTful incluyen lo siguiente:

  • IPA de cliente reactivo

    Cuando se reciben los resultados de una invocación en un recurso de destino, las mejoras en las IPAs de la etapa de finalización en Java SE permiten especificar, priorizar, combinar o concatenar la secuencia de esos resultados, y cómo se pueden manejar las excepciones.

  • Mejoras en la compatibilidad con eventos enviados por el servidor

    Los clientes pueden suscribirse a notificaciones de eventos emitidas por el servidor mediante una conexión de larga duración. Se ha agregado soporte para un nuevo tipo de medio, text/event-stream.

  • Soporte para objetos JSON-B e integración mejorada con tecnologías CDI, Servlet y Validación de Beans

La plataforma Java EE 8 requiere JAX-RS 2.1.

Beans Gestionados

Beans Gestionados, objetos livianos administrados por contenedores (POJO) con requisitos mínimos, admiten un pequeño conjunto de servicios básicos, como inyección de recursos, devoluciones de llamada del ciclo de vida e interceptores. Los Beans Gestionados representan una generalización de los beans administrados especificados por la tecnología JavaServer Faces y se pueden usar en cualquier lugar de una aplicación Java EE, no solo en módulos web.

La especificación Beans Gestionados es parte de la especificación de la plataforma Java EE 8 (JSR 366). La plataforma Java EE 8 requiere Beans Gestionados 1.0.

Contextos e Inyección de Dependencias para Java EE

Contextos e Inyección de Dependencias para Java EE (CDI) define un conjunto de servicios contextuales, proporcionados por contenedores Java EE, que facilitan a los desarrolladores el uso de beans enterprise junto con la tecnología JavaServer Faces en aplicaciones web. Diseñado para su uso con objetos con estado, CDI también tiene muchos usos más amplios, lo que permite a los desarrolladores una gran flexibilidad para integrar diferentes tipos de componentes de una manera poco acoplada pero segura.

En la plataforma Java EE 8, las nuevas características de CDI incluyen lo siguiente:

  • Una IPA para arrancar un contenedor CDI en Java SE 8

  • Soporte para el orden del observador, que determina el orden en el que se invocan los métodos del observador para un evento en particular, y soporte para disparar eventos de forma asincrónica.

  • Interfaces de configuradores, que se utilizan para definir y modificar dinámicamente objetos CDI

  • Literales de anotación incorporados, una característica conveniente para crear instancias de anotaciones y más

La plataforma Java EE 8 requiere CDI 2.0.

Inyección de Dependencias para Java

Inyección de Dependencias para Java define un conjunto estándar de anotaciones (y una interfaz) para usar en clases inyectables.

En la plataforma Java EE, CDI proporciona soporte para la inyección de dependencia. Específicamente, puede usar puntos de inyección solo en una aplicación habilitada para CDI.

La plataforma Java EE 8 requiere Inyección de Dependencias para Java 1.0.

Validación de Beans

La especificación Validación de Beans define un modelo de metadatos y una IPA para validar datos en componentes JavaBeans. En lugar de distribuir la validación de datos en varias capas, como el navegador y el servidor, puede definir las restricciones de validación en un solo lugar y compartirlas en las diferentes capas.

En la plataforma Java EE 8, las nuevas características de Validación de Beans incluyen las siguientes:

  • Soporte para nuevas funciones en Java SE 8, como la IPA de fecha y hora

  • AAdición de nuevas restricciones integradas de Validación de Beans

La plataforma Java EE 8 requiere Validación de Beans 2.0.

IPA del Servicio de Mensajería de Java

La IPA de Java Message Service (JMS) es un estándar de mensajería que permite a los componentes de la aplicación Java EE crear, enviar, recibir y leer mensajes. Permite la comunicación distribuida que está débilmente acoplada, es confiable y asincrónica.

La plataforma Java EE 8 requiere JMS 2.0.

Arquitectura de los Conectores de JavaEE

Los proveedores de herramientas e integradores de sistemas utilizan la Arquitectura de los Conectores de JavaEE para crear adaptadores de recursos que admitan el acceso a los sistemas de información empresarial que se pueden conectar a cualquier producto Java EE. Un adaptador de recursos es un componente de software que permite a los componentes de la aplicación Java EE acceder e interactuar con el administrador de recursos subyacente del EIS. Dado que un adaptador de recursos es específico de su administrador de recursos, normalmente existe un adaptador de recursos diferente para cada tipo de base de datos o sistema de información empresarial.

La Arquitectura de los Conectores de JavaEE también proporciona una integración transaccional basada en mensajes, segura, escalable y orientada al rendimiento de los servicios web basados en la plataforma Java EE con EIS existentes que pueden ser síncronos o asíncronos. Las aplicaciones existentes y los EIS integrados a través de la Arquitectura de los Conectores de JavaEE en la plataforma Java EE pueden exponerse como servicios web basados en XML utilizando modelos de componentes JAX-WS y Java EE. Así, JAX-WS y la Arquitectura de los Conectores de JavaEE son tecnologías complementarias para la integración de aplicaciones empresariales (EAI) y la integración empresarial de extremo a extremo.

La plataforma Java EE 8 requiere la Arquitectura de los Conectores de JavaEE 1.7.

IPA de JavaMail

Las aplicaciones Java EE utilizan la IPA de JavaMail para enviar notificaciones por correo electrónico. La IPA de JavaMail tiene dos partes:

  • Una interfaz a nivel de aplicación utilizada por los componentes de la aplicación para enviar correo.

  • Una interfaz de proveedor de servicios

La plataforma Java EE incluye IPA de JavaMail con un proveedor de servicios que permite a los componentes de la aplicación enviar correo por Internet.

La plataforma Java EE 8 requiere JavaMail 1.5.

Contrato de Autorización de Java para Contenedores

La especificación Contrato de Autorización de Java para Contenedores (JACC) define un contrato entre un servidor de aplicaciones Java EE y un proveedor de políticas de autorización. Todos los contenedores Java EE admiten este contrato.

La especificación JACC define clases java.security.Permission que satisfacen el modelo de autorización de Java EE. La especificación define la vinculación de las decisiones de acceso al contenedor a las operaciones en instancias de estas clases de permisos. Define la semántica de los proveedores de políticas que utilizan las nuevas clases de permisos para abordar los requisitos de autorización de la plataforma Java EE, incluida la definición y el uso de roles.

La plataforma Java EE 8 requiere JACC 1.5.

Interfaz de Proveedor de Servicios de Autenticación de Java para Contenedores

La especificación Interfaz de Proveedor de Servicios de Autenticación de Java para Contenedores (JASPIC) define una interfaz de proveedor de servicios (SPI) mediante la cual los proveedores de autenticación que implementan mecanismos de autenticación de mensajes pueden integrarse en contenedores de procesamiento de mensajes o tiempos de ejecución del cliente o servidor. Los proveedores de autenticación integrados a través de esta interfaz operan con los mensajes de red que les proporcionan sus contenedores de llamadas. Los proveedores de autenticación transforman los mensajes salientes para que la fuente de cada mensaje pueda ser autenticada por el contenedor receptor, y el destinatario del mensaje pueda ser autenticado por el remitente del mensaje. Los proveedores de autenticación autentican cada mensaje entrante y devuelven a sus contenedores de llamadas la identidad establecida como resultado de la autenticación del mensaje.

La plataforma Java EE 8 requiere JASPIC 1.1.

IPA de Seguridad de Java EE

El propósito de la especificación IPA de Seguridad de Java EE es modernizar y simplificar las IPAs de seguridad estableciendo simultáneamente enfoques y mecanismos comunes y eliminando las IPAs más complejas de la vista del desarrollador cuando sea posible. Java EE Seguridad presenta las siguientes API:

  • interfaz SecurityContext

    Proporciona un punto de acceso común y uniforme que permite a una aplicación probar aspectos de los datos de la persona que llama y conceder o denegar el acceso a los recursos.

  • interfaz HttpAuthenticationMechanism

    Autentica a los llamadores de una aplicación web y solo se especifica para su uso en el contenedor de servlets.

  • interfazIdentityStore

    Proporciona una abstracción de un almacén de identidades y que se puede utilizar para autenticar usuarios y recuperar grupos de llamantes.

La plataforma Java EE 8 requiere IPA de Seguridad de Java EE 1.0.

IPA de Java para WebSocket

WebSocket es un protocolo de aplicación que proporciona comunicaciones full-duplex entre dos pares a través de TCP. LA IPA de Java para WebSocket permite que las aplicaciones Java EE creen puntos finales utilizando anotaciones que especifican los parámetros de configuración del punto final y designan sus métodos de devolución de llamada del ciclo de vida.

La plataforma Java EE 8 requiere la IPA de Java para WebSocket 1.0.

IPA de Java para procesamiento JSON

La notación de objetos JavaScript (JSON) es un formato de intercambio de datos basado en texto derivado de JavaScript que se utiliza en servicios web y otras aplicaciones conectadas. La API de Java para procesamiento JSON (JSON-P) permite que las aplicaciones Java EE analicen, transformen y consulten datos JSON mediante el modelo de objetos o el modelo de transmisión.

En la plataforma Java EE 8, las nuevas características de JSON-P incluyen soporte para lo siguiente:

  • JSON Pointer

    Define una sintaxis de cadena para hacer referencia a un valor específico dentro de un documento JSON. JSON Pointer incluye una IPA para extraer valores de un documento de destino y transformarlos para crear un nuevo documento JSON.

  • JSON Patch

    Define un formato para expresar una secuencia de operaciones que se aplicará a un documento JSON.

  • JSON Merge Patch

    Define un formato y reglas de procesamiento para aplicar operaciones a un documento JSON que se basan en el contenido específico del documento de destino.

  • La adición de funciones de edición y transformación al procesamiento básico de documentos JSON.

  • Métodos y clases auxiliares, denominados recopiladores JSON, que aprovechan las características de la IPA Stream que se introdujo en Java SE 8.

La plataforma Java EE 8 requiere JSON-P 1.1.

IPA de Java para Enlace JSON

La IPA de Java para Enlace JSON (JSON-B) proporciona una capa de enlace para convertir objetos Java hacia y desde mensajes JSON. JSON-B también admite la capacidad de personalizar el proceso de mapeo predeterminado utilizado en esta capa de enlace mediante el uso de anotaciones Java para un campo, propiedad, tipo o paquete JavaBean dado, o proporcionando una implementación de una estrategia de nomenclatura de propiedad.

JSON-B es nuevo en la plataforma Java EE 8. La plataforma Java EE 8 requiere JSON-B 1.0.

Utilidades de Concurrencia para Java EE

Las Utilidades de Concurrencia para Java EE es una IPA estándar para proporcionar capacidades asincrónicas a los componentes de la aplicación Java EE a través de los siguientes tipos de objetos: servicio de ejecutor administrado, servicio de ejecutor programado administrado, fábrica de subprocesos administrada y servicio de contexto.

La plataforma Java EE 8 requiere Utilidades de Concurrencia para Java EE 1.0.

Aplicaciones por Lotes para la Plataforma Java

Los trabajos por lotes son tareas que se pueden ejecutar sin la interacción del usuario. La especificación Batch Applications para Plataforma Java es un marco de trabajo por lotes que brinda soporte para crear y ejecutar trabajos por lotes en aplicaciones Java. El marco de trabajo por lotes consta de un tiempo de ejecución por lotes, un lenguaje de especificación de trabajo basado en XML, una IPA de Java para interactuar con el tiempo de ejecución por lotes y una IPA de Java para implementar artefactos por lotes.

La plataforma Java EE 8 requiere aplicaciones por lotes para la Plataforma Java 1.0.
