Descripción general de la IPA de JMS

Esta descripción general define el concepto de mensajería, describe la IPA de JMS y dónde se puede usar, y explica cómo funciona la IPA de JMS dentro de la plataforma Java EE.

Aquí se tratan los siguientes temas:

  • ¿Qué es la mensajería?

  • ¿Qué es la IPA de JMS?

  • ¿Cuándo puede usar la IPA de JMS?

  • ¿Cómo funciona la IPA de JMS con la plataforma Java EE?

¿Qué es la mensajería?

La mensajería es un método de comunicación entre componentes de software o aplicaciones. Un sistema de mensajería es una instalación de igual a igual: un cliente de mensajería puede enviar y recibir mensajes de cualquier otro cliente. Cada cliente se conecta a un agente de mensajería que brinda servicios para crear, enviar, recibir y leer mensajes.

La mensajería permite la comunicación distribuida que está débilmente acoplada. Un componente envía un mensaje a un destino y el destinatario puede recuperar el mensaje del destino. Lo que hace que la comunicación esté débilmente acoplada es que el destino es todo lo que el emisor y el receptor tienen en común. El emisor y el receptor no tienen que estar disponibles al mismo tiempo para comunicarse. De hecho, el remitente no necesita saber nada sobre el receptor; ni el receptor necesita saber nada sobre el remitente. El remitente y el receptor solo necesitan saber qué formato de mensaje y qué destino utilizar. En este sentido, la mensajería difiere de las tecnologías estrechamente acopladas, como la invocación de métodos remotos (RMI), que requieren que una aplicación conozca los métodos de una aplicación remota.

La mensajería también difiere del correo electrónico (email), que es un método de comunicación entre personas o entre aplicaciones de software y personas. La mensajería se utiliza para la comunicación entre aplicaciones de software o componentes de software.

¿Qué es la IPA de JMS?

El Servicio de Mensajes de Java (JMS en inglés), es una IPA de Java que permite que las aplicaciones creen, envíen, reciban y lean mensajes. La IPA de JMS define un conjunto común de interfaces y la semántica asociada que permite que los programas escritos en el lenguaje de programación Java se comuniquen con otras implementaciones de mensajería.

La IPA de JMS minimiza el conjunto de conceptos que un programador debe aprender para utilizar los productos de mensajería, pero proporciona suficientes funciones para admitir aplicaciones de mensajería sofisticadas. También se esfuerza por maximizar la portabilidad de las aplicaciones JMS entre los proveedores de JMS.

JMS permite la comunicación que no solo está débilmente acoplada sino también

  • Asíncrona: un cliente receptor no tiene que recibir mensajes al mismo tiempo que el cliente emisor los envía. El cliente emisor puede enviarlos y pasar a otras tareas; el cliente receptor puede recibirlos mucho más tarde.

  • Confiable: un proveedor de mensajería que implementa la IPA de JMS puede garantizar que un mensaje se entregue una sola vez. Los niveles más bajos de confiabilidad están disponibles para aplicaciones que pueden darse el lujo de perder mensajes o recibir mensajes duplicados.

La versión actual de la especificación JMS es la versión 2.0. Puede descargar una copia de la especificación del sitio web de Java Community Process: http://www.jcp.org/en/jsr/detail?id=343.

¿Cuándo puede usar la IPA de JMS?

Es probable que un proveedor de aplicaciones empresariales elija una IPA de mensajería en lugar de una IPA estrechamente acoplada, como una llamada a procedimiento remoto (RPC), en las siguientes circunstancias.

  • El proveedor quiere que los componentes no dependan de la información sobre las interfaces de otros componentes, para que los componentes puedan reemplazarse fácilmente.

  • El proveedor desea que la aplicación se ejecute tanto si todos los componentes están activos como si no simultáneamente.

  • El modelo de negocio de la aplicación permite que un componente envíe información a otro y continúe operando sin recibir una respuesta inmediata.

Por ejemplo, los componentes de una aplicación empresarial para un fabricante de automóviles pueden utilizar la IPA de JMS en situaciones como las siguientes.

  • El componente de inventario puede enviar un mensaje al componente de fábrica cuando el nivel de inventario de un producto cae por debajo de cierto nivel para que la fábrica pueda fabricar más automóviles.

  • El componente de fábrica puede enviar un mensaje a los componentes de piezas para que la fábrica pueda ensamblar las piezas que necesita.

  • Los componentes de piezas, a su vez, pueden enviar mensajes a su propio inventario y pedir componentes para actualizar sus inventarios y pedir piezas nuevas a los proveedores.

  • Tanto la fábrica como los componentes de piezas pueden enviar mensajes al componente de contabilidad para actualizar las cifras del presupuesto.

  • La empresa puede publicar artículos de catálogo actualizados para su fuerza de ventas.

El uso de mensajería para estas tareas permite que los diversos componentes interactúen entre sí de manera eficiente, sin comprometer la red u otros recursos. La Figura 48-1 ilustra cómo podría funcionar este sencillo ejemplo.

¿Cómo funciona la IPA de JMS con la plataforma Java EE?

Cuando se introdujo por primera vez la IPA de JMS, su objetivo más importante era permitir que las aplicaciones Java accedieran a los sistemas de middleware orientados a la mensajería (MOM) existentes. Desde entonces, muchos proveedores han adoptado e implementado la IPA de JMS, por lo que un producto JMS ahora puede proporcionar una capacidad de mensajería completa para una empresa.

La IPA de JMS es una parte integral de la plataforma Java EE y los desarrolladores de aplicaciones pueden usar la mensajería con componentes Java EE. JMS 2.0 es parte de la versión Java EE 8.

La IPA de JMS en la plataforma Java EE tiene las siguientes características.

  • Los clientes de aplicaciones, los componentes Enterprise JavaBeans (EJB) y los componentes web pueden enviar o recibir sincrónicamente un mensaje JMS. Los clientes de aplicaciones también pueden configurar una escucha de mensajes que permita que los mensajes JMS se entreguen de forma asincrónica al recibir una notificación cuando un mensaje esté disponible.

  • Los beans controlados por mensajes, que son una especie de enterprise bean, permiten el consumo asíncrono de mensajes en el contenedor EJB. Un servidor de aplicaciones normalmente agrupa beans controlados por mensajes para implementar el procesamiento simultáneo de mensajes.

  • Las operaciones de envío y recepción de mensajes pueden participar en las transacciones de la IPA de Transacciones de Java (JTA, en inglés), que permiten que las operaciones JMS y los accesos a bases de datos se realicen en una sola transacción.

La IPA de JMS mejora las otras partes de la plataforma Java EE al simplificar el desarrollo empresarial, lo que permite interacciones asincrónicas confiables y poco acopladas entre los componentes Java EE y los sistemas heredados capaces de enviar mensajes. Un desarrollador puede agregar fácilmente un nuevo comportamiento a una aplicación Java EE que tiene eventos de negocio existentes agregando un nuevo bean controlado por mensajes para operar en eventos de negocio específicos. Además, la plataforma Java EE mejora la IPA de JMS al proporcionar soporte para transacciones JTA y permitir el consumo simultáneo de mensajes. Para obtener más información, consulte la especificación Enterprise JavaBeans, v3.2.

El proveedor JMS se puede integrar con el servidor de aplicaciones mediante la arquitectura del conector Java EE. Accede al proveedor JMS a través de un adaptador de recursos. Esta capacidad permite a los proveedores crear proveedores de JMS que se pueden conectar a varios servidores de aplicaciones y permite que los servidores de aplicaciones admitan varios proveedores de JMS. Para obtener más información, consulte la especificación de arquitectura del conector Java EE, v1.7.
