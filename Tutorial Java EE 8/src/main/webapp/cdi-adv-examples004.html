El Ejemplo producerfields: Usando los Campos de Producer para Generar Recursos

El ejemplo de producerfields, que le permite crear una lista de tareas, muestra cómo usar un campo de productor para generar objetos que luego pueden ser administrados por el contenedor. Este ejemplo genera un objeto EntityManager, pero también se pueden generar recursos como conexiones JDBC y fuentes de datos de esta manera.

El ejemplo de producerfields es el ejemplo de entidad más simple posible. También contiene un calificador y una clase que genera el administrador de la entidad. También contiene una sola entidad, un bean de sesión con estado, una página Facelets y un bean administrado.

Los archivos fuente se encuentran en el directorio tut-install/examples/cdi/producerfields/src/main/java/javaeetutorial/producerfields/ .

El Campo de Producer para el Ejemplo producerfields

El componente más importante del ejemplo producerfields es el más pequeño, la clase db.UserDatabaseEntityManager, que aísla la generación del objeto EntityManager para que pueda ser utilizado fácilmente por otros componentes de la aplicación. La clase usa un campo productor para inyectar un EntityManager anotado con el calificador @UserDatabase, también definido en el paquete db: 

                
@Singleton
public class UserDatabaseEntityManager {
    @Produces
    @PersistenceContext
    @UserDatabase
    private EntityManager em;
    ...
}

La clase no produce explícitamente un campo de unidad de persistencia, pero la aplicación tiene un archivo persistence.xml que especifica una unidad de persistencia. La clase se anota como javax.inject.Singleton para especificar que el inyector debe instanciarlo solo una vez.

La clase db.UserDatabaseEntityManager también contiene código comentado que usa los métodos create y close para generar y eliminar el campo productor: 

                
 /* @PersistenceContext
    private EntityManager em;
    @Produces
    @UserDatabase
    public EntityManager create() {
        return em;
    } */
    public void close(@Disposes @UserDatabase EntityManager em) {
        em.close();
    }

Puede eliminar los indicadores de comentarios de este código y colocarlos alrededor de la declaración de campo para probar cómo funcionan los métodos. El comportamiento de la aplicación es el mismo con cualquier mecanismo.

La ventaja de producir el EntityManager en una clase separada en lugar de simplemente inyectarlo en un bean empresarial es que el objeto se puede reutilizar fácilmente de forma segura. Además, una aplicación más compleja puede crear varios administradores de entidades utilizando varias unidades de persistencia, y este mecanismo aísla este código para facilitar el mantenimiento, como en el siguiente ejemplo: 

                
@Singleton
public class JPAResourceProducer {
    @Produces
    @PersistenceUnit(unitName="pu3")
    @TestDatabase
    EntityManagerFactory customerDatabasePersistenceUnit;
    @Produces
    @PersistenceContext(unitName="pu3")
    @TestDatabase
    EntityManager customerDatabasePersistenceContext;
    @Produces
    @PersistenceUnit(unitName="pu4")
    @Documents
    EntityManagerFactory customerDatabasePersistenceUnit;
    @Produces
    @PersistenceContext(unitName="pu4")
    @Documents
    EntityManager docDatabaseEntityManager;"
}

Las declaraciones de EntityManagerFactory también permiten que las aplicaciones utilicen un administrador de entidades administrado por la aplicación.

La Entidad producerfields y el Bean de Sesión

El ejemplo de producerfields contiene una clase de entidad simple, entity.ToDo, y un bean de sesión con estado, ejb.RequestBean, que lo utiliza.

La clase de entidad contiene tres campos: un campo id generado automáticamente, una cadena que especifica la tarea y una marca de tiempo. El campo de marca de tiempo, timeCreated, se anota con @Temporal, que es obligatorio para los campos persistentes de Date. 

                
@Entity
public class ToDo implements Serializable {
    ...
    @Id
    @GeneratedValue(strategy = GenerationType.AUTO)
    private Long id;
    protected String taskText;
    @Temporal(TIMESTAMP)
    protected Date timeCreated;
    public ToDo() {
    }
    public ToDo(Long id, String taskText, Date timeCreated) {
        this.id = id;
        this.taskText = taskText;
        this.timeCreated = timeCreated;
    }
    ...

El resto de la clase ToDo contiene los getters, setters y otros métodos de entidad habituales.

La clase RequestBean inyecta el EntityManager generado por el método productor, anotado con el calificador @UserDatabase: 

                
@ConversationScoped
@Stateful
public class RequestBean {
    @Inject
    @UserDatabase
    EntityManager em;

A continuación, define dos métodos, uno que crea y conserva un único elemento de la lista ToDo y otro que recupera todos los elementos ToDo creados hasta el momento mediante la creación de una consulta: 

                
    public ToDo createToDo(String inputString) {
        ToDo toDo = null;
        Date currentTime = Calendar.getInstance().getTime();
        try {
            toDo = new ToDo();
            toDo.setTaskText(inputString);
            toDo.setTimeCreated(currentTime);
            em.persist(toDo);
            return toDo;
        } catch (Exception e) {
            throw new EJBException(e.getMessage());
        }
    }
    public List<ToDo> getToDos() {
        try {
             List<ToDo> toDos =
                    (List<ToDo>) em.createQuery(
                    "SELECT t FROM ToDo t ORDER BY t.timeCreated")
                    .getResultList();
            return toDos;
        } catch (Exception e) {
            throw new EJBException(e.getMessage());
        }
    }
}

Las Páginas Facelets y Bean Gestionado de producerfields

El ejemplo de producerfields tiene dos páginas Facelets, index.xhtml y todolist.xhtml. El formulario simple en la página index.xhtml le pregunta al usuario solo por la tarea. Cuando el usuario hace clic en el botón Enviar, se llama al método listBean.createTask. Cuando el usuario hace clic en el botón Mostrar elementos, la acción especifica que se debe mostrar el archivo todolist.xhtml: 

                
    <h:body>
        <h2>Lista de quehaceres</h2>
        <p>Ingrese una tarea para ser completada.</p>
        <h:form id="todolist">
            <p><h:outputLabel value="Introduzca una cadena: " for="inputString"/>
                <h:inputText id="inputString"
                             value="#{listBean.inputString}"/></p>
            <p><h:commandButton value="Enviar"
                                action="#{listBean.createTask()}"/></p>
            <p><h:commandButton value="Mostrar artículos"
                                action="todolist"/></p>
        </h:form>
        ...
    </h:body>

El bean administrado, web.ListBean, inyecta el bean de sesión ejb.RequestBean. Declara la entidad entity.ToDo y una lista de la entidad junto con la cadena de entrada que pasa al bean de sesión. El inputString está anotado con la restricción @NotNull Validación de Beans, por lo que un intento de enviar una cadena vacía genera un error. 

                
@Named
@ConversationScoped
public class ListBean implements Serializable {
    ...
    @EJB
    private RequestBean request;
    @NotNull
    private String inputString;
    private ToDo toDo;
    private List<ToDo> toDos;

El método createTask llamado por el botón Enviar llama al método createToDo de RequestBean: 


    public void createTask() {
        this.toDo = request.createToDo(inputString);
    }

El método getToDos, que es llamado por la página todolist.xhtml, llama al método getToDos de RequestBean: 

                
public List<ToDo> getToDos() {
        return request.getToDos();
    }

Para obligar a la página Facelets a reconocer una cadena vacía como un valor nulo y devolver un error, el archivo web.xml establece el parámetro de contexto javax.faces.INTERPRET_EMPTY_STRING_SUBMITTED_VALUES_AS_NULL en true: 

                
<context-param>
  <param-name>javax.faces.INTERPRET_EMPTY_STRING_SUBMITTED_VALUES_AS_NULL</param-name>
  <param-value>true</param-value>
</context-param>

La página todolist.xhtml es un poco más complicada que la página index.html. Contiene un elemento dataTable que muestra el contenido de la lista ToDo. El cuerpo de la página se ve así: 

                
    <body>
        <h2>Lista de quehaceres</h2>
        <h:form id="showlist">
            <h:dataTable var="toDo"
                         value="#{listBean.toDos}"
                         rules="all"
                         border="1"
                         cellpadding="5">
                <h:column>
                    <f:facet name="header">
                        <h:outputText value="Marca de tiempo" />
                    </f:facet>
                    <h:outputText value="#{toDo.timeCreated}" />
                </h:column>
                <h:column>
                    <f:facet name="header">
                        <h:outputText value="Tarea" />
                    </f:facet>
                    <h:outputText value="#{toDo.taskText}" />
                </h:column>
            </h:dataTable>
            <p><h:commandButton id="back" value="Atrás" action="index" /></p>
        </h:form>
    </body>

El valor de dataTable es listBean.toDos, la lista devuelta por el método getToDos del bean administrado, que a su vez llama al método getToDos del bean de sesión. Cada fila de la tabla muestra los campos timeCreated y taskText de la tarea individual. Finalmente, un botón Atrás devuelve al usuario a la página index.xhtml.

Ejecutando el Ejemplo producerfields

Puede usar el EID NetBeans o Maven para compilar, empaquetar, implementar y ejecutar la aplicación producerfields.

Aquí se tratan los siguientes temas:

  • Para Construir, Empaquetar, y Desplegar el Ejemplo producerfields Usando el EID NetBeans

  • Para Construir, Empaquetar, y Desplegar el Ejemplo Usando Maven

  • Para Ejecutar el Ejemplo producerfields

Para Construir, Empaquetar, y Desplegar el Ejemplo producerfields Usando el EID NetBeans

  1. Asegúrese de que el Servidor GlassFish se haya iniciado (consulte Arrancando y Parando el Servidor GlassFish).

  2. Si el servidor de la base de datos aún no se está ejecutando, inícielo siguiendo las instrucciones de Arrancando y Parando Apache Derby.

  3. En el menú Archivo, elija Abrir proyecto.

  4. En el cuadro de diálogo Abrir proyecto, vaya a:

                            
tut-install/examples/cdi

  5. Seleccione la carpeta producerfields.

  6. Haga clic en Abrir proyecto.

  7. En la pestaña Proyectos, haga clic con el botón derecho en el proyecto producerfields y seleccione Build.

    Este comando construye y empaqueta la aplicación en un archivo WAR, producerfields.war, ubicado en el directorio target, y luego lo implementa en el Servidor GlassFish.

Para Construir, Empaquetar, y Desplegar el Ejemplo Usando Maven

  1. Asegúrese de que se haya iniciado el Servidor GlassFish (consulte Arrancando y Parando el Servidor GlassFish).

  2. Si el servidor de la base de datos aún no se está ejecutando, inícielo siguiendo las instrucciones en Arrancando y Parando Apache Derby.

  3. En una ventana de terminal, vaya a:

                            
tut-install/examples/cdi/producerfields/

  4. Ingrese el siguiente comando para implementar la aplicación:

                            
mvn install

    Este comando construye y empaqueta la aplicación en un archivo WAR, producerfields.war, ubicado en el directorio target, y luego lo implementa en el Servidor GlassFish.

Para Ejecutar el Ejemplo producerfields

  1. En un navegador web, ingrese la siguiente URL:

                            
http://localhost:8080/producerfields

  2. En la página Crear lista de tareas, ingrese una cadena en el campo y haga clic en Enviar.

    Puede ingresar cadenas adicionales y hacer clic en Enviar para crear una lista de tareas con varios elementos.

  3. Haga clic en Mostrar elementos.

    Se abre la página Lista de tareas pendientes, que muestra la marca de tiempo y el texto de cada elemento que creó.

  4. Haga clic en Atrás para volver a la página Crear lista de tareas pendientes.

    En esta página, puede ingresar más elementos en la lista.
