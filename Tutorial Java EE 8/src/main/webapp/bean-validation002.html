Usando Restricciones en la Validación de Beans

El modelo de Validación de Beans está soportado por restricciones en forma de anotaciones colocadas en un campo, método o clase de un componente JavaBeans, como un bean administrado.

Las restricciones pueden ser integradas o definidas por el usuario. Las restricciones definidas por el usuario se denominan restricciones personalizadas. Varias restricciones integradas están disponibles en el paquete javax.validation.constraints. La Tabla 23-1 enumera todas las restricciones integradas. Consulte Creación de restricciones personalizadas para obtener información sobre cómo crear restricciones personalizadas.

Tabla 23-1 Restricciones Incorporadas de Validación de Beans

Restricción

Descripción

Ejemplo

@AssertFalse

El valor del campo o propiedad debe ser false.

 
                        
@AssertFalse
boolean isUnsupported;

@AssertTrue

El valor del campo o propiedad debe ser true.

 
                        
@AssertTrue
boolean isActive;

@DecimalMax

El valor del campo o propiedad debe ser un valor decimal menor o igual que el número en el elemento de valor. 

                        
@DecimalMax("30.00")
BigDecimal discount;

@DecimalMin

El valor del campo o propiedad debe ser un valor decimal mayor o igual que el número en el elemento de valor. 

                        
@DecimalMin("5.00")
BigDecimal discount;

@Digits

El valor del campo o propiedad debe ser un número dentro de un rango especificado. El elemento integer especifica los dígitos enteros máximos para el número, y el elemento fraction especifica los dígitos fraccionarios máximos para el número. 

                        
@Digits(integer=6, fraction=2)
BigDecimal price;

@Email

El valor del campo o propiedad debe ser una dirección de correo electrónico válida. 

                        
@Email
String emailaddress;

@Future

El valor del campo o propiedad debe ser una fecha en el futuro. 

                        
@Future
Date eventDate;

@FutureOrPresent

El valor del campo o propiedad debe ser una fecha u hora en presente o futuro. 

                        
@FutureOrPresent
Time travelTime;

@Max

El valor del campo o propiedad debe ser un valor entero menor o igual que el número en el elemento de valor. 

                        
@Max(10)
int quantity;

@Min

El valor del campo o propiedad debe ser un valor entero mayor o igual que el número en el elemento de valor. 

                        
@Min(5)
int quantity;

@Negative

El valor del campo o propiedad debe ser un número negativo.

 
                        
@Negative
int basementFloor;

@NegativeOrZero

El valor del campo o propiedad debe ser negativo o cero.

 
                        
@NegativeOrZero
int debtValue;

@NotBlank

El valor del campo o la propiedad debe contener al menos un carácter que no sea un espacio en blanco. 

                        
@NotBlank
String message;

@NotEmpty

El valor del campo o propiedad no debe estar vacío. Se evalúa la longitud de los caracteres o la matriz y el tamaño de una colección o mapa. 

                        
@NotEmpty
String message;;

@NotNull

El valor del campo o propiedad no debe ser nulo.

 
                        
@NotNull
String username;

@Null

El valor del campo o propiedad debe ser nulo.

 
                        
@Null
String unusedString;

@Past

El valor del campo o propiedad debe ser una fecha en el pasado.

 
                        
@Past
Date birthday;

@PastOrPresent

El valor del campo o propiedad debe ser una fecha u hora en el pasado o presente. 

                        
@PastOrPresent
Date travelDate;

@Pattern

El valor del campo o propiedad debe coincidir con la expresión regular definida en el elemento regexp. 

                        
@Pattern(regexp="\\(\\d{3}\\)\\d{3}-\\d{4}")
String phoneNumber;

@Positive

El valor del campo o propiedad debe ser un número positivo.

 
                        
@Positive
BigDecimal area;
                        
                      

                    

                  

                

              		
            

            

              
                
@PositiveOrZero

              
                

                  El valor del campo o propiedad debe ser un número positivo o cero.
                

              		
              
                

                  

                    

                      
                        
@PositiveOrZero
int totalGoals;
                        
                      

                    

                  

                

              		
            

            

              
@Size

              
                

                  El tamaño del campo o propiedad se evalúa y debe coincidir con los límites especificados. Si 
                  el campo o la propiedad es una String, se evalúa el tamaño de la cadena. Si el 
                  campo o propiedad es una Collection, se evalúa el tamaño de la 
                  Collection. Si el campo o la propiedad es un Map, se evalúa el 
                  tamaño del Map. Si el campo o la propiedad es una matriz, se evalúa el tamaño 
                  de la matriz. Utilice uno de los elementos opcionales max o min 
                  para especificar los límites.
                

              		
              

                    

                      
                        
@Size(min=2, max=240)
String briefMessage;
                        
                      

                    

                  

                

              		
            

          
        

        

          

            En el siguiente ejemplo, se coloca una restricción en un campo usando la restricción incorporada 
            @NotNull:
          

        

        

          

            
              
public class Name {
    @NotNull
    private String firstname;
    @NotNull
    private String lastname;
    ...
}
              
            

          

        

        

          

            También puede colocar más de una restricción en un solo objeto de componente JavaBeans. Por 
            ejemplo, puede colocar una restricción adicional para el tamaño del campo en los campos 
            firstname y lastname:
          

        

        

          

            
              
public class Name {
    @NotNull
    @Size(min=1, max=16)
    private String firstname;
    @NotNull
    @Size(min=1, max=16)
    private String lastname;
    ...
}
              
            

          

        

        

          

            El siguiente ejemplo muestra un método con una restricción definida por el usuario que verifica la 
            restricción definida por el usuario que busca un patrón de número de teléfono predefinido, como un 
            número de teléfono específico de un país:
          

        

        

          

            
              
@USPhoneNumber
public String getPhone() {
    return phone;
}
              
            

          

        

        

          

            Para una restricción integrada, está disponible una implementación predeterminada. Una restricción 
            definida por el usuario o personalizada necesita una implementación de validación. En el ejemplo 
            anterior, la restricción personalizada @USPhoneNumber necesita una clase de 
            implementación.
          

        

      

    

    

      
Anotaciones repetidas

      

        

          

            En Validación de Beans 2.0, puede especificar la misma restricción varias veces en un objetivo de 
            validación usando la anotación repetida:
          

        

        

          

            
              
public class Account {
    @Max (value = 2000, groups = Default.class, message = "max.value")
    @Max (value = 5000, groups = GoldCustomer.class, message = "max.value")
    private long withdrawalAmount;
}
              
            

          

        

        

          

            Todas las restricciones integradas del paquete javax .validation.constraints admiten 
            anotaciones repetibles. De manera similar, las restricciones personalizadas pueden usar la 
            anotación @Repeatable. En el siguiente ejemplo, dependiendo de si el grupo es 
            PeakHour o NonPeakHour, la instancia de automóvil se valida como un 
            automóvil basado en dos o tres pasajeros, y luego se incluye como elegible en el carril de la 
            piscina del coche:
          

        

        

          

            
              
/**
 * Validate whether a car is eligible for car pool lane
 */
@Documented
@Constraint(validatedBy = CarPoolValidator.class)
@Target({ METHOD, FIELD, ANNOTATION_TYPE, CONSTRUCTOR, PARAMETER, TYPE_USE })
@Retention(RUNTIME)
@Repeatable(List.class)
public @interface CarPool {
    String message() default "{CarPool.message}";
    Class<?>[] groups() default {};
    int value();
    Class<? extends Payload>[] payload() default {};
    /**
     * Defines several @CarPool annotations on the same element
     * @see (@link CarPool}
     */
    @Target({ METHOD, FIELD, ANNOTATION_TYPE, CONSTRUCTOR, PARAMETER, TYPE_USE })
    @Retention(RUNTIME)
    @Documented
    @interface List {
        CarPool[] value();
    }
}
public class Car{
  private String registrationNumber;
  @CarPool(value = 2, group = NonPeakHour.class)
  @CarPool(value = 3, group = {Default.class, PeakHour.class})
  private int totalPassengers;
}
              
            

          

        

        

          

            Cualquier falla de validación se maneja con gracia y se puede mostrar con la etiqueta 
            h:messages.
          

        

        

          

            Cualquier bean administrado que contenga anotaciones de Validación de Beans obtiene 
            automáticamente restricciones de validación en los campos de las páginas web de una aplicación 
            JavaServer Faces.
