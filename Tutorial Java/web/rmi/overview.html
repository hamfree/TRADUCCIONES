Una descripción de las Aplicaciones RMI (Remote Method Interface)

Las aplicaciones RMI a menudo se componen de dos programas separados, un servidor y un cliente. Un programa típico de servidor crea algunos objetos remots, hace referencias a estos objetos accesibles, y espera para que los clients invoquen métodos en estos objetos. Un programa cliente típico obtiene una referencia remota a uno o más objetos remotos en un servidor y después invoca sus métodos. RMI proporciona el mecanismo mediante el cual el servidor y el cliente se comunican y pasan información de un lado a otro. Esta aplicación a veces se denomina aplicación de objeto distribuido.

Las aplicaciones de objetos distribuídos necesitan hacer lo siguiente:

  • Localizar objetos remotes. Las aplicaciones pueden usar varios mecanismos para obtener referencias a objetos remotos. Por ejemplo, una aplicación puede registrar sus objetos remotos con la sencilla función de nomenclatura de RMI, el registro RMI. Alternativamente, una aplicación puede pasar y devolver referencias de objetos remotos como parte de otras invocaciones remotas.
  • Comunicar con objetos remotos. Los detalles de la comunicación entre objetos remotos son manejados por RMI. Para el programador, la comunicación remota es similar a las invocaciones de de métodos Java habituales.
  • Cargar definiciones de clases para objetos que se pasan. Debido a que RMI permite que los objetos se pasen de un lado a otro, proporciona mecanismos para cargar las definiciones de clase de un objeto, así como para transmitir los datos de un objeto.

La ilustración siguiente muestra una aplicación distribuida RMI que utiliza el registro RMI para obtener una referencia a un objeto remot. El servidor llama al registro para asociar (o vincular) un nombre con un objeto remoto. El cliente busca el objeto remoto por su nombre en el registro del servidor y después invoca un método en él. La ilustración también muestra que el sistema RMI usa un servidor web existente para cargar definiciones de clase, de servidor a cliente y de cliente a servidor, para los objetos cuando sea necesario.

Ventajas de la Carga de Código Dinámico

Uno de las principales y únicas características de RMI es su capacidad para descargar la definición de la clase de un objeto si la clase no está definida en la máquina virtual de Java del receptor. Todos los tipos y comportamientos de un objeto, disponibles previamente sólo en una única máquina virtual de Java, se pueden transmitir a otra máquina virtual de Java, posiblemente remota. RMI pasa los objetos por sus clases reales, por lo que el comportamiento de los objetos no cambia cuando se envían a otra máquina virtual Java. Esta capacidad permite introducir nuevos tipos y comportamientos en una máquina virtual Java remota, extendiendo así dinámicamente el comportamiento de una aplicación. El ejemplo del motor de cálculo en este recorrido usa esta capacidad para introducir un comportamiento nuevo a un programa distribuido.

Interfaces Remotos, Objetos, y Métodos

Como cualquier otra aplicación Java, una aplicación distribuida construída usando RMI de Java se compone de interfaces y clases. Las interfaces declaran métodos. Las clases implementan los métodos declarados en las interfaces y, quizás, declaran métodos adicionales también. En una aplicación distribuida, algunas implementaciones podrían residir en algunas máquinas virtuales de Java pero no en otras. Los objetos con métodos que se pueden invocar a través de máquinas virtuales de Java se llaman objetos remotoss.

Un objeto se convierte en remoto implementando una interfaz remota, que tiene las características siguientes:

  • Una interfaz remota extiende la interfaz java.rmi.Remote.
  • Cada método de la interfaz declara java.rmi.RemoteException en su cláusula throws, además de otras excepciones específicas de la aplicación.

RMI trata a un objeto remoto de forma diferente a un objeto no remoto cuando el objeto se pasa de una máquina virtual Java a otra máquina virtual Java. En vez de hacer una copia del objeto de implementación en la máquina virtual de Java que recibe, RMI pasa un esbozo (Nota del traductor: stub, en inglés) remoto para un objeto remoto. El esbozo actúa como el representante local, o apoderado, para el objeto remoto y básicamente es, para el cliente, la referencia remota. El cliente invoca un método en el esbozo local, que es responsable de llevar a cabo la invocación del método en el objeto remoto.

Un esbozo para un objeto remoto implementa el mismo conjunto de interfaces remotas que el objeto remoto implementa. Esta propiedad permite a un esbozo ser convertido a cualquiera de las interfaces que el objeto remoto implementa. Sin embargo, solo esos métodos definidos en una interfaz remota están disponibles para llamarse desde la máquina virtual de Java receptora.

Crear Aplicaciones Distribuídas Usando RMI

Usar RMI para desarrollar una aplicación distribuida implica estos pasos generales:

  1. Diseñar e implementar los componentes de su aplicación distribuida.
  2. Compilar los fuentes.
  3. Hacer que las clases estén disponibles a través de la red.
  4. Iniciar la aplicación.

Diseñar e Implementar los Componentes de la Aplicación

Primero, determine la arquitectura de su aplicación, incluyendo qué componentes son objetos locales y qué componenetes son accesibles de forma remota. Este paso incluye:

  • Definir las interfaces remotas. Una interfaz remota especifica los métodos que se pueden invocar de forma remota por un cliente. Los clientes programan en interfaces remotas, no en las clases de implementación de esas interfaces. El diseño de tales interfaces incluye la determinación de los tipos de objetos que serán utilizados como parámetros y valores de retorno para estos métodos. Si alguna de estas interfaces o clases no existe aún, también debe definirlas.
  • Implementar los objetos remotos. Los objetos remotos deben implementar una o más interfaces remotas. La clase del objeto remot puede incluir implementaciones de otras interfadces y métodos que están disponibles sólo localmente. Si alguna clase local se va a usar para parámetros o valores de retorno de alguno de estos métodos, también deben implementarse.
  • Implementar los clientes. Los clientes que usan objetos remotos pueden ser implementados en cualquier momento después de que se definen las interfaces remotas, incluso después de que se hayan desplegado los objetos remotos.

Compilar los Fuentes

Como con cualquier programa Java, use el compilador javac para compilar los ficheros fuente. Los ficheros fuente contienen las declaraciones de las interfaces remotas, sus implementaciones, cualquier otra clase del servidor, y las clases del cliente.

Nota: Con versiones anteriores a la Plataforma Java, Edición Estándar 5.0, se requería un paso adicional para construir las clases de esbozo, usando el compilador rmic. Sin embargo, este paso ya no es necesario.

Hacer que las Clases Sean Accesibles por Red

En este paso, haga que ciertas definiciones de clases sean accesibles por red, tales cmo las definiciones para las interfaces remotas y sus tipos asociados, y las definiciones para las clases que necesitan ser descargadas a los clientes o servidores. Las definiciones de clases se hacen accesibles habitualmente para la red a través de un servidor web.

Iniciar la Aplicación

Iniciar la aplicación incluye ejecutar el registro RMI de objetos remotos, el servidor, y el cliente.

El resto de esta sección describe los pasos que se utilizan para crear un motor de cálculo.

Construir un Motor de Cálculo Genérico

Este recorrido se centra en una aplicación distribuida simple pero poderosa llamada motor de cálculo. El motor de cálculo es un objeto remoto en el servidor que toma tareas de los clientes, ejecuta las tareas, y devuelve los resultados. Las tareas son ejecutadas en la máquina donde se ejecuta el servidor. Este tipo de aplicación distribuida puede permitir que una variedad de máquinas cliente hagan uso de una máquina potenten o una máquina que tenga hardware especializado.

El aspecto novedoso del motor de cálculo es que las tareas que ejecuta no necesitan definirse cuando se escribe o se inicia el motor de cálculo. Se pueden crear nuevas clases de tareas en cualquier momento y después dadas al motor de cálculo para ser ejecutadas. El único requerimiento de una tarea es que su clase implemente una interfaz particular. El sistema RMI puede descargar el código necesario para realizar la tarea en el motor de cálculo. Después, el motor de cálculo ejecuta la tarea, usando los recursos de la máquina en la que el motor de cálculo se está ejecutando.

La habilidad de realizar tareas arbitrarias es permitida por la naturaleza dinámica de la plataforma Java, que RMI extiende a la red. RMI carga dinámicamente el código de la tarea dentro de la máquina virtual Java del motor de cálculo y ejecuta la tarea sin conocimiento previo de la clase que implementa la tarea. Dicha aplicación, que tiene la habilidad de descargar código dinámicamente, a menudo se denomina una aplicación basada en el comportamiento. Tales aplicaciones usualmente requieren infraestructuras completas habilitadas por agentes. Con RMI, tales aplicaciones son parte de los mecanismos básicos para la computación distribuida en la plataforma Java.
