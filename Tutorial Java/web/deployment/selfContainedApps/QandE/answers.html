Respuestas a las Preguntas y Ejercicios: Aplicaciones Autocontenidas

Preguntas

  1. Pregunta: ¿Cuáñ de los siguientes items no es una ventaja de las aplicaciones autocontenidas?

    1. Los usuarios instalan la aplicación con un instalador que es familiar para ellos.
    2. La aplicación se ejecuta como una aplicación nativa.
    3. La aplicación requiere menos espacio en la máquina del usuario.
    4. Usted controla la versión del Entorno de Ejecución de Java (EEJ) que es usado por la aplicación.
    5. La aplicación no requiere un navegador para ejecutarse.

    Respuesta: C. La aplicación requiere más espacio porque el Entorno de Ejecución de Java (EEJ) es empaquetado con la aplicación.

  2. Pregunta: Verdadero o Falseo: El tipo MIME debe ser siempre usado para configurar una asociación de fichero.

    Respuesta: Falso. Dependiendo de la plataforma y el empaquetador usado, un tipo MIME o una extensión de fichero puede ser usada. Para Linux, el tipo MIME es requerido. Para Windows, la extensió de fichero es requerida. Para OS X, o el tipo MIME o la extensión de fichero es requerida. Es una buena práctica el ofrecer tanto el tipo MIME como la extensión de fichero cuando se configuran asociaciones de fichero, independientemente de la plataforma.

  3. Pregunta: ¿Qué elementos son usados para identifcar los puntos de entrada para las aplicaciones autocontenidas en la tarea de Ant <fx:deploy>?

    Respuesta: El atributo mainClass del elemento <fx:application> es usado para identificar el punto de entrada principla. Si la aplicación autoconteida tiene múltiples puntos de entrada, el elemento <fx:secondaryLauncher> es usado para cada punto de entrada secundario.

Ejercicios

  1. Ejercicio: Escriba la tarea de Ant <fx:deploy> para generar un paquete Windows MSI para una aplicación simple llamada Mi Apli Simple. El fichero JAR para la aplicación está en el directoriodist , la clase principal es samples.MyApp, y los ficherosa de salida son los escritos al directorio actual.

    Respuesta: 

    <fx:deploy outdir="."
           outfile="MySampleApp"
           nativeBundles="msi>

    <fx:application name="Mi Aplicación de Ejemplo"
                    mainClass="samples.MyApp"/>

    <fx:resources>
        <fx:fileset dir="dist" includes="*.jar"/>
    </fx:resources>

    <fx:info title="Mi Aplicación de Ejemplo"
             description="Una apli simple de ejemplo"/>
</fx:deploy>

  2. Ejercicio: Mejore el código en el ejercicio previo para crear paquetes para todos los instaladores de Windows y defina una asociación de fichero para los ficheros de texto.

    Respuesta: 

    <fx:deploy outdir="."
           outfile="MiAppDeEjemplo"
           nativeBundles="installer">

    <fx:application name="Mi Aplicación de Ejemplo"
                    mainClass="samples.MyApp"/>

    <fx:resources>
        <fx:fileset dir="dist" includes="*.jar"/>
    </fx:resources>

    <fx:info title="Mi Aplicación de Ejemplo"
             description="Una simple apli de ejemplo">
        <fx:association extension="txt"
                        description="Ficheros de texto">
         </fx:association>
     </fx:info>
</fx:deploy>

    Cuando el atributo nativeBundles es fijado a installer, el empaquetador intenta construir paquetes para todos los instaladores soportados para esa plataforma. La imagen de disco no es creada. Si las herramientas necesitadas para construir un paquete particular no están disponibles, ese paquete es saltado.

    Windows requiere solo el atributo extension cuando se definen asociaciones de fichero.
