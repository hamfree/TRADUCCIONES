Directrices para Asegurar Aplicaciones Ricas de Internet

Las siguientes pautas proporcionan los pasos que puede seguir para reducir la vulnerabilidad de las aplicaciones de Internet enriquecidas (ARI) que proporciona a los usuarios.

  • Siga las Pautas de Codificación Segura
  • Pruebe con la Última Versión del EEJ
  • Incluya Atributos en el Manifiesto
  • Use un Fichero JNLP Firmado
  • Firme con Sellos de Tiempo los Ficheros JAR
  • Use el Protocolo HTTPS
  • Evite ARIs Locales

Siga las Pautas de Codificación Segura

Siga las recomendaciones de Pautas de codificación segura para el lenguaje de programación Java. En la Sección 4, "Accesibilidad y Extensibilidad" se describe como limitar la accesibilidad a clases y paquetes, lo cual reduce la vulnerabilidad de su código.

El código de JavaScript es considerado inseguro y está restringido al entorno limitado de seguridad por defecto. Minimize las interacciones entre la ARI y el código JavaScript. Use el bloque AccessController.doPrivileged con cuidado porque permite el acceso desde cualquier página HTML o código JavaScript.

Pruebe con la Última Versión del EEJ

Asegúrese de que su ARI se ejecuta en la última versión segura del EEJ. La plataforma Java soporta la capacidad para las ARIs de especificar la versión Java que se necesita para ejecutar la ARI, sin embargo, exigir a los usuarios que mantengan más de una versión del EEJ, especialmente versiones antiguas e inseguras, es un riesgo de seguridad para el usuario.

Uno de los beneficios de las ARIs es que las versiones actualizadas de la ARI son descargadas automáticamente en el sistema del usuario. Pruebe su ARI contra cada actualización del EEJ y asegúrese de que funciona. Si se necesitan cambios, actualize su ARI en el servidor de forma que los usuarios puedan instalar la última versión del EEJ y aún ejecutar la ARI.

Incluya Atributos en el Manifiesto

Agregue atributos al fichero de manifiesto del JAR que describe las propiedades de la ARI. Los valores en el fichero JNLP o la etiqueta applet son comparados con los valores en el manifiesto para verificar que el código correcto es ejecutado.

Solicite permisos de entorno limitado cuando su RIA no requiera acceso más allá del entorno limitado de seguridad. El entorno limitado de Java ofrece protecciones adicionales para los usuarios, y los usuarios podrían no ejecutar una aplicación privilegiada si no entienden por qué solicita acceso ilimitado a su sistema.

Los atributos del manifiesto pueden también usarse para identificar las ubicaciones desde las cuales su ARI puede ser accedida. Esto incluye las ubicaciones desde las cuales el código de JavaScript puede llamar a su ARI, y las ubicaciones de los ficheros JNLP o etiquetas de applet que pueden iniciar su RIA. Vea Mejorando la Seguridad con los Atributos del Manifiesto para información de los atributos de manifiesto que están disponibles.

Use un Fichero JNLP Firmado

Si su ARI necesita acceder a propiedades del sistema no seguras o argumentos de la MVJ, use un JNLP firmado. Si se requiere alguna variación entre los ficheros JNLP externos e internos, use plantillas JNLP. Vea Ficheros JNLP Firmados en la Guía de implementación de la Plataforma Java, Edición Estándar para obtener información.

Para acceder a propiedades del sistema no segura o argumentos de la MVJ, incluya la propiedad o argumento en el fichero JNLP como se describe en in Fijando Argumentos Confiables y Propiedades Seguras.

Firme con Sellos de Tiempo los Ficheros JAR

Obtenga un certificado de firmado de código de una autoridad de certificación confiables y úselo para firmar los ficheros JAR de su ARI. Despliegue a los usuarios sólo ARIs que estén firmadas con un certificado válido.

Cuando firme su fichero JAR, también marque la fecha en la firma. La marca de tiempo verifica que el certificado era válido en el momento en el que se fijó el JAR, por lo que la ARI no se bloquea automáticamente cuando expira el certificado. Vea Firmando Ficheros JAR para información sobre firma y sellado de tiempo.

Las ARIs autofirmadas y las no firmadas son consideradas inseguras y no se permite su ejecución a menos que se configure una lista de sitios de excepción o un conjunto de reglas de implementación para permitir a aplicaciones específicas. Sin embargo, el auto-firmado puede ser útil con fines de prueba. Para probar su ARI auto-firmada, puede importar el certificado auto-firmado en el almacén de claves de confianza.

Use el Protocolo HTTPS

Use el protocolo HTTPS en el servidor web desde el cual los usuarios obtienen su ARI. El protocolo HTTPS es cifrado y validado por el servidor, lo que hace que sea más difícil para cualquiera manipular su ARI.

Evite ARIs Locales

Las ARIs locales no están destinadas para su uso en producción. Para asegurar que los usuarios ejecutan el código que usted desea que ejecuten, almacene su ARI en un servidor de aplicaciones.

Para probar, se recomiendo el uso de un servidor web. Otra opción es agregar su aplicación a una lista de sitios de excepción, la cual es gestionada en la pestaña de Seguridad del Panel de Control de Java.
