Lección: Conceptos de Nombrado y Directorio

Conceptos de Nombrado

Una facilidad fundamental en cualquier sistema de computadoras es el servicio de nombres — los medios por los cuales los nombres se asocian con objetos y los objetos se encuentran en función de sus nombres. Cuando usa casi cualquier programa de computadora o sistema, está siempre nombrando un objeto u otro. Por ejemplo, cuando usa un sistema de correo electrónico, debe proporcionar el nombre del destinatario. Para acceder a un fichero en la computadora, debe suplir su nombre. Un servicio de nombres le permite buscar un objeto dado su nombre.

La función primaria de un servicio de nombres es asignar nombres descriptivos de personas a objetos, como direcciones, identificadores, o objetos usados típicamente por programas de ordenador.

Por ejemplo, el Sistema de Nombres de Dominio de Internet (DNS) asigna nombres de máquina a direcciones IP: 

www.example.com ==> 192.0.2.5

Un sistema de ficheros asigna un nombre de fichero a una referencia de fichero que un programa puede usar para acceder a los contenidos del fichero. 

c:\bin\autoexec.bat ==> Referencia de Fichero

Estos dos ejemplos también ilustran el amplio rango de la escala en la cual existen los servicios de nombre — desde nombrar un objeto en Internet a nombrar un fichero en el sistema de ficheros local.

Nombres

Para buscar un objeto en un sistema de nombres, le proporciona el nombre del objeto. El sistema de nombrado determina la sintaxis que el nombre debe seguir. Esta sintaxis se llama a veces la convención de denominación de los sistemas de nombres. Un nombre se compone de componentes. La representación de un nombre consiste de un separador de componente marcando los componentes del nombre.

La convención de denominación de los sistemas de nombres
Sistema de Nombrado Separador de Componente Nombres
Sistema de ficheros UNIX "/" /usr/hello
DNS "." sales.Wiz.COM
LDAP "," and "=" cn=Rosanna Lee, o=Sun, c=US

La convención de denominación del sistema de ficheros UNIX es que se nombre un fichero desde su ruta relativa a la raíz del sistema de ficheros, con cada componente en la ruta separada de izquierda a derecha usando el carácter de barra inclinada ("/"). El nombre de ruta UNIX, /usr/hello, por ejemplo, nombra un archivo hello en el directorio de ficheros usr, que está ubicado en la raíz del sistema de ficheros.

La convención de denominación de DNS exige que los componentes del nombre DNS se ordenen de derecha a izquierda y estén delimitados por el carácter punto ("."). Así el nombre DNS sales.Wiz.COM nombra una entrada DNS con el nombre sales, relativo a la entrada DNS Wiz.COM. La entrada DNS Wiz.COM, a su vez, nombra una entrada con el nombre Wiz en la entrada COM.

La convención de denominación del Protocolo Ligero de Acceso a Directorios (LDAP) ordena los componentes de derecha a izquierda, delimitados por el carácter coma (","). Así el nombre LDAP cn=Rosanna Lee, o=Sun, c=US nombra una entrada LDAP cn=Rosanna Lee, relativa a la entrada o=Sun, que a su vez, es relativa a c=us. LDAP tiene la regla adicional de que cada componente del nombre debe ser un par de nombre/valor con el nombre y el valor separados por un carácter igual (" = ").

Vinculaciones

La asociación de un nombre con un objeto se llama una vinculación (Nota del Traductor: binding, en inglés). Un nombre de fichero está vinculado a un fichero.

El DNS contiene vinculaciones que asignan nombres de máquina a direcciones IP. Un nombre LDAP está vinculado a una entrada LDAP.

Referencias y Direcciones

Dependiendo del servicio de nombres, algunos objetos no pueden ser almacenados directamente por el servicio de nombres; esto es, una copia del objeto no puede ser colocada dentro del servicio de nombres. En vez de eso, deben ser almacenados por referencia; esto es, un puntero o referencia al objeto se ubica dentro del servicio de nombres. Una referencia representa información sobre cómo acceder a un objeto. Típìcamente, es una representación compacta que se puede usar para comunicar con el objeto, mientras el mismo objeto podría contener más información de estado. Usando la referencia, puede contactar con el objeto y obtener más información sobre el objeto.

Por ejemplo, un objeto de avión podría contener una lista de los pasajeros del avión y la tripulación, su plan de vuelo, y estado del combustibles y los instrumentos, y su número de vuelo y hora de salida. Por constraste, una referencia al objeto de avión podría contener su número de vuelo y hora de salida. La referencia es una representación mucho más compacta de la información sobre el objeto avión y puede usarse para obtener información adicional. Un objeto fichero, por ejemplo, es accedido usando una referencia de fichero. Un objeto impresora, por ejemplo, podría contener el estado de la impresora, tal como su cola actual y la cantidad de papel en la bandeja de papel. Una referencia al objeto impresora, por otro lado, podría contener sólo información sobre cómo alcanzar la impresora, así como el nombre del servidor de impresión y el protocolo de impresión.

Aunque en general una referencia puede contener cualquier información arbitraria, es útil que se refiera a sus contenidos como direcciones (o puntos finales de comunicación): información específica de cómo acceder al objeto.

Por simplicidad, este tutorial usa "objeto" para referirse tanto a objetos como a referencias de objetos cuando no se requiere una distinción entre los dos.

Contexto

Un contexto es un conjunto de vinculaciones nombre-a-objeto. Cada contexto tiene una convención de denominación asociada. Un contexto siempre proporciona una operación de búsqueda (resolución) que devuelve el objeto, por lo general, también proporciona operaciones como las de vincular nombres, desvincular nombres y enumerar nombres vinculados. Un nombre en un objeto contexto puede ser vinculado a otro objeto contexto (llamado un subcontexto) que tiene la misma convención de denominación.

Un directorio de ficheros, tal como /usr, en el sistema de ficheros UNIX representa un contexto. Un directorio de ficheros nombrado de forma relativa a otro directorio de ficheros representa un subcontexto (los usuarios de UNIX se refieren a esto como un subdirectorio). Esto es, en un directorio de ficheros /usr/bin, el directorio bin es un subcontexto de usr. Un dominio DNS, tal como COM, representa un contexto. Un dominio de DNS nombrado de forma relativa a otro dominio DNS representa un subcontexto. Para el dominio DNS Sun.COM, el dominio DNS Sun es un subcontexto de COM.

Finalmente, una entrada LDAP, tal como c=us, representa un contexto. Una entrada LDAP nombrada de forma relativa a otra entrada LDAP representa un subcontexto. Para la entrada LDAP o=sun,c=us, la entrada o=sun es un subcontexto de c=us.

Sistemas de Nombres Y Espacios de Nombres

Un sistema de nombres es un conjunto de contextos conectados del mismo tipo (tienen la misma convención de denominación) y proporciona un conjunto común de operaciones.

Un sistema que implementa el DNS es un sistema de nombres. Un sistema que se comunica usando el LDAP es un sistema de nombres.

Un sistema de nombres proporciona un servicio de nombres a sus clientes para realizar operaciones relacionadas-con-nombres. Se accede a un servicio de nombres a través de su propia interfaz. El DNS ofrece un servicio de nombres que asocia nombres de máquinas a direcciones IP. LDAP ofrece un servicio de nombres que asocia nombres LDAP a entradas LDAP. Un sistema de ficheros ofrece un servicio de nombres que asocia nombres de fichero a ficheros y directorios.

Un espacio de nombres es el conjunto de todos los posibles nombres en el sistema de nombres. El sistema de ficheros UNIX tiene un espacio de nombres que consiste de todos los nombres de ficheros y directorios en ese sistema de ficheros. El espacio de nombres DNS contiene los nombres de los dominios y entradas DNS. El espacio de nombres LDAP contiene los nombres de las entradas LDAP.
