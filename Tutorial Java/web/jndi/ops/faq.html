Consejos para Solucionar Problemas

Estos son los problemas más comunes que puede encontrar cuando intenta ejecutar un programa compilado exitosamente que usa las clases JNDI.

  1. Sin Contexto Inicial
  2. Conexión Denegada
  3. La Conexión Falla
  4. El Programa Se Bloquea
  5. Nombre no Encontrado
  6. No Se Puede Conectar con Hosts Arbitrarios
  7. No Se Puede Acceder a las Propiedades del Sistema para la Configuración
  8. No Se Puede Autenticar Mediante CRAM-MD5

1. Obtiene una NoInitialContextException.

Causa: No especificó qué implementación usar para el contexto inicial. Específicamente, la propiedad de entorno Context.INITIAL_CONTEXT_FACTORY no fue establecida al nombre de la clase de la factoria que creará el contexto inicial. O, no puso a disposición del programa las clases del proveedor de servicio nombrado por Context.INITIAL_CONTEXT_FACTORY.

Solución: Establezca la propiedad de entorno Context.INITIAL_CONTEXT_FACTORY al nombre de la clase de la implementación del contexto inicial que está usando. Vea la sección Configuration para los detalles.

Si la propiedad fue establecida, entonces asegúrese de que el nombre de la clase no fue mal escrita, y que la clase nombrada está a disposición de su programa (ya sea en su ruta de clases o instalada en el directorio jre/lib/ext del EEJ). La Platforma Java incluye proveedores de servicio para LDAP, nombrado COS, DNS, y el registro RMI. Todos los otros proveedores de servicio deben ser instalados y agregados al entorno de ejecución.

2. Obtiene una CommunicationException, indicando "conexión denegada."

Causa: El servidor y puerto identificados por la propiedad de entorno Context.PROVIDER_URL no están siendo atendidos por el servidor. Quizás alguien ha deshabilitado o apagado la máquina en la cual el servidor se está ejecutando. O, puede ser que haya escrito mal el nombre del servidor o el número de puerto.

Solución: Compruebe que efectivamente haya un servidor ejecutándose en ese puerto y reinicie el servidor si es necesario. La forma en la que realize esta comprobación depende del servidor LDAP que esté usando. Usualmente, se dispone de una consola administrativa o herramienta que puede usar para administrar el servidor. Puede usar esa herramienta para verificar el estado del servidor.

3. El servidor LDAP responde a otras utilidades (tales como su consola de administración) pero parece no responder a las peticiones de su programa.

Causa: El servidor no responde a las peticiones de conexión LDAP v3. Algunos servidores (especialmente los servidores públicos) no responden correctamente a LDAP v3, ignorando las peticiones en vez de rechazarlas. También, algunos servidores LDAP v3 tienen problemas gestionando un control que el proveedor de servicio de LDAP de Oracle envía automáticamente y a menudo devuelven un código de fallo específico del servidor.

Solución. Intente establecer la propiedad de entorno "java.naming.ldap.version" en "2". El proveedor de servicio LDAP de forma predeterminada intenta conectar a un servidor LDAP usando la versión 3 de LDAP; si eso falla, entonces usa la versión 2 de LDAP. Si el servidor ignora silenciosamente la petición v3, entonces el proveedor asumirá que la petición funcionó. Para evitar dichos servidores, debe establecer explícitamente la versión del protocolo para garantizar el comportamiento adecuado del servidor.

Si el servidor es un servidor con versión 3, entonces intente establecer la propiedad de entorno siguiente antes de crear el contexto inicial: 

env.put(Context.REFERRAL, "throw");

Esto deshabilitará el control que el proveedor LDAP envía automáticamente. (Compruebe el Tutorial JNDI para los detalles).

4. El programa se bloquea.

Causas: Algunos servidores (especialmente los públicos) no responderán ( Some servers (especially public ones) won't respond (ni siquiera con una respuesta negativa) si intenta realizar una búsqueda que generaría demasiados resultados o que requeriría al servidor examinar demasiadas entradad para generar la respuesta. Tales servidores están intentando limitar la cantidad de recursos que gastan por solicitud.

O, ha intentado usar la Capa de Conexión Segura (SSL, en inglés) contra un servidor/puerto que no lo admite, y viceversa (esto es, intentó usar un socket plano para hablar a un puerto SSL).

Y, por último, el servidor responde muy lentamente debido a una gran carga o no responde en absoluto por alguna razón.

Solución: Si su programa se bloquea porque el servidor está intentando restringir el uso de sus recursos, entonces reintente su petición usando una consulta que devuelva un resultado único o sólo unos pocos resultados. Esto ayudará a determinar si el servidor está activo. Si es así, puede ampliar su consulta inicial y volver a enviarla.

Si su programa se bloquea a causa de problemas SSL, entonces necesita encontrar si el puerto es un puerto SSL y después establecer la propiedad de entorno Context.SECURITY_PROTOCOL apropiadamente. Si el puerto es un puerto SSL, entonces esta propiedad debería ser establecida a "ssl". Si no es un puerto SSL, entonces no se debería establecer esta propiedad.

Si su programa se cuelga por ninguna de las razones anteriores, la propiedad com.sun.jndi.ldap.read.timeout resulta útil para especificar el tiempo de espera de lectura. El valor de esta propiedad es la representación en cadena de un entero que representa el tiempo de espera de lectura en milisegundos para las operaciones LDAP. Si el proveedor LDAP no puede obtener un respuesta LDPA dentro de ese period, abortará el intento de lectura. El entero debería ser mayor que cero. Un entero menor o igual que cero significa que no se especifica tiempo de espera de lectura lo cual es equivalente a esperar indefinidamente la respuesta hasta que se reciba.

Si no se especifica esta propiedad, lo predeterminado es esperar la respuesta hasta que sea recibida.

Por ejemplo,

env.put("com.sun.jndi.ldap.read.timeout", "5000"); causa que el proveedor de servicio LDAP aborte el intento de lectura si el servidor no responde con una respuesta en 5 segundos.

5. Obtiene una NameNotFoundException.

Causas: Cuando inicializó el contexto inicial para LDAP, proporcionó un nombre distinguido de raíz. Por ejemplo, si establece la propiedad de entorno Context.PROVIDER_URL para el contexto inicial a "ldap://ldapserver:389/o=JNDITutorial" y después proporcionó un nombre como "cn=Joe,c=us", entonces el nombre completo que pasó al servicio LDAP fue "cn=Joe,c=us,o=JNDITutorial". Si este era realmente el nombre que pretendía, debe comprobar que su servidor para asegurarse de que contenga dicha entrada.

Además, el Servidor de Directorio Oracle devuelve este error si proporciona un nombre distinguido incorrecto para propósitos de autenticación. Por ejemplo, el proveedor LDAP lanzará una NameNotFoundException si establece la propiedad de entorno Context.SECURITY_PRINCIPAL a "cn=Admin, o=Tutorial", y "cn=Admin, o=Tutorial" no tiene una entrada en el servidor LDAP. El error correcto para el Servidor de Directorio de Oracle devuelva en realidad debería ser algo relacionado con la autenticación, en lugar de "nombre no encontrado".

Solución: Verifique que el nombre que proporcionó es el de una entrada existente en el servidor. Puede hacer esto listando las entradas del contexto padre o usando alguna otra herramienta como la consola de administración del servidor de directorio.

Aquí tiene algunos problemas que podría encontrar cuando intenta desplegar un applet que usa las clases JNDI.

6. Obtiene una AppletSecurityException cuando su applet intenta comunicar con un servidor de directorio que está ejecutándose en una máquina diferente a la que se cargó el applet

Causa: Su applet no fue firmado, por lo que sólo puede conectar a la máquina desde la cual se cargó. O, si el applet fue firmado, el navegador no ha dado permiso al applet para conectar a la máquina del servidor directorio.

Solución: Si quiere permitir que el applet se conecte a servidores de directorio que se ejecutan en máquinas arbitrarias, entonces debe firmar tanto su applet como todos los JARs JNDI que su applet vaya a usar. Para encontrar información sobre el firmado de JARs, vea Firmando y Verificando los Ficheros JAR.

7. Obtiene una AppletSecurityException cuando su applet intenta configurar las propiedades de entorno usando propiedades del sistema.

Causa: Los navegadores web limitan el acceso a las propiedades del sistema y lanzan una SecurityException si intenta leerlas.

Solución: Si necesita obtener una entrada para su applet, intente utilizar los parámetros del applet en su lugar.

8. Obtiene una AppletSecurityException cuando un applet ejecutándose dentro de Firefox intenta autenticar usando CRAM-MD5 al servidor LDAP.

Causa: Firefox deshabilita el acceso a los paquetes java.security. El proveedor LDAP usó la funcionalidad resumen del mensaje proporcionada por java.security.MessageDigest para implementar CRAM-MD5.

Solución: Use el Complemento de Java.
