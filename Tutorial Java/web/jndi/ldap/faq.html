Preguntas Frecuentes

Esta lección responde a las preguntas frecuentes que los usuarios tienen cuando usan JNDI para acceder a servicios LDAP. Algunos de los problemas comunes se responden en Consejos para solucionar problemas de la lección de Operaciones de Nombrado y Directorio.

1. ¿Es el contexto seguro para acceso multiproceso, o necesito bloquear/sincronizar el acceso a un contexto?

La respuesta depende de la implementación. Esto es porque las interfaces Context y DirContext no especifican requerimientos de sincronización. La implementación LDAP en el KDJ se optimizó para el acceso de un solo subproceso. Si tiene múltiples hilos accediendo a la misma instancia Context, entonces cada hilo necesita bloquear la instancia Context cuando lo use. Esto también se aplica a cualquier NamingEnumeration que se derive de la misma instancia Context, entonces cada hilo necesita bloquear la instancia Context cuando la usa. Sin embargo. los hilos múltiples pueden acceder a diferentes instancias de Context (incluso esas derivadas del mismo contexto inicial) simultaneamente sin bloqueos.

2. ¿Porqué el proveedor LDAP ignora mis propiedades de entorno de seguridad si no establezco la propiedad Context.SECURITY_CREDENTIALS ("java.naming.security.credentials") o la establece a la cadena vacía?

Si usted proporcoina una cadena vacía, una matriz vacía de byte/char, o null a la propiedad de entorno Context.SECURITY_CREDENTIALS, entonces una vinculación anónima ocurrirá incluso si la propiedad Context.SECURITY_AUTHENTICATION fue establecida en "simple". Esto es porque para la autenticación simple, LDAP requiere que la contraseña no esté vacía. Si una contraseña no se proporciona, entonces el protocolo automáticamente convierte la autenticación a "none".

3. ¿Porqué sigo obteniendo una CommunicationException cuando intento crear un contexto inicial?

Podría estar hablando a un servidor que admite sólo la v2 de LDAP. Vea la lección Misceláneo del Tutorial JNDI para un ejemplo de cómo establecer el número de versión.

4. ¿Cómo puedo obtener un rastro de los mensajes LDAP?

Intente usar la propiedad de entorno "com.sun.jndi.ldap.trace.ber". Si el valor de esta propiedad es una instancia de java.io.OutputStream, entonces, la información de seguimiento sobre los búferes BER enviados y recibidos por el proveedor LDAP se escribe en ese flujo. Si el valor de la propiedad es null, entonces no se escribe ninguna salida de seguimiento.

Por ejemplo, el código siguiente enviará la salida de seguimiento a System.err. 

env.put("com.sun.jndi.ldap.trace.ber", System.err);

5. ¿Cómo uso un mecanismo de autenticación diferente como Kerberos?

Siga las instrucciones en la autenticación GSS-IPA / Kerberos v5 del Tutorial JNDI para información de cómo usar la autenticación Kerberos. Para usar otros mecanismos de autenticación, vea la sección Mecanismos Arbitrarios SASL del Tutorial JNDI.

6. ¿Debo habilitar ssl al cambiar la contraseña? /

Realmente depende del servidor de directorio que esté utilizando. Algunos servidores de directorio no le permitirán cambiar la contraseña si SSL no está habilitado pero algunos lo permiten. Es bueno tener SSL habilitado para tener su contraseña asegurada en el canal de comunicación.

7. Cuando pido un atributo, obtengo otro. ¿Por qué?

El nombre de atributo que está usando podría ser un sinónimo para otro atibuto. En este caso, el servidor LDAP podría devolver el nombre de atributo canónico en vez del que ha proporcionado. Cuando mire en los Attributes devueltos por el servidor, necesita usar el nombre canónico en vez del sinónimo.

Por ejemplo, "fax" podría ser un sinónimo par el nombre de atributo canónico "facsimiletelephonenumber". Si usted pide "fax", el servidor devolverá el atributo llamado "facsimiletelephonenumber". Vea la lección Operaciones de Nombrado y Directorio para los detalles sobre los sinónimos y otros problemas relacionados con los atributos de nombres.

8. ¿Cómo sé el tipo de valor de un atributo?

Un valor de atributo puede ser o java.lang.String o byte[]. Vea la sección Miscelánea del Tutorial JNDI para información sobre qué valores de atributos se devuelven como byte[]. Para hacer esto programadamente, puede usar el operador instanceof para examinar el valor de atributo que obtuvo desde el proveedor LDAP.

9. ¿Cómo recupero el valor de un atributo en una forma que no sea una cadena o una matriz de bytes?

Actualmente no puede. El proveedor LDAP devuelve sólo valores de atributo que son o java.lang.String o byte[]. Vea la sección Miscelánea del Tutorial JNDI.

10. ¿Por qué poner un & quot; * & quot; como valor de atributo no funciona como se esperaba en mi búsqueda?

Cuando usa la forma siguiente de search(), los valores de atributo son tratados como literales; esto es, el atributo en la entrada del directorio se espera que contenga exactamente ese valor: search(Name name, Attributes matchingAttrs) Para usar comodines, debe usar la forma de cadena de filtros de search(), como sigue. search(Name name, String filter, SearchControls ctls)
search(Name name, String filterExpr, Object[]filterArgs, SearchControls ctls)

Para la última forma, los caracteres comodín deben aparecer en el argumento filterExpr, y no en filterArgs. Los valores en filterArgs son también tratados como literales.

11. ¿Por qué no siempre funcionan los comodines en los filtros de búsqueda?

Un comodín que aparezca antes o después del valor de atributo (tal como en "attr=*I*") indica que el servidor debe buscar valores de atributo coincidentes utilizando la regla de coincidencia de subcadenas del atributo. Si la definición del atributo no tiene una regla de coincidencia de subcadena, entonces el servidor no puede encontrar el atributo. Tendrá que buscar usando una igualdad o el filtro "present" por el contrario.

12. ¿Por qué obtengo solo n número de entradas cuando sé que hay más en el directorio? Algunos servidores se configuran para limitar el número de entradas que pueden ser devueltas. Otras también limitan el número de entradas que pueden ser examinados durante una búsqueda. Compruebe su configuración de servidor.

13. ¿Cómo paso los controles con mi búsqueda?

Los controles no son explicados en este tutorial. Compruebe el Tutorial JNDI.

14. ¿Cómo averiguo cuántos resultados de búsqueda obtuve?

Debe llevar la cuenta a medidas que enumera los resultados. LDAP no proporciona esta información.

15. ¿Por qué obtengo una cadena vacía como nombre en mi SearchResult?

getName() siempre devuelve un nombre relativo al contexto objetivo de la búsqueda. Asi, si el contexto objetivo satisface el filtro de búsqueda, entonces el nombre devuelto será "" (el nombre vacío) porque ése el nombre relativo al contexto objetivo. Vea la sección Resultados de la Búsqueda para los detalles.

16. ¿Por qué obtengo una cadena de URL como nombre en mi SearchResult?

La entrada LDAP fue recuperada siguiendo o un alias o una referencia, así que su nombre es una URL. Vea la lección Resultados de la Búsqueda para los detalles.

17. ¿Qué tipos es el argumento Name pasada a el Context y métodos de DirContext? - ¿un CompoundName o un CompositeName?

Las formas de cadena aceptan la representación de cadena de un nombre compuesto. Es decir, usar un nombre de cadena equivale a llamar new CompositeName(stringName) y pasar los resultados al método Context/DirContext. El argumento Name puede ser cualquier objeto que implementa la interfaz Name. Si este es una instancia de CompositeName, entonces el nombre se trata como un mobre compuesto; de otra forma, se trata como un nombre compuesto.

18. ¿Puedo pasar el nombre que obtuve de NameParser a los métodos Context?

Esto está relacionado con la pregunta previa. Sí, usted puede. NameParser.parse() devuelve un nombre compuesto que implementa la interfaz Name. Este nombre puede ser pasado a los métodos de Context, que lo interpretarán como un nombre compuesto.

19. ¿Cual es la relación entre el nombre que uso para la propiedad Context.SECURITY_PRINCIPAL y el directorio?

Puede pensar que el nombre principal proviene de un espacio de nombres diferente al del directorio. Vea el RFC 2829 y la sección de Seguridad para los detalles de los mecanismos de autenticación LDAP. El proveedor de servicio LDAP en el KDJ acepta una cadena de nombre principal, que se pasa directamente al servidor LDAP. Algunos servidores LDAP aceptan DNs, mientras otros soportan los esquemas propuestos por RFC 2829.

20. ¿Por qué hay comillas extrañas y escapes en los nombres que leo del directorio?

El analizador de nombres de LDAP en el KDJ es conservador con respecto a las reglas de entrecomillado, pero sin embargo produce nombres "correctos". También, recuerde que los nombres de las entradas devueltas por NamingEnumerations son nombres compuestos que pueden ser pasados de vuelta a los métodos Context y DirContext. Así, si el nombre contiene un carácter que entra en conflicto con la sintaxis del nombre compuesto (como un carácter de barra derecha "/"), entonces el proveedor LDAP proporcionará una codificación para asegurar que el carácter de barra sea tratado como parte del nombre LDAP en vez de ser tratado como un separador del nombre compuesto.

Empiece a usar las clases LdapName y Rdn que habilitan la manipulación fácil del nombre.

21. ¿Cómo obtengo el DN completo de una entrada LDAP?

Puede usar NameClassPair.getNameInNamespace().
