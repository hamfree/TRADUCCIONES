SASL

El protocolo LDAP v3 usa SASL para admitir autenticación conectable. Esto significa que el cliente y servidor LDAP pueden ser configurados para negociar y usar mecanismos de autenticación posiblemente no estándar y/o personalizados, dependiendo del nivel de protección deseado por el cliente y el servidor. El protocolo LDAP v2 no soporta SASL.

Actualmente se definen varios mecanismos SASL:

  • Anónimo (RFC 2245)
  • CRAM-MD5 (RFC 2195)
  • Digest-MD5 (RFC 2831)
  • Externo (RFC 2222)
  • Kerberos V4 (RFC 2222)
  • Kerberos V5 (RFC 2222)
  • SecurID (RFC 2808)
  • S/Key (RFC 2222)

Mecanismo SASL Soportados por Servidores LDAP

De los mecanismos de la lista anterior, los servidores LDAP populares (tales como los de Oracle, OpenLDAP, y Microsoft) admiten Externo, Digest-MD5, y Kerberos V5. RFC 2829 Propone el uso de Digest-MD5 como el mecanismo predeterminado obligatorio para los servidores LDAP v3.

Aquí tiene un programa simple para conocer la lista de mecanismos SASL que admite un servidor LDAP. 

// Create initial context
DirContext ctx = new InitialDirContext();

// Read supportedSASLMechanisms from root DSE
Attributes attrs = ctx.getAttributes(
    "ldap://localhost:389", new String[]{"supportedSASLMechanisms"});

Aquí tiene la salida producida al ejecutar este programa contra un servidor que admite el mecanismo Externo de SASL. 

{supportedsaslmechanisms=supportedSASLMechanisms: 
                         EXTERNAL, GSSAPI, DIGEST-MD5}

Especificar el Mecanismo de Autenticación

Para usar un mecanismo particular de SASL, especifique el nombre registrado de la Autoridad de Asignación de Números de Internet (IANA) en la propiedad de entorno Context.SECURITY_AUTHENTICATION. Puede también especificar una lista de mecanismos para que el proveedor LDAP intente usar. Esto se hace especificando una lista ordenada de nombres de mecanismos separados por espacios. El proveedor LDAP usará el primer mecanismo para el cual encuentra una implementación.

Aquí tiene un ejemplo que pide al proveedor LDAP que intente obtener la implementación del mecanismo DIGEST-MD5 y si ese no está disponible, que use el de GSSAPI. 

env.put(Context.SECURITY_AUTHENTICATION, "DIGEST-MD5 GSSAPI");

Podría obtener esta lista de mecanismos de autenticación desde el usuario de su aplicación. O podría obtenerla preguntando al servidor LDAP, a través de una llamada similar a la mostrada anteriormente. El proveedor LDAP por sí mismo no consulta al servidor por esta información. Simplemente intenta ubicar y usar la implementación de los mecanismos especificados.

El proveedor LDAP en la plataforma tiene soporte integrado para los mecanismos External, Digest-MD5, y GSSAPI (Kerberos v5) de SASL. Puede agregar soporte para mecanismos adicionales.

Especificación de Entrada para el Mecanismo de Autenticación

Algunos mecanismos, como el Externo, no requieren entrada adicional — el nombre del mecanismo por sí solo es suficiente para que se proceda con la autenticación. El ejemplo externo muestra cómo usar el mecanismo Externo de SASL.

La mayoría de los otros mecanismo requieren alguna entrada adicional. Dependiendo del mecanismo, el tipo de entrada podría varíar. Lo soguiente son entradas comunes requeridas por los mecanismos.

  • Id de Autenticación. La identidad de la entidad que realiza la autenticación.
  • Id de Autorización. La identidad de la entidad para la que se deben realizar las comprobaciones de control de acceso si la autenticación tiene éxito.
  • Credenciales de Autenticación. Por ejemplo, una contraseña o una clave.

Los ids de autenticación y autorización podrían diferir si el programa (como un servidor proxy) se está autenticando en nombre de otra entidad. El id de autenticación se especifica usando la propiedad de entorno Context.SECURITY_PRINCIPAL. Es del tipo java.lang.String.

La contraseña/clave del id de autenticación se especifica usando la propiedad de entorno Context.SECURITY_CREDENTIALS. Es del tipo java.lang.String, matriz de char (char[]), o matriz de byte (byte[]). Si la contraseña es una matriz de byte, entonces se transforma en una matriz de char usando una codificación UTF-8.

Si se ha establecido la propiedad "java.naming.security.sasl.authorizationId", entonces su valor se usa como el ID de autorización. Su valor debe ser del tipo java.lang.String. Por defecto, se usa la cadena vacía como el ID de autorización, que indica al servidor que obtenga un ID de autorización a partir de las credenciales de autenticación del cliente.

El ejemplo Digest-MD5 muestra cómo usar las propiedades Context.SECURITY_PRINCIPAL y Context.SECURITY_CREDENTIALS para la autenticación Digest-MD5.

Si un mecanismo requiere otra entrada de aquellas ya descritas, entonces necesita definir un objeto de retrollamada (Nota del Traductor: callback, en inglés) para el mecanismo a usar, puede comprobar el ejemplo de retrollamada en el Tutorial JNDI. La siguiente parte de esta lección describe cómo usar el mecanismo de autenticación Digest-MD5 de SASL. Las Políticas SASL, IPA de GSS (Kerberos v5) y los mecanismos CRAM-MD5 se cubren en el Tutorial de JNDI.
