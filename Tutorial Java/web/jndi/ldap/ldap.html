LDAP v3

Internacionalización

La internacionalización se aborda a través de un conjunto de caracteres internacionales (ISO 10646) para representar elementos del protocolo que son cadenas (como los DNs). La versión 3 también difiere de la versión 2 en que usa UTF-8 para codificar sus cadenas.

Autenticación

Además de la autenticación anónima y simple (contraseña de texto sin cifrar), LDAP v3 utiliza el marco de autenticación de la Capa de Seguridad y Autenticación Simple (SASL, en inglés) ( RFC 2222) que permite diferentes mecanismos de autenticación a usarse con LDAP. SASL especifica un protocolo de desafío-respuesta en el cual se intercambian datos entre el cliente y el servidor con fines de autenticación.

Actualmente se definen varios mecanismos SASL: DIGEST-MD5, CRAM-MD5, Anónimo, Externo, S/Key, GSSAPI, y Kerberos v4. Un cliente LDAP v3 puede usar cualquiera de esto mecanismos SASL, siempre que el servidor LDAP v3 los admita. Además, se pueden utilizar nuevos mecanismos SASL (aún por definir) sin que sea necesario realizar cambios en el LDAP.

Referencias

Una referencia es información que un servidor envía de vuelta al cliente indicando que la información pedida puede ser encontrada en otra ubicación (posiblemente en otro servidor). En LDAP v2, se suponía que los servidores manejaban las referencias y no las devolvían al cliente. Esto se debe a que manejar las referencias puede ser muy complicado y, por lo tanto, resultaría en clientes más complicados. Como los servidores fueron construidos y desplegados, las referencias fueron encontradas útiles, pero no muchos servidores soportaban el manejo del lado del servidor de las referencias. Así que se encontró una forma de adaptar el protocolo para permitirle devolver referencias. Esto fue hecho colocando la referencia dentro del mensaje de error de una respuesta de error "resultado parcial".

LDAP v3 tiene soporte explícito para referencias y permite a los servidores devolver las referencias directamente al cliente. Las referencias no estarán cubiertas en esta lección pero puede siempre referirse al Tutorial JNDI para gestionar las referencias en su aplicación.

Despliegue

Un protocolo común como LDAP es útil para asegurar que todos los clientes y servidores de directorio "hablan el mismo idioma". Cuando muchas aplicaciones cliente de directorio y servicios de directorio se despliegan en una red, también es muy útil que todas estas entidades hablen de los mismos objetos.

Un A esquema de directorio especifica, entre otras cosas, los tipos de los objetos que un directorio puede tener y los atributos obligatorios y opcionales que cada tipo de objeto puede tener. LDAP v3 define un esquema (RFC 2252 y RFC 2256) basados en el estándar X.500 para los objetos comunes encontrados en la red, tales como países, localidades, organizaciones, usuarios/personas, grupos, y dispositivos. También define una forma para que una aplicación cliente acceda al esquema del servidor de forma que pueda encontrar los tipos de objetos y atributos que un servidor particular admite.

LDAP v3 adema´s define un conjunto de sintaxis para representar valores de atributo (RFC 2252). Para escribir aplicaciones de JAva que necesitan acceder al esquema por favor refiérase al Tutorial JNDI.

Extensiones

Además del repertorio de operaciones predefinidas, tales como "buscar" y "modificar", LDAP v3. define una operación "extendida". La operación "extendida" toma una petición como un argumento y devuelve una respuesta. La petición contiene un identificador que identifica la petición y los argumentos de la petición. La respuesta contien los resultados de realizar la petición. El par de operaciones "extendidas" petición/respuesta se llama una extensión. Por ejemplo, puede haber una extensión para Start TLS, que es una solicitud del cliente al servidor para activar el protocolo Start TLS.

Estas extensiones pueden ser estándar (definidas por la comunidad LDAP) o propietarias (definidas por un fabricante de un directorio particular). Para escribir aplicaciones que necesitan usar Extensiones refiérase al Tutorial JNDI.

Controles

Otra forma de agregar funciones es usando un control. LDAP v3 permite que el comportamiento de cualquier operación sea modificada a través del uso de controles. Cualquier número de controles puede ser enviado junto con una operación, y cualquier número de controles puede ser devuelto con sus resultados. Por ejemplo, puede enviar un control Sort junto con una operación de "búsqueda" que le dice al servidor que clasifique los resultados de la búsqueda de acuerdo al atributo "name".

Como las extensiones, dichos controles pueden ser estándar o propietarios. Los controles estándares se proporcionan en la plataforma. Para escribir aplicaciones que necesiten el uso de controles refiérase al Tutorial JNDI.
