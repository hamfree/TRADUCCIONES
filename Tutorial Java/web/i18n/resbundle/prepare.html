Preparándose para Utilizar un ResourceBundle

Identificando los Objetos Específicos de la Configuración Regional

Si tu aplicación tiene una interfaz de usuario, contiene muchos objetos específicos de configuración regional. Para empezar, debería recorrer su código fuente y buscar objetos que varíen con Locale. Su lista podría incluir objetos instanciados desde las clases siguientes:

  • String
  • Image
  • Color
  • AudioClip

Notará que esta lista no contiene objetos que representan números, fechas, horas, o monedas. El formato de visualización de estos objetos varía con Locale, pero los objetos en sí mismos no. Por ejemplo, usted da formato a una Date de acuerdo a Locale, pero usa el mismo objeto Date a pesar del Locale. En lugar de aislar estos objetos en un ResourceBundle, usted les da formato con unas clases especiales de formato sensibles a la configuración regional. Aprenderá cómo hacer esto en la sección Fechas y Horas de la lección Formateando.

En general, los objetos almacenados en un ResourceBundle están predefinidos y se envían con el producto. Estos objetos no son modificados mientras se ejecuta el programa. Por ejemplo, debe almacenar una etiqueta Menu en un ResourceBundle porque esta es específica de la configuración regional y no cambiará durante la sesión del programa. Sin embargo, no debe aislar en un ResourceBundle un objeto String que el usuario final introduce en un TextField. Los datos tales como este String pueden variar de un día a otro. Es específico de la sesión del programa, no del Locale en el cual el programa se ejecuta.

Usualmente la mayoría de los objetos que necesita aislar en un ResourceBundle son objetos String. Sin embargo, no todos los objetos String son específicos de una configuración regional. Por ejemplo, si un String es un elemento de protocolo usado por la comunicación entre procesos, no necesita ser traducido, por que los usuarios finales nunca lo ven.

La decisión de si traducir algunos objetos String no siempre es clara. Los ficheros de trazas son un buen ejemplo. Si un fichero de trazas es escrito por un programa y leído por otro, ambos programa están usando el fichero de trazas como un buffer para la comunicación. Suponga que los usuarios finales ocasionalmente comprueban los contenidos de este fichero de trazas. ¿No debería ser traducido el fichero de trazas? Por otro lado, si los usuarios finales raramente comprueban el fichero de trazas, el coste de la traducción podría no valer la pena. Su decisión para traducir este fichero de trazas depende de una variedad de factores: el diseño del programa, la facilidad de uso, el coste de la traducción, y la compatibilidad.

Organizar Objetos ResourceBundle

Puede organizar sus objetos ResourceBundle de acuerdo a la categoría de objetos que contienen. Por ejemplo, podría querer cargar todas las etiquetas del IGU para una ventana de entrada de pedidos en un ResourceBundle llamado OrderLabelsBundle. Usar múltiples objetos ResourceBundle ofrece varias ventajas:

  • Su código es más fácil de leer y mantener.
  • Evitará enormes objetos ResourceBundle, que pueden tardar demasiado en cargarse en memoria.
  • Puede reducir el uso de memoria cargando cada ResourceBundle sólo cuando se necesite.
