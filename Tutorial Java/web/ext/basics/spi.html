Introducción

Una aplicacion extensible es una que puede extender sin modificar su base de código original. Puede mejorar su funcionalidad con nuevos módulos o plug-ins. Los desarrolladores, vendedores de software, y clientes pueden agregar nueva funcionalidad o interfaces de programación de aplicaciones (IPAs) agregando un nuevo fichero Archivo de Java (JAR) en la ruta de clases de la aplicación o dentro del directorio de extensión específico de la aplicación.

Esta sección describe cómo crear aplicaciones con servicios extensibles, los cuales le capacitan a usted o a otros a proporcionar implementaciones de servicios que no requieren modificaciones a la aplicación original. Al diseñar una aplicación extensible, proporciona una forma de actualizar o mejorar partes específicas de un producto sin cambiar la aplicación principal.

Un ejemplo de una aplicación extensible es un procesador de palabras que permita al usuario final agregar un diccionario nuevo o un corrector ortográfico. En este ejemplo, el procesador de palabras proporciona un diccionario o una función de ortografía que otros desarrolladores, o incluso clientes, pueden extender proporcionando sus propias implementaciones de la funcionalidad.

Lo siguiente son términos y definiciones importantes para entender las aplicaciones extensibles:

Servicio
Un conjunto de interfaces y clases de programación que proporcionan acceso a alguna funcionalidad específica de la aplicación o característica. El servicio puede definir las interfaces para la funcionalidad y una forma de recuperar una implementación. En el ejemplo del procesador de palabras, un servicio de diccionario puede definir una forma de recuperar un diccionario y la definición de una palabra, pero no implementa el conjunto subyacente de la característica. En vez de eso, confía en un proveedor de servicio para implementar esa funcionalidad.
Interfaz del Proveedor de Servicio (Nota del Traductor: Service provider interface en inglés o SPI, se usará el acrónimo en inglés)
El conjunto de interfaces públicas y clases abstractas que define un servicio. La SPI define las clases y métodos disponibles para su aplicación.
Proveedor de Servicio
Implementa la SPI. Una aplicación con servicios extensibles le capacitan a usted, a los vendedores, y los clientes para agregar proveedores de servicio sin modificar la aplicación original.

Ejemplo de Servicio de Diccionario

Considere cómo podría diseñar un servicio de diccionario en un procesador de palabras o editor. Una forma es definir un servicio representado por una clase llamada DictionaryService y una interfaz del proveedor de servicio llamada Dictionary. El DictionaryService proporciona un objeto singleton de DictionaryService. (Vea la sección El Patrón de Diseño Singleton para más información). Este objeto recupera las definiciones de las palabras desde los proveedores de Dictionary. Los clientes del servicio de Diccionario — su código de aplicación — recuperan una instancia de este servicio, y el servicio buscará, instanciará, y usará los proveedores de servicio de Dictionary.

Aunque el desarrollador del procesador de palabras probablemente proporcionará un diccionario general básico con el producto original, el cliente podría requerir un diccionario especializado, quizás conteniendo términos técnicos o legales. Idealmente, el cliente es capaz de crear o comprar nuevos diccionarios y agregarlos a la aplicación existente.

Entendiendo el Ejemplo DictionaryServiceDemo

Los pasos siguientes le muestran como re-crear los contenidos del fichero DictionaryServiceDemo.zip. Estos pasos le muestran cómo funciona el ejemplo y cómo ejecutarlo.

1. Definir la Interfaz del Proveedor de Servicio

El ejemplo DictionaryServiceDemo define un SPI, la interfaz Dictionary.java. Contiene sólo un método: 


package dictionary.spi;

public interface Dictionary {
    public String getDefinition(String word);
}

El ejemplo almacena el fichero de clase compilado en el directorio DictionaryServiceProvider/build.

2. Definir el Servicio Que Recupera las Implementaciones del Proveedor de Servicio

La clase DictionaryService.java carga y accede a los proveedores de servicio disponibles de Dictionary en el nombre de los clientes del servicio de diccionario: 


package dictionary;

import dictionary.spi.Dictionary;
import java.util.Iterator;
import java.util.ServiceConfigurationError;
import java.util.ServiceLoader;

public class DictionaryService {

    private static DictionaryService service;
    private ServiceLoader<Dictionary> loader;

    private DictionaryService() {
        loader = ServiceLoader.load(Dictionary.class);
    }

    public static synchronized DictionaryService getInstance() {
        if (service == null) {
            service = new DictionaryService();
        }
        return service;
    }


    public String getDefinition(String word) {
        String definition = null;

        try {
            Iterator<Dictionary> dictionaries = loader.iterator();
            while (definition == null && dictionaries.hasNext()) {
                Dictionary d = dictionaries.next();
                definition = d.getDefinition(word);
            }
        } catch (ServiceConfigurationError serviceError) {
            definition = null;
            serviceError.printStackTrace();

        }
        return definition;
    }
}

El ejemplo almacena el fichero de clase compilado en el directorio DictionaryServiceProvider/build.

La clase DictionaryService implementa el patrón de diseño singleton. Esto significa que sólo se crea una única instancia de la clase DictionaryService. Vea la sección El Patrón de Diseño Singleton para más información.

La clase DictionaryService es el punto de entrada del cliente del servicio del diccionario para usar cualquier proveedor de servicio de Dictionary instalado. Use el método ServiceLoader.load para recuperar el miembro estático privado DictionaryService.service, el punto de entrada al servicio singleton. Después la aplicación puede llamar al método getDefinition, la cual itera a través de los proveedores de Dictionary disponibles hasta que encuentre la palabra buscada. El método getDefinition devuelve nulo si ninguna instancia de Dictionary contiene la definición especificada de la palabra.

El servicio de diccionario usa el método ServiceLoader.load para buscar la clase objetivo. El SIP está definido por la interfaz dictionary.spi.Dictionary, así que el ejemplo usa esta clase como el argumento del método load. El método por defecto load busca en la ruta de clases de la aplicación con el cargador de clases por defecto.

Sin embargo, una versión sobreescrita de este método le habilita a especificar cargadores de clase personalizados si lo desea. Eso le habilita a hacer búsquedas de clases más sofisticadas. Un programador particularmente entusiasta podría, por ejemplo, crear una instancia de ClassLoader que puede buscar en subdirectorios específicos-de-la-aplicación que contengan los JARs de proveedores agregados durante la ejecución. El resultado es una aplicación que no requerie un reinicio para acceder a nuevas clases de proveedor.

Después de que un cargador para este clase exista, puede usar su método iterador para acceder y usar cada proveedor que éste encuentra. El método getDefinition usa un iterador de Dictionary para recorrer los proveedores hasta que enecuentra una definición para la palabra especificada. El método iterador almacena en memoria las instancias Dictionary, de forma que las llamadas sucesivas requieren menos tiempo de procesado. Si se han colocado nuevos proveedores en el servicio desde la última invocación, el método iterador los agrega a a la lista.

La clase DictionaryDemo.java usa este servicio. Para usar el servicio, la aplicación obtiene una instancia de DictionaryService y llama al método getDefinition. Si una definición está disponible, la aplicación la imprime. Si no está disponible la definición, la aplicación imprime un mensaje indicando que ningún diccionario disponible contiene la palabra.

El Patrón de Diseño Singleton

Un patrón de diseño es una solución general a un problema común en el diseño de software. La idea es que la solución puede ser translada a código, y que el código puede ser aplicado en situaciones diferentes donde el problema ocurre. El patrón singleton describe una técnica que se asegura que sólo una única instancia de una clase sea creada. En esencia, la técnica toma la siguiente aproximación: no permite a nadie fuera de la clase a crear instancias del objeto.

Por ejemplo, la clase DictionaryService implementa el patrón singleton como sigue:

  • Declara el constructor de DictionaryService como private, lo cual previene que todas las demás clases, excepto DictionaryService, creen instancias del mismo.
  • Define la variable miembro service de DictionaryService como static, lo que asegura de que sólo exista una única instancia de DictionaryService.
  • Define el método getInstance, el cual habilita el acceso controlado a otras clases a la variable miembro service de DictionaryService.

3. Implementar el Proveedor de Servicios

Para proporcionar este servicio, debe crear una implementación de Dictionary.java. Para mantener las cosas simples, cree un diccionario general que defina sólo unas pocas palabras. Puede implementar el diccionario con una base de datos, un conjunto de ficheros de propiedades, o cualquier otra tecnología. La forma más fácil de demostrar el patrón del proveedor es incluir todas las palabras y definiciones en un único fichero.

El código siguiente le muestra una implementación de la SPI Dictionary, la clase GeneralDictionary.java. Note que proporciona un constructor sin argumentos e implementa el método getDefinition definido por la SPI. 


package dictionary;

import dictionary.spi.Dictionary;
import java.util.SortedMap;
import java.util.TreeMap;

public class GeneralDictionary implements Dictionary {

    private SortedMap<String, String> map;
    
    public GeneralDictionary() {
        map = new TreeMap<String, String>();
        map.put(
            "book",
            "a set of written or printed pages, usually bound with " +
                "a protective cover");
        map.put(
            "editor",
            "a person who edits");
    }

    @Override
    public String getDefinition(String word) {
        return map.get(word);
    }

}

El ejemplo almacena el fichero de clase compilado en el directorio GeneralDictionary/build. Nota: Debe compilar las clases dictionary.DictionaryService y dictionary.spi.Dictionary antes de la clase GeneralDictionary.

El proveedor GeneralDictionary para este ejemplo define simplemente dos palabras: book y editor. Obviamente, un diccioario más utilizable proporcionaría una lista más substancial de vocabulario usado generalmente.

Para demostrar cómo múltiples proveedores pueden implementar la misma SPI, el código siguiente muestra aún otro posible proveedor. El proveedor de servicio ExtendedDictionary.java es un diccionario extendido conteniendo términos técnicos familiares para la mayoría de desarrolladores de software. 


package dictionary;

import dictionary.spi.Dictionary;
import java.util.SortedMap;
import java.util.TreeMap;

public class ExtendedDictionary implements Dictionary {

        private SortedMap<String, String> map;

    public ExtendedDictionary() {
        map = new TreeMap<String, String>();
        map.put(
            "xml",
            "a document standard often used in web services, among other " +
                "things");
        map.put(
            "REST",
            "an architecture style for creating, reading, updating, " +
                "and deleting data that attempts to use the common " +
                "vocabulary of the HTTP protocol; Representational State " +
                "Transfer");
    }

    @Override
    public String getDefinition(String word) {
        return map.get(word);
    }

}

El ejemplo almacena el fichero de la clase compilada en el directorio ExtendedDictionary/build. Nota: Debe compilar las clases dictionary.DictionaryService y dictionary.spi.Dictionary antes de la clase ExtendedDictionary.

Es fácil imaginar a los clientes usando un conjunto completo de proveedores de Dictionary para sus propias necesidades especiales. La IPA del cargador de servicio les habilita a agregar nuevos diccionarios a su aplicación a medida que cambian sus necesidades o preferencias. Ya que la aplicación procesadora de palabras subyacente es extensible, no se requiere codificación adicional para que los clientes usen los nuevos proveedores.

4. Registrar Proveedores de Servicio

Para registrar su proveedor de servicio, cree un fichero de configuración de proveedor, el cual se almacena en el directorio META-INF/services del fichero JAR del proveedor de servicio. El nombre del fichero de configuración es el nombre de clase completamente cualificados del proveedor de servicio, en el cual cada componentes del nombre se separa por un punto (.), y las clases anidadas se separan por un signo de dolar ($).

El fichero de configuración del proveedor contiene los nombres completamente cualificados de las clases de sus proveedores de servicio, un nombre por línea. El fichero debe estar codificado en UTF-8. Adicionalmente, puede incluir comentarios en el fichero empezando la línea de comentarios con el signo de almohadilla (#).

Por ejemplo, para registrar el proveedor de servicio GeneralDictionary cree un fichero de texto llamado dictionary.spi.Dictionary . Este fichero contiene una línea: 

dictionary.GeneralDictionary

De forma similar, para registrar el proveedor de servicio ExtendedDictionary cree un fichero de texto llamado dictionary.spi.Dictionary . Este fichero contiene una línea: 

dictionary.ExtendedDictionary

5. Crear un Cliente Que Use el Servicio y los Proveedores de Servicio

Debido a que el desarrollo de una aplicación de procesador de texto completo es una tarea importante, este tutorial proporciona una aplicación más simple que usa DictionaryService y el SPI de Dictionary. El ejemplo de DictionaryDemo busca por las palabras book, editor, xml, y REST desde cualesquiera proveedores de Dictionary en la ruta de clases y recuperar sus definiciones.

Lo siguiente es el ejemplo DictionaryDemo. Este requiere una definición de la palabra objetivo desde la instancia DictionaryService, la cual pasa la petición a sus proveedores Dictionary conocidos. 


package dictionary;

import dictionary.DictionaryService;

public class DictionaryDemo {

  public static void main(String[] args) {

    DictionaryService dictionary = DictionaryService.getInstance();
    System.out.println(DictionaryDemo.lookup(dictionary, "book"));
    System.out.println(DictionaryDemo.lookup(dictionary, "editor"));
    System.out.println(DictionaryDemo.lookup(dictionary, "xml"));
    System.out.println(DictionaryDemo.lookup(dictionary, "REST"));
  }

  public static String lookup(DictionaryService dictionary, String word) {
    String outputString = word + ": ";
    String definition = dictionary.getDefinition(word);
    if (definition == null) {
      return outputString + "Cannot find definition for this word.";
    } else {
      return outputString + definition;
    }
  }
}

El ejemplo almacena el fichero de clase compilada en el directorio DictionaryDemo/build. Nota: Debe compilar las clases dictionary.DictionaryService y dictionary.spi.Dictionary antes de la clase DictionaryDemo.

La Clase ServiceLoader

La clase java.util.ServiceLoader le ayuda a encontrar, cargar, y usar proveedores de servicio. Busca proveedores de servicio en la ruta de clases de su aplicación o en su directorio de extensiones del entorno de ejecución. Los carga y capacita a su aplicación a usar las IPAs del proveedor. Si agrega nuevos proveedores a la ruta de clases o al directorio de extensiones del entorno de ejecución, la clase ServiceLoader los encuentra. Si su aplicación conoce la interfaz del proveedor, puede encontrar y usar diferentes implementaciones de esa interfaz. Puede usar la primera instancia a cargar de la interfaz o iterar a través de todas las interfaces disponibles.

La clase ServiceLoader es final, lo que significa que no puede hacer subclases o sobreescribir sus algoritmos de carga. No puede, por ejemplo, cambiar su algoritmo para buscar servicios desde una ubicación diferente.

Desde la perspectiva de la clase ServiceLoader, todos los servicios tienen un tipo único, el cual es usualmente un interfaz única o una clase abstracta. El proveedor en sí mismo contiene uno o más clases concretas que extienden el tipo de servicio con una implementación específica para su propósito. La clase ServiceLoader requiere que el tipo de proveedor único expuesto tenga un constructor por defecto, que no requiere argumentos. Esto permite a la clase ServiceLoader instanciar fácilmente los proveedores de servicio que encuentra.

Los proveedores son ubicados e instanciados a demanda. Un cargador de servicio mantiene una caché de los proveedores que fueron cargados. Cada invocación del método iterator del cargador devuelve un iterador que primero produce todos los elementos de la caché, en orden de instanciación. Después el cargador de servicios localiza e instancia cualquier proveedor nuevo, agregando cada uno a la cache a su vez. Puede eliminar la cache de proveedores con el método reload.

Para crear un cargador para una clase específica, proporciona la clase en sí misma al método load o al método loadInstalled. Puede usar los cargadores de clases por defecto o proporcionar su propia subclase de ClassLoader.

El método loadInstalled busca en el directorio de extensín del entorno de ejecución por los proveedores instalados en ejecución. La ubicación por defecto de las extensiones es el directorio jre/lib/ext de su entorno de ejecución. Debe usar la ubicación de la extensión solo para proveedores de confianza conocidos porque esta ubicación se convierte en parte de la ruta de clases para todas las aplicaciones. En este artículo, los proveedores no usan el directorio de extensiones sino que dependerán de una ruta de clase específica de la aplicación.

Limitaciones de la IPA de ServiceLoader

La IPA de ServiceLoader es útil, pero tiene limitaciones. Por ejemplo, es imposible derivar una clase desde la clase ServiceLoader, así que no puede modificar su comportamiento. Puede usar subclases personalizadas de ClassLoader para cambiar cómo se encuentra las clases, pero ServiceLoader en sí misma no puede ser extendida. Además, la clase ServiceLoader acctual no puede indicar a su aplicación cuando están disponibles nuevos proveedores en tiempo de ejecución. Adicionalmente, no puede agregar change-listeners al cargador para saber si un proveedor nuevo fue colocado en un directorio de extensión específico de la aplicación.

La IPA pública de ServiceLoader está disponible en Java SE 7. Aunque el servicio cargador existía ya en el KDJ 1.3, la IPA era privada y sólo estaba disponible para el código interno del entorno de ejecución de Java.

Sumario

Las aplicaciones extensibles proporcionan puntos de servicio que pueden ser extendidos por proveedores de servicio. La forma más fácil de crear una aplicación extensible, es usar ServiceLoader, que está disponible en Java SE 6 y posteriores. Usando esta clase, puede agregar implementaciones de proveedores a la ruta de clases de la aplicación para hacer disponibles nuevas funcionalidade. La clase ServiceLoader es final, así que no puede modificar sus habilidades.
