Entendiendo la Carga de Clases de Extensión

El marco de trabajo de extensión hace uso del mecanismo de delegación de carga de clases. Cuando el entorno de ejecución necesita cargar una clase nueva para una aplicación, busca la clase en las siguientes ubicaciones, en orden:

  1. Clases de bootstrap: las clases del entorno de ejecución en rt.jar, las clases de internacionalización en i18n.jar, y otras.
  2. Extensiones instaladas: las clases en los ficheros JAR en el directorio lib/ext del EEJ, y en el directorio de extensiones específico para la plataforma y para todo el sistema (tal como /usr/jdk/packages/lib/ext en el Sistema Operativo Solaris™, pero note que el uso de este directorio se aplica sólo a Java™ 6 y posteriores).
  3. La ruta de clases: clases, incluyendo clases en ficheros JAR, en las rutas indicadas por la propiedad del sistema java.class.path. Si un fichero JAR en la ruta de clases tiene un manifiesto con el atributo Class-Path, los ficheros JAR especificados por el atributo Class-Path serán buscados también. Por defecto, el valor de la propiedad java.class.path es ., el directorio actual. Puede cambiar el valor usando las opciones de línea de comandos -classpath o -cp, o establecer la variable de entorno CLASSPATH. Las opciones de la línea de comando sobreescriben el establecimiento de la variable de entorno CLASSPATH.

La lista precedente le indica, por ejemplo, que la ruta de clases se busca sólo si una clases a ser cargada no ha sido encontrada entre las clases en rt.jar, i18n.jar o las extensiones instaladas.

A menos que su software instancie sus propios cargadores de clases para propósitos especiales, no necesita realmente saber mucho más que tener en mente esta lista de precedencia. En particular, debería estar prevenido de cualquier conflicto de nombres de clase que podrían presentarse. Por ejemplo, si enumera una clase en la ruta de clases, obtendrá resultados inesperados si el entorno de ejecución carga otra clase con el mismo nombre que encontró en una extensión instalada.

El Mecanismo de Carga de Clases de Java

La plataforma Java usa un modelo de delegación para cargar clases. La idea básica es que cada cargador de clases tiene un cargador de clases "padre". Cuando carga una clase, un cargador de clase primero "delega" la búsqueda de la clase en su cargador de clases padre antes de intentar encontrar la clase por sí mismo.

A continuación, se muestran algunos aspectos destacados de la IPA de carga de clases:

  • Los constructores en java.lang.ClassLoader y sus subclases le permiten especificar un padre cuando instancia un nuevo cargador de clases. Si no especifica explícitamente un padre, se asignará el cargador de clases del sistema de la máquina virtual como padre por defecto.
  • El método loadClass en ClassLoader realiza estas tareas, en orden, cuando se llama a una clase:
    1. Si una clase ha sido ya cargada, la devuelve.
    2. De otra forma, delega la búsqueda para la nueva clase al cargador de clases padre.
    3. Si el cargador de clases padre no encuentra la clase, loadClass llama al método findClass para bsucar y cargar la clase.
  • El método findClass de ClassLoader busca por la clase en el cargador de clases actual si la clase no fue encontrada por el cargador de clases padre. Probablemente desee sobreescribir este método cuando cree una instancia de una subclase del cargador de clases en su aplicación.
  • La clase java.net.URLClassLoader sirve como el cargador de clases básico para las extensiones y otros ficheros JAR, sobreescribiendo el método findClass de java.lang.ClassLoader para buscar una o más URLs especificadas por clases y recursos.

Para ver una aplicación de muestra que utiliza algunas de las IPA en lo que respecta a los archivos JAR, consulte la lección Usando IPAs relacionadas con el formato JAR en este tutorial.

Carga de Clases y el Comando java

El mecanismo de carga de clases de la plataforma Java se refleja en el comando java.

  • En la herramienta java, la opción -classpath es un forma corta de establecer la propiedad java.class.path.
  • Las opciones -cp y -classpath son equivalentes.
  • La opción -jar ejecuta aplicaciones que están empaquetadas en ficheros JAR. Para una descripción y ejemplos de esta opción, vea la lección Ejecutando Software Empaquetado en Ficheros JAR en este tutorial.
