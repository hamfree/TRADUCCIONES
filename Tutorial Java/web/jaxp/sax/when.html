Cuando usar SAX

Es útil comprender el modelo de eventos SAX cuando desee convertir datos existentes a XML. La clave del proceso de conversión es modificar una aplicación existente para entregar eventos SAX mientras lee los datos.

SAX es rápido y eficiente, pero su modelo de eventos lo hace más útil para este tipo de filtrado independiente del estado. Por ejemplo, un analizador SAX llama a un método en su aplicación cuando se encuentra una etiqueta de elemento y llama a un método diferente cuando se encuentra texto. Si el procesado que está haciendo es independiente del estado (lo que significa que no depende de los elementos que han venido antes), entonces SAX funciona bien.

Por otro lado, para el procesado dependiente del estado, donde el programa necesita hacer una cosa con los datos bajo el elemento A pero algo diferente con los datos bajo el elemento B, entonces un analizador de extensión tal como la IPA de Transmisión para XML (StAX) sería una elección mejor. Con un analizador de extracción, obtiene el próximo nodo, sea el que sea, en cualquier punto en el código que lo solicite. Así es fácil variar la forma en la que procesa el texto (por ejemplo), porque puede procesarlo en múltiples lugares en el programa (para más detalle, vea Información Adicional).

SAX requiere mucha menos memoria que DOM, porque SAX no construye una representación interna (árbol de estructura) de los datos XML, como un DOM hace. En vez de eso, SAX simplemente envía datos a la aplicación mientras los lee; su aplicación puede despues hacer lo que quiere con los datos que ve.

Los analizadores de extracción y la IPA de SAX actúan como un flujo en serie de E/S. Ve los datos como en los flujos, pero no puede volver atrás a una posición anterior o saltar hacia adelante a una posición diferente. En general, tales analizadores trabajan bien cuando simplemente quiere leer datos y hacer que la aplicación actúe sobre ellos.

Pero cuando necesita modificar una estructura XML — especialmente cuando necesita modificarla interactivamente — una estructura en memoria tiene más sentido. DOM es uno de tales modelos. Sin embargo, aunque DOM proporciona muchas capacidades poderosas para documentos de gran escala (como libros y artículos), también requiere un montón de codificación compleja. Los detalles de ese proceso se resaltan en Cuándo Usar DOM en la lección siguiente.

Para aplicaciones más simples, esa complejidad puede muy bien ser innecesaria. Para un desarrollo más rápido y aplicaciones más simples, uno de los estándares de la programación XML orientado a objetos, tal como JDOM (http://www.jdom.org) y DOM4J (http://www.dom4j.org/), podrían tener más sentido.
