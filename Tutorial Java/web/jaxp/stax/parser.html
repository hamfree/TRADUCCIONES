Implementación del Analizador de Flujos XML de Oracle

El Servidor de Aplicaciones 9.1 incluye la implementación JSR 173 (StAX) de Sun Microsystems, llamado el Analizador de Transmisión XML de Sun Java (conocido como Analizador Transmisión XML). El Analizador de Transmisión XML es un analizador de extracción de transmisión XML de alta velocidad, no validador, y que cumple con W3C XML 1.0 y Namespace 1.0 contruído sobre la base de código Xerces2.

En la implementación del Analizador de Transmisión XML de Sun, las capas más bajas de Xerces2, particularmente Scanner y las clases relacionadas, han sido rediseñadas para comportarse de forma extractiva. Además de los cambios en las capas inferiores, el Analizador de Transmisión de XML incluye funciones adicionales relacionadas con StAX y muchas mejoras de rendimiento. El Analizador de Transmisión XML se implementa en los ficheros appserv-ws.jar y javaee.jar, los cuales están localizados en el directorio install_dir/lib/.

Junto con la implementación de referencia de JAXP están los ejemplos de código StAX, ubicados en el directorio INSTALL_DIR/jaxp-version/samples/stax, que ilustran cómo trabaja la implementación del Analizador de Transmisión XML de Sun. Estos ejemplos están descritos en Código de Ejemplo.

Antes de que proceda con el código de ejemplo, hay dos aspectos del Analizador de Transmisión XML que debe tener en cuenta:

  • Informar Eventos CDATA

  • Implementación de las Factorías del Analizador de Transmisión XML

Estos temas se tratan a continuación.

Informar Eventos CDATA

El javax.xml.stream.XMLStreamReader implementado en el Analizador de Transmisión XML no informa de los eventos CDATA. Si tiene una aplicación que necesita recibir dichos eventos, configure la XMLInputFactory para que establezca la siguiente propiedad específica de la implementación report-cdata-event: 

XMLInputFactory factory = XMLInptuFactory.newInstance();
factory.setProperty("report-cdata-event", Boolean.TRUE);

Implementación de Factorías de Analizadores de Retransmisión XML

La mayoría de las aplicacoines no necesitan saber el nombre de la clase de implementación de la factoría. Simplemente agregando los ficheros javaee.jar y appserv-ws.jar a la ruta de clases es suficiente para la mayoría de las aplicaciones porque estos dos jars proporcionan el nombre de la clase de implementación de la factoría para varias propiedades del Analizador de Transmisisión XML bajo el directorio META-INF/services — por ejemplo, javax.xml.stream.XMLInputFactory, javax.xml.stream.XMLOutputFactory, y javax.xml.stream.XMLEventFactory — que es el tercer paso de una operación de búsqueda cuando una aplicación pregunta por la instancia de factoría. Ve la documentación Java para el método XMLInputFactory.newInstance para obtener más información sobre el mecanismo de búsqueda.

Sin embargo, puede haber escenarios en los que una aplicación desea conocer el nombre de la clase de implementación de la factoría y establecer la propiedad explícitamente. Estos escenarios podrían incluir casos donde hay múltiples implementaciones del JSR 173 en la ruta de clases y la aplicación quiere elegir una, quizás una que tenga rendimiento superior, contenga una corrección de error crucial, o algo similar.

Si una aplicación establece la SystemProperty, es el primer paso en una operación de búsqueda, por lo que obtener la instancia de la factoría sería rápido en comparación con otras opciones; por ejemplo: 

javax.xml.stream.XMLInputFactory -->
  com.sun.xml.stream.ZephyrParserFactory

javax.xml.stream.XMLOutputFactory -->
  com.sun.xml.stream.ZephyrWriterFactor

javax.xml.stream.XMLEventFactory -->
  com.sun.xml.stream.events.ZephyrEventFactory
