¿Porqué StAX?

El proyecto StAX fue encabezado por BEA con el apoyo de Sun Microsystems, y la especficación JSR 173 pasó la votación de aprobación final del Java Community Process en marzo de 2004. El objetivo primario de la IPA de StAX es “dar control de análisis al programador mediante la exposición de una IPA basada en iterador simple. Esto permite al programador solicitar el siguiente evento (extraer el evento) y permite que el estado se almacene de forma procedimental.” StAX fue creado para abordar las limitaciones en las dos IPA de análisis más frecuentes, SAX y DOM.

Retransmisión contra DOM

Generalmente hablando, hay dos modelos de programación para trabajar con conjuntos de información XML: retransmisión y el modelo de objetos de documento (DOM).

El modelo DOM implica la creación de objetos en memoria que representan un árbol de documento entero y el estado del conjunto de información completo para un documento XML. Una vez en memoria, se pueden navegar libremente los árboles DOM y analizarse arbitrariamente y, como tal, proporcionan la máxima flexibilidad para los desarrolladores. Sin embargo, el coste de esta flexibilidad es una huella de memoria potencialmente grande y requisitos de procesador significativos, porque la representación entera del documento debe ser mantenida en memoria como objetos durante la duración del procesado del documento. Esto puede no ser un problema cuando trabaja con documentos pequeños, pero los requerimientos de memoria y procesador pueden escalar rápidamente con el tamaño del documento.

La transmisión se refiere a un modelo de programación en el cual los conjuntos de información XML se transmiten y analizan serialmente en tiempo de ejecución de la aplicación, a menudo en tiempo real, y a menudo desde fuentes dinámicas cuyos contenidos no son precisamente conocidos de antemano. Además, los analizadores basados en flujo pueden comenzar a generar resultados inmediatamente, y los elementos del conjunto de información se pueden descartar y recolectar la basura inmediatamente después de su uso. Mientras proporcionan una huella de memoria más pequeña, y requerimientos de procesador reducidos, y un rendimiento más alto en ciertas situaciones, la principal desventaja con el procesamiento de secuencias es que sólo puede ver el estado del conjunto de información en una ubicación a la vez en el documento. Está esencialmente limitado a la vista de “tubo de cartón” de un documento, lo que implica que necesita saber qué procesamiento desea hacer antes de leer el documento XML.

Los modelos de transmisión para el procesado de XML son particularmente útiles cuando su aplicación tiene limitaciones estrictas de memoria, como con un teléfono móvil ejecutando la Plataforma Java, Edición Micro (Plataforma Java ME), o cuando su aplicación necesita procesar varias peticiones simultáneamente, como con un servidor de aplicaciones. En realidad, se puede argumentar que la mayoría de la lógica empresarial XML puede beneficiarse del procesamiento de flujo y no requiere el mantenimiento en memoria de árboles DOM completos.

Análisis de Extracción contra Análisis de Inserción

El análisis de extracción de transmisión se refiere a un modelo de programación en el cual una aplicación cliente llama a métodos en una librería de análisis XML cuando necesita interacturar con un conjunto de información XML—es decir, el cliente sólo obtiene (extrae) datos XML cuando los solicita explícitamente.

El análisis de inserción de transmisión se refiere a un modelo de programación en el cual un analizador XML envía (inserta) datos XML al cliente cuando el analizador encuentra elementos en un conjunto de información XML—es decir, el analizador envía los datos ya esté o no listo el cliente para usarlos en ese momento.

El análisis de extracción proporciona varias ventajas sobre el análisis de inserción cuando se trabaja con flujos XML:

  • Con el análisis de extracción, el cliente controla el hilo de la aplicación, y puede llamar a los métodos en el analizador cuando lo necesita. Por el contrario, con el procesamiento de inserción, el analizador controla el hilo de la aplicación y el cliente sólo puede aceptar invocaciones del analizador.

  • Las librerías de análisis de extracción pueden ser mucho más pequeñas y el código de cliente para interactuar con esas librerías más simple que con las librerías de inserción, incluso para los documentos más complejos.

  • Los clientes de extracción pueden leer varios documentos a la vez con un sólo hilo.

  • Una analizador de extracción StAX puede filtrar documentos XML de forma que se puedan ignorar los elementos innecesarios para el cliente y puede admitir vistas XML de datos no XML.

Casos de Uso de StAX

La especificación StAX define una serie de casos de uso para la IPA:

  • Enlace de datos

    • Desagrupamiento de un documento XML

    • Ordenamiento de un documento XML

    • Procesamiento de documentos en paralelos

    • Comunicación inalámbrica

  • Procesamiento de mensajes del Protocolo Simple de Acceso a Objetos (SOAP)

    • Analizar estructuras predecibles simples

    • Analizar representaciones de gráficos con referencias directas

    • Análisis del Lenguaje de Descripción de Servicios Web (WDSL)

  • Fuentes de datos virtuales

    • Ver como datos XML almacenados en bases de datos

    • Ver datos en objetos Java creados por enlace de datos XML

    • Navegar por un árbol DOM como una secuencia de eventos

  • Análisis de vocabularios XML específicos

  • Procesamiento XML canalizado

Un descripción completa de todos estos casos de uso está más allá del alcance de esta lección. Por favor refiérase a la especificación StAX para obtener más información.

Comparando StAX con Otras IPAs de JAXP

Como una IPA en la familia JAXP, StAX puede compararse, entre otras IPAs, a SAX, TrAX, y JDOM. De las dos últimas, StAX no es tan potente o flexible como TrAX o JDOM, pero tampoco requiere tanta memoria o carga de procesador para ser útil, y StAX puede, en muchos casos, superar a las IPAs basadas en DOM. Los mismos argumentos descritos anteriormente, que sopesan el costo/beneficio del modelo DOM frente al modelo de transmisión, se aplican aquí.

Con esto en mente, se pueden hacer las comparaciones más cercanas entre StAX y SAX, y es aquí donde StAX ofrece características que son beneficiosas en muchos casos; algunas de estas incluyen:

  • Los clientes habilitados para StAX son generalmente más fáciles de codificar que los clientes SAX. Si bien se puede argumentar que los analizadores SAX son un poco más fáciles de escribir, el código del analizador StAX puede ser más pequeño y el código necesario para que el cliente interactúe con el analizador más simple.

  • StAX es una IPA bidireccional, lo que significa que puede leer y escribir documentos XML. SAX es de sólo lectura, por lo que se necesita otra IPA si se desea escribir documentos XML.

  • SAX es una IPA de envío, mientras que StAX es de extracción. Las compensaciones entre las IPAs de envío y extracción descritas anteriormente se aplican aquí.
