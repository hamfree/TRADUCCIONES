IPA de StAX

La IPA de StAX expone métodos para el procesado iterativos, basado en eventos de documentos XML. Los documentos XML son tratados como series filtradas de eventos, y los estados de los conjuntos de información pueden almacenarse de forma procedimental. Es más, a diferencia de SAX, la IPA de StAX es bidireccional, lo que permite tanto la lectura como la escritura de documentos XML.

La IPA de StAX son realmente dos conjuntos de IPA distintas: una IPA de cursor y una IPA de iterador. Estos dos conjuntos de IPA se explican en gran detalle más tarde en esta lección, pero sus características principales se describen brevemente debajo.

IPA de Cursor

Como el nombre implica, la IPA de StaX cursor representa un cursor con el cual puede recorrer un documento XML, desde el principio hasta el final. Este cursor puede apuntar a una cosa a la vez, y siempre se mueve hacia delante, nunca hacia atrás, habitualmente un elemento del conjunto de información a la vez.

Las dos interfazes principales de cursor son XMLStreamReader y XMLStreamWriter. XMLStreamReader incluye métodos de acceso para toda la posible información recuperable del modelo de información XML, incluyendo la codificación del documento, nombres de elemento, atributos, espacios de nombres, nodos de texto, etiquetas de inicio, comentarios, instrucciones de procesamiento, límites del documento, etcétera; por ejemplo: 

public interface XMLStreamReader {
    public int next() throws XMLStreamException;
    public boolean hasNext() throws XMLStreamException;

    public String getText();
    public String getLocalName();
    public String getNamespaceURI();
    // ... other methods not shown
}

Puede llamar a los métodos en XMLStreamReader, tales como getText y getName, para obtener datos de la ubicación actual del cursor. XMLStreamWriter proporciona métodos que corresponden a los tipos de evento StartElement y EndElement; por ejemplo: 

public interface XMLStreamWriter {
    public void writeStartElement(String localName) throws XMLStreamException;
    public void writeEndElement() throws XMLStreamException;
    public void writeCharacters(String text) throws XMLStreamException;
    // ... other methods not shown
}

La IPA de cursor refleja SAX de muchas formas. Por ejemplo, los métodos están disponibles para acceder directamente a la información de cadena y caracteres, y los índices enteros pueden usarse para acceder a la información de atributos y espacios de nombres. Como con SAX, los métodos de la IPA de cursor devuelven la información XML como cadenas, lo que minimiza los requisitos de asignación de objetos.

IPA de Iterador

La IPA iterador de StAX representa un flujo de un documento XML como un conjunto de objetos de eventos discretos. La aplicación extrae estos eventos y el analizador los proporciona en el orden en que se leen en documento XML de origen.

La interfaz base del iterador se llama XMLEvent, y hay subinterfaces para cada tipo de evento listado en la Tabla XMLEvent. La interfaz del analizador primario para leer eventos del iterador es XMLEventReader, y la interfaz primaria para escribir eventos del iterador es XMLEventWriter. La interfaz XMLEventReader contiene cinco métodos, el más importante de los cuales es nextEvent, que devuelve el siguiente evento en un flujo XML. XMLEventReader implementa java.util.Iterator, lo que significa que lso retornos de XMLEventReader se pueden almacenar en caché o pasar a rutinas que pueden funcionar con el iterador de Java estándar; por ejemplo: 

public interface XMLEventReader extends Iterator {
    public XMLEvent nextEvent() throws XMLStreamException;
    public boolean hasNext();
    public XMLEvent peek() throws XMLStreamException;
    // ...
}

Del mismo mod, en el lado de salida de la IPA del iterador, tiene: 

public interface XMLEventWriter {
    public void flush() throws XMLStreamException;
    public void close() throws XMLStreamException;
    public void add(XMLEvent e) throws XMLStreamException;
    public void add(Attribute attribute) throws XMLStreamException;
    // ...
}

Tipos de Eventos de Iterador

XMLEvent Tipos Definidos en la IPA del Iterador de Eventos

Tipo de Evento

Descripción

StartDocument

Informa del principio de un conjunto de eventos XML, incluyendo la codificación, la versión de XML, y propiedades independientes.

StartElement

Informa del inicio de un elemento, incluyendo cualquier atributo y declaraciones de espacios de nombre; también proporciona acceso al prefijo, la URI del espacio de nombres, y el nombre local de la etiqueta de inicio.

EndElement

Informa de la etiqueta final de un elemento. Los espacios de nombres que han salido del alcance se pueden recuperar aquí si se han establecido explícitamente en su StartElement correspondiente.

Characters

Corresponde a las secciones CData de XML y las entidades CharacterData. Tenga en cuenta que los espacios en blanco ignorables y los espacios en blanco significativos son también informados como eventos Character.

EntityReference

Las entidades de carácter pueden ser informadas como eventos discretos, que un desarrollador de aplicaciones puede optar por resolver o pasar sin resolver. Por defecto, las entidades se resuelven. Alternativamente, si no quiere informar la entidad como un evento, el texto de reemplazo puede ser sustituido e informado como Characters.

ProcessingInstruction

Reporta el objetivo y los datos para una instrucción de procesamiento subyacente.

Comment

Devuelve el texto de un comentario.

EndDocument

Informa del final de un conjunto de eventos XML.

DTD

Informa como java.lang.String información sobre la DTD, si la hay, asociada con el flujo, y proporciona un método para devolver los objetos personalizados que se encuentran en la DTD.

Attribute

Los atributos se informan generalmente como parte de un evento StartElement. Sin embargo, hay veces cuando es deseable devolver un atributo como un evento Attribute independiente; por ejemplo, cuando se devuelve un espacio de nombres como el resultado de una expresión XQuery o XPath.

Namespace

Como con los atributos, los espacios de nombre se informan usualmente como parte de un StartElement, pero hay veces en las que es deseable informar un espacio de nombres como un evento discreto Namespace.

Tenga en cuenta que los eventos DTD, EntityDeclaration, EntityReference, NotationDeclaration, y ProcessingInstruction se crean sólo si el documento que está siendo procesado contiene una DTD.

Ejemplo de Asignación de Eventos

Como un ejemplo de cómo la IPA del iterador de eventos se asigna a un flujo XML, considere el siguiente documento XML: 

<?xml version="1.0"?>
<BookCatalogue xmlns="http://www.publishing.org">
    <Book>
        <Title>Yogasana Vijnana: the Science of Yoga</Title>
        <ISBN>81-40-34319-4</ISBN>
        <Cost currency="INR">11.50</Cost>
    </Book>
</BookCatalogue>

Este documento sería analizado en dieciocho eventos principales y secundarios, como se muestra en la tabla siguiente. Tenga en cuenta que los eventos secundarios, mostrados entre llaves ({}), normalmente se acceden desde un evento principal en lugar de hacerlo directamente.

Hay varias cosas importantes a tener en cuenta en este ejemplo:

  • Se crean los eventos en el orden en el cual se encuentran los elementos XML correspondientes en el documento, incluyendo el anidado de los elementos, apertura y cierre de elementos, el orden de los atributos, el inicio del documento y el final del documento, etcétera.

  • Al igual que con la sintaxis XML adecuada, todos los elementos del contenedor tienen eventos de inicio y finalización correspondientes; por ejemplo, cada StartElement tiene un EndElement correspondiente, incluso para elementos vacíos.

  • Los eventos Attribute se tratan como eventos secundarios, y se accede a ellos desde su evento StartElement correspondiente.

  • De forma similar a los eventos Attribute, los eventos Namespace se tratan como secundarios, pero aparecen dos veces y son accesibles dos veces en el flujo de eventos, primero desde su StartElement correspondiente y después desde su EndElement correspondiente.

  • Los eventos Character se especifican para todos los elementos, incluso si esos elementos no tienen datos de carácter. De manera similar, los eventos Character se pueden dividir entre eventos.

  • El analizador StAX mantiene una pila de espacios de nombres, que mantiene información sobre todos los espacios de nombres XML definidos para el elemento actual y sus ancestros. La pila de espacios de nombres, que se expone a través de la interfaz javax.xml.namespace.NamespaceContext, puede ser accedida por el prefijo del espacio de nombres o la URI.

Elegir entre las IPAs de Cursor e Iterator

Es razonables el preguntarse en este punto,“¿Qué IPA debo elegir? ¿Debo crear instancias de XMLStreamReader o XMLEventReader? De todos modos, ¿por qué hay dos tipos de IPA? ”

Objetivos de Desarrollo

Los autores de la especificación StAX se dirigieron a tres tipos de desarrolladores:

  • Desarrolladores de librería e infraestructura: Crean servidores de aplicaciones, JAXM, JAXB, JAX-RPC e implementaciones similares; necesitan IPA de bajo nivel altamente eficientes con requisitos mínimos de extensibilidad.

  • Desarrolladores Java ME: Necesitan bibliotecas pequeñas, simples, de análisis de extracción, y tienen necesidades mínimas de extensibilidad.

  • Desarrolladores de la Plataforma Java, Plataforma Java y Edición Empresarial (Java EE), Edición Estándar (Java SE): Necesita bibliotecas de análisis de extracción limpias y eficientes, además de la flexibilidad para leer y escribir secuencias XML, crear nuevos tipos de eventos y ampliar atributos y elementos de documentos XML.

Dadas estas categorías de desarrollo de amplio alcance, los autores de StAX consideraron que era más útil definir dos IPA pequeñas y eficientes en lugar de sobrecargar una IPA más grande y necesariamente más compleja.

Comparando las IPAs de Cursor e Iterator

Antes de elegir entre las IPAs de cursor e iterador, debe tener en cuenta algunas cosas que puede hacer con la IPA del Iterador que no puede hacer con la IPA del cursor:

  • Los objetos creados desde las subclases XMLEvent son inmutables, y se pueden usar en matrices, listas, y mapas, y se pueden pasar a través de su aplicaciones incluso después de que el analizador se ha movido a los eventos siguientes.

  • Puede crear subtipos de XMLEvent que son elementos de información completamente nuevos o extensiones de elementos existentes pero con métodos adicionales.

  • Puede agregar y eliminar eventos desde un flujo de eventos XML en formas mucho más simples que con la IPA del cursor.

Del mismo modo, tenga en cuenta algunas recomendaciones generales al hacer su elección:

  • Si está programando para un entorno particularmente restringido de memoria, como Java ME, puede hacer código más pequeño y más eficiente con la IPA de cursor.

  • Si el rendimiento es su más alta prioridad — por ejemplo, cuando se crean librerías de bajo nivel o infraestructuras — la IPA del cursor es más eficiente.

  • Si quiere crear canalizaciones de procesamiento XML, use la IPA del iterador.

  • Si quiere modificar el flujo de eventos, use la IPA del iterador.

  • Si quiere que su aplicación sea capaz de manejar el procesamiento conectable del flujo de eventos, use la IPA del iterador.

  • En general, si no tiene una preferencia fuerte de una forma u otra, se recomienda usar la IPA del iterador porque es más flexible y extensible, por lo que es una “prueba de futuro” para sus aplicaciones.
