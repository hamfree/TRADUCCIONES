Lección: Problemas Comunes (y Sus Soluciones)

Problemas del Compilador

Mensajes de Error Comunes en sistemas Microsoft Windows

'javac' no es reconocido como un comando interno o externo, programa ejecutable o fichero batch

Si recibe este error, Windows no puede encontar el compilador (javac).

Aquí tiene una forma de decirle a Windows donde encontrar javac. Suponga que usted instaló el KDJ en C:\jdk1.8.0. En el prompt debería escribir el siguiente comando y presionar Enter: 

C:\jdk1.8.0\bin\javac HolaMundoApl.java

Si usted elije ésta opción, tendrá que preceder sus comandos javac y java con C:\jdk1.8.0\bin\ cada vez que compile y ejecute un programa. Para evitar este tecleo extra, consulte la sección Actualizando la variable PATH en las instrucciones de instalación del KDJ 8.

Los nombres de clase, 'HolaMundoApl', sólo se aceptan si se solicita explícitamente procesamiento de anotación

Si uste recibe este error, olvidó incluir el sufijo .java cuando compilaba el programa. Recuerde, el comando es javac HolaMundoApl.java no javac HolaMundoApl.

Mensajes de Error Comunes en Sistemas UNIX

javac: Comando no encontrado

Si recibe este error, UNIX no puede encontar el compilador, javac.

Aquí tiene una forma de decirle a UNIX dónde encontrar javac. Suponga que instaló el KDJ en /usr/local/jdk1.8.0. En el prompt escribiría el siguiente comando y presionaría Retorno: 

/usr/local/jdk1.8.0/javac HolaMundoApl.java

Nota: Si elije ésta opción, cada vez que compile o ejecute un programa, usted deberá preceder sus comandos javac y java con /usr/local/jdk1.8.0/. Para evitar este tecleo extra, usted podría añadir ésta información a su variable PATH. Los pasos para hacerlo variarán dependiendo del shell que esté actualmente ejecutando.

Los nombres de clase, 'HolaMundoApl', sólo se aceptan si se solicita explícitamente procesamiento de anotación

Si recibe este error, olvidó incluir el sufijo .java cuando compilaba el programa. Recuerde, el comando es javac HolaMundoApl.java no javac HolaMundoApl.

Errores de Sintaxis (Todas las Plataformas)

Si escribe incorrectamente parte de un programa, el compilador puede emitir un error de sintaxis. El mensaje usualmente visualiza el tipo de error, el número de línea donde el error fue detectado, el código en esa línea, y la posición del error en el código. Aquí tiene un error causado por omitir un punto y coma (;) al final de la sentencia: 

testing.java:14: `;' expected.
System.out.println("Input has " + count + " chars.")
                                                    ^
1 error

Algunas veces el compilador no podrá adivinar su intención e imprimirá un mensaje de error confuso o múltiples mensajes de error si el error se expande sobre varias líneas. Por ejemplo, el siguiente trozo de código omite un punto y coma (;) desde la línea en negrita: 

while (System.in.read() != -1)
    contador++
System.out.println("La entrada tiene " + contador + " caracteres.");

Cuando procesa este código, el compilador emite dos mensajes de error: 

testing.java:13: Invalid type expression.
        contador++
                 ^
testing.java:14: Invalid declaration.
    System.out.println("La entrada tiene " + contador + " caracteres.");
                                             ^
2 errors

El compilador emite dos mensajes de error porque después de ello procesa contador++, el estado del compilador indica que está en el medio de una expresión. Sin el punto y coma, el compilador no tiene forma de saber que la sentencia está completa.

Si ve cualquier error del compilador, entonces su programa no se compiló exitosamente, y el compilador no creó el fichero .class . Verifique cuidadosamente el programa, arregle cualquier error que detecte, e inténtelo otra vez.

Errores Semánticos

Además de verificar que su programa es sintácticamente correcto, el compilador comprueba otras normas básicas. Por ejemplo, el compilador le avisa cada vez que usted utiliza una variable que no ha sido inicializada: 

testing.java:13: Variable contador may not have been initialized.
        contador++
        ^
testing.java:14: Variable contador may not have been initialized.
    System.out.println("La entrada tiene " + contador + " caracteres.");
                                             ^
2 errors

De nuevo, su programa no se compiló exitosamente, y el compilador no creó el fichero .class. Arregle el error e inténtelo de nuevo.

Problemas en tiempo de ejecución

Mensajes de Error en Sistemas Microsoft Windows

Exception in thread "main" java.lang.NoClassDefFoundError: HolaMundoApl

Si recibe este error, java no puede encontrar su fichero de bytecode, HolaMundoApl.class.

Uno de los lugares donde java intenta encontrar su fichero .class es en su directorio actual. Así si su fichero .class está en C:\java, usted deberá cambiar su directorio actual a ese. Para cambiar su directorio, escriba el siguiente comando en el prompt y presione Intro: 

cd c:\java

El prompt debe cambiar a C:\java>. Si usted introduce dir en el prompt, usted debería ver sus ficheros .java y .class. Ahora introduzca java HolaMundoApl de nuevo.

Si usted aún tiene problemas, puede que tenga que cambiar su variable CLASSPATH. Para ver si esto es necesario, intente eliminar el classpath con el siguiente comando. 

set CLASSPATH=

Ahora introduzca java HolaMundoApl de nuevo. Si el programa funciona ahora, usted tendrá que cambiar su variable CLASSPATH. Para establecer esta variable, consulte la sección Actualizando la variable PATH en las instrucciones de instalación del KDJ 8. La variable CLASSPATH es establecida de la misma manera.

Could not find or load main class HolaMundoApl.class

Un error común realizado por programadores principiantes es intentar ejecutar el lanzador java sobre el fichero .class que fue creado por el compilador. Por ejemplo, usted obtendrá este error si intenta ejecutar su programa con java HolaMundoApl.class en vez de java HolaMundoApl. Recuerde, el argumento es el nombre de la clase que quiere usar , no el nombre del fichero.

Exception in thread "main" java.lang.NoSuchMethodError: main

La MV (Máquina Virtual) de Java requiere que la clase que ejecute tenga un método main con el cual empezar la ejecución de su aplicación. Una Mirada Más Cercana a la Aplicación "¡Hola Mundo!" analiza el método main en detalle.

Mensajes de Error en Sistemas UNIX

Exception in thread "main" java.lang.NoClassDefFoundError: HolaMundoApl

Si usted recibe este error, java no puede encontrar su fichero bytecode, HolaMundoApl.class.

Uno de los lugares donde java intenta encontrar su fichero bytecode es su directorio actual. Así, por ejemplo, si su fichero bytecode está en /home/jdoe/java, usted deberá cambiar su directorio actual a ese. Para cambiar su directorio, escriba el siguiente comando en el prompt y pulse Intro: 

cd /home/jdoe/java

Si usted introduce pwd en el prompt, usted debería ver /home/jdoe/java. Si usted introduce ls en el prompt, usted debería ver sus ficheros .java y .class. Ahora introduzca java HolaMundoApl de nuevo.

Si usted tiene aún problemas, tendría que cambiar su variable de entorno CLASSPATH. Para ver si esto es necesario, intente cambiar el classpath con el siguiente comando. 

unset CLASSPATH

Ahora introduzca java HolaMundoApl de nuevo. Si el programa funciona ahora, usted tendrá que cambiar su variable CLASSPATH de la misma forma que la variable PATH arriba.

Exception in thread "main" java.lang.NoClassDefFoundError: HolaMundoApl/class

Un error común hecho por programadores principiantes es intentar ejecutar el lanzador java sobre el fichero .class que fue creado por el compilador. Por ejemplo, obtendrá este error si intenta ejecutar su programa con java HolaMundoApl.class en vez de java HolaMundoApl. Recuerde, el argumento es el nombre de la clase que quiere usar, no el nombre del fichero.

Exception in thread "main" java.lang.NoSuchMethodError: main

La MV de Java requiere que la clase que ejecute tenga el método main por el cual empieza la ejecución de su aplicación. Una Mirada Más Cercana a la Aplicación "¡Hola Mundo!" analiza el método main en detalle.

El Applet o la Aplicación Java Web Start Está Bloqueada

Si usted está ejecutando una aplicación a través del navegador y tiene avisos de seguridad que dicen que la aplicación está bloqueada, comprueba los siguientes puntos:

  • Verifique que los atributos del fichero de manifiesto del JAR están establecidos correctamente para el entorno en el cual la aplicación se está ejecutando. El atributo Permissions es requerido. En un proyecto Netbeans, puede abrir el fichero de manifiesto desde la pestaña de Ficheros del EID Netbeans expandiendo la carpeta del proyecto y haciendo doble-click sobre el fichero manifest.mf.

  • Verifique que la aplicación está firmada por un certificado válido y que el certificado está localizado en el Almacén de Firmas de CA.

  • Si uste ejecuta un applet local, configure el servidor web para usarlo como testeador. También puede añadir su aplicación a la lista de excepciones de sitios, la cual es gestionada en la pestaña de Seguridad del Panel de Control de Java.
