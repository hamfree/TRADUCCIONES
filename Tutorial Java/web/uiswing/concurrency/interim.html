Tareas que Tienen Resultados Provisionales

Es útil a menudo para una tarea en segundo plano suministrar resultados provisionales mientras está aún trabajando. La tarea puede hacer esto invocando SwingWorker.publish. Este método acepta un número variable de argumentos. Cada argumento debe ser del tipo especificado por el segundo tipo de parámetro de SwingWorker.

Para recopilar los resultados proporcionados por publish, sobreescriba SwingWorker.process. Este método será invocado desde el hilo despachador de eventos. Los resultados de múltiples invoaciones de publish son a menudo acumulados para una invocación única de process.

Veamos la forma en que el ejemplo Flipper.java usa publish para suministrar resultados provisionales. Pulse el botón Lanzar para ejecutar Flipper usando Java™ Web Start ( descargue KDJ 7 o posterior). O, para compilar y ejecutar el ejemplo usted mismo, consulte el índice de ejemplos.


Este programa comprueba la imparcialidad de java.util.Random generando una serie de valores aleatorios booleanos en una tarea en segundo plano. Esto es equivalente a lanzar una moneda; de ahí el nombre Flipper. Para informar de sus resultados, la tarea en segundo plano usa un objeto del tipo FlipPair: 

private static class FlipPair {
    private final long heads, total;
    FlipPair(long heads, long total) {
        this.heads = heads;
        this.total = total;
    }
}

El campo heads es el número de veces que el valor aleatorio ha sido true; el campo total es el número total de valores aleatorios.

La tarea en segundo plano es representada por una instancia de FlipTask: 

private class FlipTask extends SwingWorker<Void, FlipPair> {

Ya que la tarea no devuelve un resultado final, no importa que primer tipo de parametro sea; Void es usado como una marca de posición. La tarea invoca publish después de cada "lanzamiento de moneda": 

@Override
protected Void doInBackground() {
    long heads = 0;
    long total = 0;
    Random random = new Random();
    while (!isCancelled()) {
        total++;
        if (random.nextBoolean()) {
            heads++;
        }
        publish(new FlipPair(heads, total));
    }
    return null;
}

(El método isCancelled es descrito en la siguiente sección). Ya que publish es invocado muy frecuentemetne, un montón de valores FlipPair serán probablemente acumulados antes de que process sea invocado en el hilo despachador de eventos; process sólo está interesado en el último valor informado cada vez, usándolo para actualizar el IGU: 

protected void process(List<FlipPair> pairs) {
    FlipPair pair = pairs.get(pairs.size() - 1);
    headsText.setText(String.format("%d", pair.heads));
    totalText.setText(String.format("%d", pair.total));
    devText.setText(String.format("%.10g", 
            ((double) pair.heads)/((double) pair.total) - 0.5));
}

Si Random es justo, el valor visualizado en devText deber acercarse cada vez más a 0 mientras Flipper se ejecuta.

Nota: El método setText usado en Flipper es en realidad "seguro para multihilo" como se definió en su especificación. Eso significa que podríamos prescindir de publish y process y establecer los campos de texto directamente desde el hilo de trabajo. Hemos elegido ignorar este hecho para proporcionar una demostración simple de los resultados provisionales de SwingWorker.
