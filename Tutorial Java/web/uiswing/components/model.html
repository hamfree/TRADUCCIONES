Cómo usar Modelos

La mayoría de los componentes de Swing tienen modelos. Un botón (JButton), por ejemplo, tiene un modelo (un objeto ButtonModel) que almacena el estado del botón ─ cual es su mnemotécnico de teclado, si está habilitado, seleccionado, o presionado, y así. Algunos componentes tienen múltiples modelos. Una lista (JList), por ejemplo, usa un ListModel para almacenar los contenidos de la lista, y un ListSelectionModel para monitorizar la selección actual de la lista.

A menudo no necesita saber sobre los modelos que un componente usa. Por ejemplo, los programas que usan botones usualmente tratan directamente con el objeto JButton, y no tratan de ninguna manera con el objeto ButtonModel.

¿Por qué entonces existen los modelos? La razón más grande es que los modelos le dan flexibilidad para determinar cuantos datos son almacenados y recuperados. Por ejemplo, si diseña una aplicación de hoja de cálculo que muestra datos en una tabla escasamente poblada, puede crear su propio modelo de tabla que esté optimizado para tal uso.

Los modelos tienen otros beneficios, también. Significan que los datos no se copian entre las estructuras de datos de un programa y las de los componentes Swing. Además, los modelos automáticamente propagan los cambios a todos los oyentes interesados, haciendo fácil para el IGU permanecer en sincronía con los datos. Por ejemplo, para agregar elementos a una lista puede invocar métodos en el modelo de la lista. Cuando los datos del modelo cambian, el modelo dispara eventos para la JList y cualesquiera otros oyentes registrados, y la IGU son actualizados en consecuencia.

Aunque la arquitectura del modelo Swing a veces se conoce como un diseño Modelo-Vista-Controlador (MVC), no es realmente así. Los componentes de Swing son implementados generalmente de forma que la vista y el controlador son indivisibles, implementados por un único objeto de IU (Interfaz de Usuario, UI en inglés) suministrado por la apariencia. La arquitectura de modelo de Swing se describe más acertadamente como una modelo de arquitectura separable. Si está interesado en aprender más sobre el modelo de arquitectura de Swing, vea Una Visión de Conjunto de la Arquitectura de Swing, un artículo de The Swing Connection.

Un Ejemplo: Converter

Esta sección presenta un ejemplo llamado Converter, que es una aplicación que convierte continuamente las mediciones de distancia entre las unidades métricas y de E.E.U.U. Puede ejecutar Converter ( descargue KDJ 7 o posterior). O, para compilar y ejecutar usted mismo el ejemplo, consulte el índice de ejemplo.

Como la siguiente foto muestra, Converte presenta dos deslizadores, cada uno enlazado a un campo de txto. Los enlazadores y los comapos de texto muestran los mismo datos ─ una distancia ─ pero usando dos unidades de medición distintas.

Lo importante para este programa es asegurar que un sólo modelo controla el valor de los datos. Hay varias formas de conseguir esto; lo hicimos al aplazar el modelo del control deslizante superior. El modelo del deslizador de abajo (una instancia de una clase personalizada llamada FollowerRangeModel) reenvía todas las consultas de datos al modelo del control deslizante superior (una instancia de una clase personalizada llamada ConverterRangeModel). Cada campo de texto es mantenida en sincronía con su deslizador, y vice versa, por manejadores de evento que escuchan los cambios en el valor. Se tiene cuidado para garantizar que el modelo del control deslizante superior tenga la última palabra sobre la distancia que se muestra.

Cuando empezamos a implementar los modelos personalizados de los deslizadores, primero miramos la sección de la IPA de Cómo Usar Deslizadores. Nos informó que todos los modelos de datos del deslizador deben implementar el interfaz BoundedRangeModel. La documentación de la IPA de BoundedRangeModel nos informa que el interfaz tiene una clase que la implementa llamdada DefaultBoundedRangeModel. La documentación de la IPA de DefaultBoundedRangeModel muestra que es una implementación de propósito general de BoundedRangeModel.

No usamos DefaultBoundedRangeModel directamente porque almacena los datos como enteros, y Converter usa datos en coma flotante. Así, implementamos ConverterRangeModel como una subclase de Object. Después implementamos FollowerRangeModel como una subclase de ConverterRangeModel.

Para Más Información

Para conocer los modelos de componentes individuales, consulte las páginas "Cómo" y la documentación de la IPA para los componentes individuales. Aquí tiene algunos de nuestros ejemplos que usan modelos directamente:

  • Todos menos el más simple de los ejemplos de tabla implementan modelos de datos de tabla personalizados.
  • Las demostraciones de selección de color tienen oyentes de cambios en el modelo de selección del selector de color de forma que pueden ser notificados cuando el usuario selecciona un color nuevo. En ColorChooserDemo2, la clase CrayonPanel usa directamente el color del modelo de selección para establecer el color actual.
  • El ejemplo DynamicTreeDemo establece el modelo del árbol (a una instancia de DefaultTreeModel), interactúa directamente con él, y escucha por los cambios en él.
  • ListDemo establece el modelo de datos de la lista (a una instancia de DefaultListModel) e interactúa directamente con él.
  • SharedModelDemo define una clase SharedDataModel que extiende DefaultListModel e implementa TableModel. Una JList y una JTable comparten una instancia de SharedDataModel, suministrando vistas diferentes de los datos del modelo.
  • En los ejemplos de los oyentes de eventos, ListDataEventDemo crea y usa un DefaultListModel directamente.
  • Nuestros ejemplos de contador numérico crean modelos de contadores numéricos.
  • Como ya ha visto, el ejemplo Converter define dos modelos de deslizadores personalizados.
