Un Ejemplo del Fichero de Políticas

Debajo hay un archivo completo de políticas para un usuario que quiere ejecutar ExampleGame.

La sintaxis del archivo de políticas no se describe aquí; si está interesado, vea la página Sintaxis del Archivo de Política e Implementación de Políticas Predeterminadas.

No necesita saber la sintaxis; puede siempre usar la Herramienta de Políticas para crear archivos de políticas, como se mostró en las lecciones Crear un Archivo de Políticas, Visita Rápida para Controlar Aplicaciones, y Firmando Código y Dándole Permisos.

Debajo está el archivo políticas de ejemplo, seguido por una descripción de las entradas individuales. Asuma que

  • El archivo de políticas están en la computadora de Kim, y que el almacén de claves de Kin se llama kim.keystore.
  • ExampleGame ha sido firmado con la clave privada de Terry, el creador del juego, y la correspondiente clave pública está en la entrada del almacén de claves con el alias "terry".
  • Las clases HighScore y HighScorePermissions fueron firmadas por la clave privada de la persona que las implementó (Chris), y la correspondiente clave pública está en la entrada del almacén de claves con el alias "chris".

Aquí está el archivo de políticas: kim.policy 

keystore "kim.keystore";

// Aquí está el permiso que ExampleGame necesita.
// Otorga al código firmado por "terry" el 
// HighScorePermission, si el 
// HighScorePermission fue firmado por "chris"
grant SignedBy "terry" {
  permission
    com.scoredev.scores.HighScorePermission
      "ExampleGame", signedBy "chris";
};

// Aquí está el conjunto de permisos que la clase HighScore
// necesita:
grant SignedBy "chris" {
  // La clase HighScore necesita permiso para leer 
  // "user.home" para encontrar la ubicación del archivo 
  // de puntuaciones altas

  permission java.util.PropertyPermission
    "user.home", "read";

  // Necesita permiso para leer y escribir el 
  // archivo de puntuaciones altas en sí mismo

  permission java.io.FilePermission
      "${user.home}${/}.highscore", "read,write";

  // Necesita que se le otorgue su propio permiso, 
  // de forma que pueda llamar a checkPermission
  // para ver si su llamante tiene permiso.
  // Solo le otorga el permiso
  // si el mismo permiso fue firmado por 
  // "chris"

  permission
    com.scoredev.scores.HighScorePermission 
      "*", signedBy "chris";
};

La Entrada del Almacén de Claves

Un almacén de claves es un repositorio de claves y certificados, y se usa para buscar las claves públicas de los firmantes especificados en el archivo de poíticas ("terry" y "chris" en este ejemplo).

La utilidad keytool se usa para crear y administrar almacenes de clave.

Para esta lección, asuma que a Kim le gustaría jugar con ExampleGame. Si el almacén de claves de Kin se llama kim.keystore, el archivo de políticas de Kin necesita la siguiente línea al principio: 

keystore "kim.keystore";

La Entrada ExampleGame

Una entrada del archivo de políticas especifica uno o más permisos para código de un código fuente particular - o código de una ubicación particular (URL), o código firmado por una entidad particular, o ambas.

Nuestro archivo de políticas necesita una entrada para cada juego, otorgando al código frimado por una clave del creador del juego un HighScorePermission cuyo nombre es el nombre del juego. Ese permiso permite al juego llamar a los métodos de HighScore para obtener o actualizar el valor de la puntuación más alta del usuario para ese juego particular.

La entrada requerida para ExampleGame es: 

grant SignedBy "terry" {
    permission
        com.scoredev.scores.HighScorePermission 
            "ExampleGame", signedBy "chris";
};

Exigiendo que ExampleGame esté firmada por "terry" habilita a Kim a saber que el juego es el juego real que Terry desarrolló. Para que esto funcione, Kim debe tener ya almacenado el certificado y la clave pública de Terry dentro de kim.keystore usando el alias "terry".

Tenga en cuenta que el HighScorePermission necesita estar firmado por "chris", la persona que en realidad implementó ese permiso. Esto asegura que se otorga a ExampleGame el permiso real implementado por "chris", y no cualquier otro. Como antes, para que esto funcione Kim debe tener ya almacenado el certificado de clave pública de Chris dentro de kim.keystore usando el alias "chris".

La Entrada HighScore

La entrada final en el archivo de políticas otorga permisos a la clase HighScore. Mas específicamente, otorga permisos al código firmado por "chris", que creó y firmó la clase. Requerir que la clase esté firmada por "chris" asegura que cuando ExampleGame llama a esta clase para actualizar la puntuación más alta del usuario, ExampleGame sepa de forma segura que está usando la clase original implementada por "chris".

Para actualizar el valor de puntuación más alta del usuario y que cualquier juego que la llame haga eso, la clase HighScore requiere tres permisos:

1. Permiso para leer el valor de la propiedad "user.home".

La clase HighScore almacena los valores de puntuación más alta del usuario en un archivo .highscore en el directorio de inicio del usuario. Por lo tanto esta clase necesita un java.util.PropertyPermission que le permita leer el valor de la propiedad "user.home" para encontrar exactamente dónde reside el directorio de inicio del usuario: 

permission java.util.PropertyPermission 
    "user.home", "read";

2. Permiso para leer y escribe en el mismo archivo de puntuaciones más altas.

Este permiso se necesita de forma que los métodos getHighScore y setHighScore de HighScore puedan acceder al archivo .highscore del usuario para obtener o establecer, respectivamente, la puntuación más alta actual para el juego actual.

Aquí está el permiso requerido: 

permission java.io.FilePermission
    "${user.home}${/}.highscore", "read,write";

Nota: La notación ${propName} especifica el valor de una propiedad. Por lo tanto, ${user.home} será reemplazado por el valor de la propiedad "user.home". La notación ${/} es una forma independiente de la plataforma de especificar un separador de ficheros.

3. Todos los HighScorePermissions (p.e, HighScorePermissions de cualquier nombre).

Este permiso es necesario para que el HighScore verifique que el juego que llama ha recibido un HighScorePermission cuyo nombre es el nombre del juego que funcionará. Es decir, a la clase HighScore debe también serle otorgado el permiso, ya que una comprobación de permiso requiere que todo el código actualmente en la pila tenga el permiso especificado.

Aquí está el permiso requerido: 

permission com.scoredev.scores.HighScorePermission
    "*", signedBy "chris";

Como antes, el mismo HighScorePermission necesita ser firmado por "chris", la persona que en realidad implementó el permiso.
