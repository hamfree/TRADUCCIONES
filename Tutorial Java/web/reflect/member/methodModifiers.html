Recuperando y Analizando Modificadores del Método

Existen varios modificadores que pueden ser parte de la declaración de un método:

  • Modificadores de acceso: public, protected, y private
  • Modificador que restringe a una instancia: static
  • Modificador que prohíbe la modificación del valor: final
  • Modificador que requiere anulación: abstract
  • Modificador que evita la reentrada: synchronized
  • Modificador que indica que la implementación está en otro lenguaje de programación: native
  • Modificador que fuerza el comportamiento estricto de la coma flotante: strictfp
  • Anotaciones

El ejemplo MethodModifierSpy lista los modificadores de un métodos con un nombre dado. También visualiza si el método es sintético (generado por el compilador), o de aridad variable, o un método puente (generado por el compilador para soportar interfaces genéricas). 


import java.lang.reflect.Method;
import java.lang.reflect.Modifier;
import static java.lang.System.out;

public class MethodModifierSpy {

    private static int count;
    private static synchronized void inc() { count++; }
    private static synchronized int cnt() { return count; }

    public static void main(String... args) {
	try {
	    Class<?> c = Class.forName(args[0]);
	    Method[] allMethods = c.getDeclaredMethods();
	    for (Method m : allMethods) {
		if (!m.getName().equals(args[1])) {
		    continue;
		}
		out.format("%s%n", m.toGenericString());
		out.format("  Modifiers:  %s%n",
			   Modifier.toString(m.getModifiers()));
		out.format("  [ synthetic=%-5b var_args=%-5b bridge=%-5b ]%n",
			   m.isSynthetic(), m.isVarArgs(), m.isBridge());
		inc();
	    }
	    out.format("%d matching overload%s found%n", cnt(),
		       (cnt() == 1 ? "" : "s"));

        // production code should handle this exception more gracefully
	} catch (ClassNotFoundException x) {
	    x.printStackTrace();
	}
    }
}

Unos pocos ejemplos de la salida MethodModifierSpy produce lo siguiente.

  (java.lang.Class<T>[]) throws java.lang.NoSuchMethodException,
  java.lang.SecurityException

$ java MethodModifierSpy java.lang.String compareTo
public int java.lang.String.compareTo(java.lang.String)
  Modifiers: public
  [ synthetic=false var_args=false bridge=false ]
public int java.lang.String.compareTo(java.lang.Object)
  Modifiers: public volatile
  [ synthetic=true  var_args=false bridge=true  ]
2 matching overloads found

Tenga en cuenta que Method.isVarArgs() devuelve true para Class.getConstructor(). Esto indica que la declaración del método se parece a esto: 

public Constructor<T> getConstructor(Class<?>... parameterTypes)

y no como esto: 

public Constructor<T> getConstructor(Class<?> [] parameterTypes)

Observe que la salida para String.compareTo() contiene dos métodos. El método declarado en String.java: 

public int compareTo(String anotherString);

y un segundo método sintético o bridge generado por el compilador. Esto ocurre porque String implementa la interfaz parametrizada Comparable. Durante el borrado de tipo, el tipo del argumento del método heredado Comparable.compareTo() se cambia de java.lang.Object a java.lang.String. Ya que los tipos de parámetros para los métodos compareTo en Comparable y String no coinciden ya después del borrado, la anulación no puede ocurrir. En todas las demás circunstancias, esto produciría un error en tiempo de compilación porque la interfaz no está implementada. La agregación del método puente evita este problema.

Method implementa java.lang.reflect.AnnotatedElement. Así cualquier anotación en tiempo de ejecución con java.lang.annotation.RetentionPolicy.RUNTIME puede ser recuperada. Para un ejemplo de obtener anotaciones vea la sección Examinar los Modificadores y Tipos de la Clase.
