Agrupaciones de Hilos

La mayoría de las implementaciones del executor en java.util.concurrent usan agrupaciones de hilos, los cuales consisten de hilos de trabajo. Esta clase de hilo existe y se ejecuta separadamente de las tareas Runnable y Callable y es a menudo usado para ejecutar múltiples tareas.

Usar hilos de trabajo minimiza la sobrecarga debida a la creación de hilos. Los objetos Thread usan una significante cantidad de memoria, y en una aplicación a gran escala, asignar y desasignar muchos objetos Thread crea una importante sobrecarga de administración de memoria.

Un tipo común de agrupación de hilos es la agrupación fija de hilos. Este tipo de agrupación siempre tiene el número especificado de hilos ejecutándose; si un hilo es de alguna forma terminado mientras está aún en uso, es automáticamente reemplazado con un nuevo hilo. Las tareas son submitidas a la agrupación por medio de una cola interna, la cual mantiene las tareas extra cuando hay más tareas activas que hilos.

Una ventaja importante de las agrupaciones fijas de hilos es que las aplicaciones que las usan se degradan progresivamente. Para entender esto, considere una aplicación de servidor web donde cada petición HTTP es manejada por un hilo separado. Si la aplicación simplemente crea un nuevo hilo para cada nueva petición HTTP, y el sistema recibe más peticiones que las que puede manejar inmediatamente, la aplicación parará de repente de responder a todas las peticiones cuando la sobrecarga de todos esos hilos exceda la capacidad del sistema. Con un límite en el número de hilos que pueden ser creados, la aplicación no servirá tantas peticiones HTTP tan rápidamente como llegan, pero las servirá tan rápidamente como el sistema puede sostenerlas.

Una forma simple de crear un ejecutor que usa una agrupación fija de hilos es invocar el método de factoria newFixedThreadPool en java.util.concurrent.Executors . Esta clase también ofrece los siguientes métodos factoría:

  • El método newCachedThreadPool crea un ejecutor con una agrupación de hilos expandible. Este ejecutor es adecuado para aplicaciones que lanzan muchas tareas de corta-vida.
  • El método newSingleThreadExecutor crea un ejecutor que ejecuta una tarea única a la vez.
  • Varios métodos factoría son versiones de ScheduledExecutorService de los ejecutores de arriba.

Si ninguno de los ejecutores ofrecidos por los métodos de factoría de arriba coinciden con sus necesidades, construir instancias de java.util.concurrent.ThreadPoolExecutor o java.util.concurrent.ScheduledThreadPoolExecutor le dará opciones adicionales.
