Una Estrategia para Definir Objetos Inmutables

Las siguientes reglas definen una estrategia simple para crear objetos inmutables. No todas las clases documentadas como "inmutables" siguen estas reglas. Esto no significa necesariamente que los creadores de estas clases fueran poco rigurosos — pueden tener una buena razón para creer que las instancias de sus clases nunca cambian después de la construcción. Sin embargo, tales estrategias requieren análisis sofisticados y no son para principiantes.

  1. No establezca métodos "modificadores" — métodos que modifican los campos u objetos referidos por esos campos.
  2. Haga todos los campos final y private.
  3. No permita a las subclases sobreescribir métodos. La forma más simple de hacer esto es declarar la clase como final. Un enfoque más sofisticado es hacer el constructor private y construir instancias en métodos de factoría.
  4. Si los campos de la instancia incluyen referencias a objetos mutables, no permita que esos objetos sean cambiados:
    • No ofrezca métodos que modifiquen los objetos mutables.
    • No comparta las referencias a los objetos mutables. Nunca almacene referencias a objetos externos, mutables pasados al constructor; si es necesario, cree copias, y almacene referencias a las copias. Similarmente, cree copias de sus objetos mutables internos cuando sea necesario para evitar devolver los originales en sus métodos.

Aplicar esta estrategia a RGBSincronizado resulta en los siguientes pasos:

  1. Hay dos métodos modificadores en esta clase. El primero, fija, arbitrariamente transforma el objeto, y no hay lugar para él en una versión inmutable de la clase. El segundo, invierte, puede ser adaptado tomando lo que crea como un nuevo objeto en vez de modificar el existente.
  2. Todos los campos ya son private; ellos serán cualificados como final.
  3. La clase en sí misma es declarada final.
  4. Sólo un campo se refiere a un objeto, y ese objeto es en sí mismo inmutable. En consecuencia, no son necesarias salvaguardias contra el cambio del estado de objetos mutables "contenidos".

Después de estos cambios, tenemos RGBInmutable: 


final public class RGBInmutable {

    // Los valores deben estar entre 0 y 255.
    final private int rojo;
    final private int verde;
    final private int azul;
    final private String nombre;

    private void comprueba(int rojo,
                       int verde,
                       int azul) {
        if (rojo < 0 || rojo > 255
            || verde < 0 || verde > 255
            || azul < 0 || azul > 255) {
            throw new IllegalArgumentException();
        }
    }

    public RGBInmutable(int rojo,
                        int verde,
                        int azul,
                        String nombre) {
        comprueba(rojo, verde, azul);
        this.rojo = rojo;
        this.verde = verde;
        this.azul = azul;
        this.nombre = nombre;
    }


    public int dameRGB() {
        return ((rojo << 16) | (verde << 8) | azul);
    }

    public String dameNombre() {
        return nombre;
    }

    public RGBInmutable invierte() {
        return new RGBInmutable(255 - rojo,
                       255 - verde,
                       255 - azul,
                       "Inverso de " + nombre);
    }
}
