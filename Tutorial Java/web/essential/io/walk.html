Recorriendo el Árbol de Ficheros

¿Necesita crear una aplicación que visitará recursivamente todos los ficheros en un árbol de ficheros? Quizás necesita borrar cada fichero .class en un árbol, o encontrar cada fichero que no se ha accedido en el último año. Puede hacerlo con el interfaz FileVisitor .

Esta sección cubre lo siguiente:

  • El Interfaz FileVisitor
  • Comenzar el Proceso
  • Consideraciones al Crear un FileVisitor
  • Controlar el Flujo
  • Ejemplos

El Interfaz FileVisitor

Para recorrer un árbol de ficheros, primero necesita implementar un FileVisitor. Un FileVisitor especifica el comportamiento requerido en puntos claves del proceso transversal: cuando un fichero es visitado, antes de que un directorio sea accedido, después de que un directorio sea accedido, o cuando ocurre un fallo. El interfaz tiene cuatro métoods que se corresponden con estas situaciones:

  • preVisitDirectory – Invocado antes de que las entradas de un directorio sean visitadas.
  • postVisitDirectory – Invocado después de que todas las entradas en un directorio son visitadas. Si algún error fué encontrado, la excepción específica es pasada al método.
  • visitFile – Invocado cuando el fichero está siendo visitado. Los BasicFileAttributes del fichero son pasados al método, o puede usar el paquete atributos de fichero para leer un conjunto específico de atributos. Por ejemplo, puede elegir leer la DosFileAttributeView del fichero para determinar si el fichero tiene establecido el bit "oculto".
  • visitFileFailed – Invocado cuando el fichero no puede ser accedido. La excepción específica es pasada al método. Puede elegir si lanzar la excepción, imprimirla en la consola o en un fichero de trazas, etcétera..

Si no necesita implementar todos los cuatro métodos de FileVisitor , en vez de imeplemntar el interfaz FileVisitor , puede extender la clase SimpleFileVisitor . Esta clase, la cual implementa el interfaz FileVisitor, visita todos los ficheros en un árbol y lanza una IOError cuando se encuentra un error. Puede extender ésta clase y sobreeescribir sólo los métodos que necesite.

Aquí tiene un ejemplo que exteinde SimpleFileVisitor para imprimir todas las entradasd en un árbol de ficheros. Imprime la entrda si la entrada es un fichero regular, un enlace simbólico, un directorio, o algún otros tipo "no especificado" de fichero. Tambén imprime el tamaño, en bytes, de cada fichero. Cualquier excepción que se encuentre es impresa en la consola.

Los métodos de FileVisitor están mostrados en negrita: 

import static java.nio.file.FileVisitResult.*;

public static class PrintFiles
    extends SimpleFileVisitor<Path> {

    // Imprime información sobre
    // cada tipo de fichero.
    @Override
    public FileVisitResult visitFile(Path file,
                                   BasicFileAttributes attr) {
        if (attr.isSymbolicLink()) {
            System.out.format("Enlace simbólico: %s ", file);
        } else if (attr.isRegularFile()) {
            System.out.format("Fichero regular: %s ", file);
        } else {
            System.out.format("Otros: %s ", file);
        }
        System.out.println("(" + attr.size() + "bytes)");
        return CONTINUE;
    }

    // Imprime cada directorio visitado
    @Override
    public FileVisitResult postVisitDirectory(Path dir,
                                          IOException exc) {
        System.out.format("Directorio: %s%n", dir);
        return CONTINUE;
    }

    // Si hay algun error accediendo
    // el fichero, se lo decimos al usuario.
    // Si no quiere sobreescribir éste método
    // y un error courre, se lanzará una
    // IOException.
    @Override
    public FileVisitResult visitFileFailed(Path file,
                                       IOException exc) {
        System.err.println(exc);
        return CONTINUE;
    }
}

Comenzar el Proceso

Una vez ha implementado su FileVisitor, ¿cómo inicia el recorrido de ficheros? Hay dos métodos de walkFileTree en la clase Files.

  • walkFileTree(Path, FileVisitor)
  • walkFileTree(Path, Set<FileVisitOption>, int, FileVisitor)

El primer método requiere sólo un punto de arranque y una instancia de su FileVisitor. Usted peude invocar el PrintFiles del visitador de ficheros como sigue: 

Path startingDir = ...;
PrintFiles pf = new PrintFiles();
Files.walkFileTree(startingDir, pf);

El segundo método walkFileTree le habilita a especificar adiccionalmente un límite en el número de niveles visitados y establecer las enumeraciones de FileVisitOption. Si quiere asegurarse que este método recorre el árbol de ficheros entero, puede especificar Integer.MAX_VALUE en el argumento de máxima profundidad.

Puede especificar la enumeración FileVisitOption , FOLLOW_LINKS, la cual indica que los enlaces simbólicos deben ser seguidos.

Este trozo de código muestra como el método de 4-argumentos puede ser invocado: 

import static java.nio.file.FileVisitResult.*;

Path startingDir = ...;

EnumSet<FileVisitOption> opts = EnumSet.of(FOLLOW_LINKS);

Finder finder = new Finder(pattern);
Files.walkFileTree(startingDir, opts, Integer.MAX_VALUE, finder);

Consideraciones al Crear un FileVisitor

Un árbol de ficheros es recorrido en profundidad primero, pero no puede hacer ninguna asunción sobre el orden de la iteración en el que los subdirectorios son visitados.

Si su programa va a estar cambiando el sistema de ficheros, necesita considerar cuidadosamente cómo implementar su FileVisitor.

Por ejemplo, si está escribiendo un borrado recursivo, primero borra los ficheros en un directorio antes de borrar el directorio en sí mismo. En este caso, usted borra el directorio en postVisitDirectory.

Si está escribiendo una copia recursiva, usted crea el directorio nuevo en preVisitDirectory antes de intentar copiar los ficheros en él (en visitFiles). Si usted quiere preservar los atributos del directorio origen ( similar al comando de UNIX cp -p), necesita hacer eso después de que los ficheros han sido copiados, en postVisitDirectory. El ejemplo Copia muestra cómo hacer esto.

Si está escribiendo una búsqueda de ficheros, usted realiza la comparación en el método visitFile. Este método busca todos los ficheros que coinciden con sus criterio, pero no busca los directorios. Si quiere encontrar tanto ficheros como directorios, debe también realizar la comparación tanto en el método preVisitDirectory como postVisitDirectory. El ejemplo Busca muestra cómo hacer esto.

Necesita decidir si quiere que los enlaces simbólicos sean segudos. Si está borrando ficheros, por ejemplo, el seguir los enlaces simbólicos podría no ser aconsejable. Si está copiando un árbol de ficheros, podría querer permitirlo. Por defecto, walkFileTree no sigue los enlaces simbólicos.

El método visitFile es invocado para ficheros. Si usted ha especificado la opción FOLLOW_LINKS y su árbol de ficheros tiene un enlace circular a su directorio padre, el directorio que provoca el bucle es informado en el método visitFileFailed con la FileSystemLoopException. El siguiente trozo de código muestra cómo capturar un enlace circular y es desde el ejemplo Copia : 

@Override
public FileVisitResult visitFileFailed(Path file, IOException exc) {
    if (exc instanceof FileSystemLoopException) {
        System.err.println("Ciclo detectado: " + file);
    } else {
        System.err.format("No se puede copiar: %s: %s%n", file, exc);
    }
    return CONTINUE;
}

Este caso puede ocurrir sólo cuando el programa está resolviendo los enlaces simbólicos.

Controlar el Flujo

Quizás quiere recorrer el árbol de ficheros buscando por un directorio particular, y, cuando lo encuentre, quiere que el proceso termine. Quizás quiere saltarse directorios específicos.

El método FileVisitor devuelve un valor FileVisitResult . Puede abotar el proceso de recorrido de ficheros o controlar si un directorio es visitado con los valores que devuelva en los métodos FileVisitor :

  • CONTINUE – Indica que el recorrido de ficheros continuará. Si el método preVisitDirectory devuelve CONTINUE, el directorio es visitado.
  • TERMINATE – Aborta inmediatamente el recorrido de ficheros. No son invocados más métodos de recorrido de ficheros después de que se haya devuelto este valor.
  • SKIP_SUBTREE – Cuando preVisitDirectory devuelve este valor, el directorio especificado y sus subdirectorios son saltados. Esta rama es "podada" del árbol.
  • SKIP_SIBLINGS – Cuando preVisitDirectory devuelve este valor, el directorio especificado no es visitado, postVisitDirectory no es invocado, y no más hermanos no visitados son visitados. Si se devolvió desde el método postVisitDirectory, no más hermanos son visitados. Esencialmente, nada más ocurre en el directorio especificado.

En este trozo de código, cualquier directorio llamado SCCS es saltado 

import static java.nio.file.FileVisitResult.*;

public FileVisitResult
     preVisitDirectory(Path dir,
         BasicFileAttributes attrs) {
    (if (dir.getFileName().toString().equals("SCCS")) {
         return SKIP_SUBTREE;
    }
    return CONTINUE;
}

En este trozo de código, tan pronto un fichero particular es localizado, el nombre de fichero es impreso en la salida estándar, y el recorrido de ficheros termina: 

import static java.nio.file.FileVisitResult.*;

// El fichero que estamos buscando.
Path lookingFor = ...;

public FileVisitResult
    visitFile(Path file,
        BasicFileAttributes attr) {
    if (file.getFileName().equals(lookingFor)) {
        System.out.println("Fichero localizado: " + file);
        return TERMINATE;
    }
    return CONTINUE;
}

Ejemplos

Los siguientes ejemplos demuestran el mecanismo de recorrido de ficheros:

  • Busca – Atraviesa un árbol de ficheros buscando por ficheros y directorios que coincidan con un patrón glob particular. Este ejemplo es descrito en Encontrando Ficheros.
  • Chmod – Cambia recursivamente los permisos en un árbol de ficheros (para sistemas POSIX sólo).
  • Copia – Copia recursivamente un árbol de ficheros.
  • MonitorizaDir – Demuestra el mecanismo que monitoriza un directorio por ficheros que han sido creados, borrados o modificados. Llamando a este programa con la opción -r observa un árbol entero por cambios. Para más información sobre el servicio de notificación de ficheros, vea Monitorizando un Directorio en busca de Cambios.
