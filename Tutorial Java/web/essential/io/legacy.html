Código de E/S de Ficheros Heredado

Interoperabilidad con código heredado

Anteriormente a la publicación de Java SE 7, la clase java.io.File era el mecanismo usado para la E/S de ficheros, pero tenía varios inconvenientes.

  • Muchos métodos no lanzaban excepciones cuando fallaban, así que era imposible obtener un mensaje de error útil. Por ejemplo, si el borrado de un fichero fallaba, el programa recibiría un "borrado fallido" pero no sabríamos si fue porque el fichero no existía, el usuario no tenía permisos, o había algún otro problema.
  • El método rename no funcionaba consistentemente entre plataformas.
  • No había soporte real para los enlaces simbólicos.
  • Más soporte para los metadatos era deseado, como permisos de ficheros, propietario del fichero, y otros atributos de seguridad.
  • El acceso a los metadatos del fichero era ineficiente.
  • Muchos de los métodos de File no escalaban. Requerir un listado de un gran directorio en un servidor podría resultar en un cuelgue. Los directorios grandes podían también dar problemas de recursos en la memoria, resultando en una denegación de servicio.
  • No era posible escribir código confiable que pudiera recorrer recursivamente un árbol de ficheros y responder apropiadamente si había enlaces simbólicos circulares.

Quizás tenga código heredado que usa java.io.File y le gustaría aprovechar las ventajas de la funcionalidad de java.nio.file.Path con impacto mínimo sobre su código.

La clase java.io.File ofrece el método toPath , la cual convierte una instancia del viejo estilo File en una instancia java.nio.file.Path , como sigue: 

Path input = file.toPath();

Puede aprovechar la ventaja del conjunto de ricas características de la clase Path.

Por ejemplo, asuma que tiene algún código que borraba un fichero: 

file.delete();

Podría modificar este código para usar el método Files.delete, como sigue: 

Path fp = file.toPath();
Files.delete(fp);

Por el contrario, el método Path.toFile construye un objeto java.io.File para un objeto Path.

Asignación de la funcionalidad java.io.File en java.nio.file

Ya que la implementación de E/S de ficheros de Java ha sido completamente re-estructurada en la publicación de Java SE 7, no puede intercambiar un método por otro método. Si quiere usar la rica funcionalidad ofrecida por el paquete java.nio.file , su solución más fácil es usar el método File.toPath como se sugirió en la sección previa. Sin embargo, si no quiere usar ese enfoque o no es suficiente para sus necesidades, debe reescribir su código de E/S de ficheros.

No hay una correspondencia uno-a-uno entre las dos IPA, pero la siguiente tabla le da una idea general de qué funcionalidad en la IPA java.io.File se asigna en la IPA de java.nio.file y le indica dónde puede obtener más información.
