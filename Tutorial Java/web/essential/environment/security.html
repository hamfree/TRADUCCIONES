El Gestor de Seguridad

Un gestor de seguridad es un objeto que define una política de seguridad para una aplicación. Esta política especifica acciones que son no seguras o comprometidas. Cualquier acción no permitida por la política de seguridad causa que una SecurityException sea lanzada. Una aplicación puede también preguntar a su gestor de seguridad para descubrir qué acciones están permitidas.

Típicamente, un applet de web se ejecuta dentro de un gestor de seguridad ofrecido por el navegador o el plugin de Java Web Start. Otras clases de aplicaciones normalmente se ejecutan sin un gestor de seguridad, a menos que la aplicación por sí misma defina uno. Si no está presente el gestor de seguridad, la aplicación no tiene políticas de seguridad y actúa sin restricciones.

Esta sección explica cómo un aplicación interactúa con un gestor de seguridad existente. Para obtener información más detallada, incluyendo información sobre cómo diseñar un gestor de seguridad, refíerase a la Guía de Seguridad.

Interactuar con el Gestor de Seguridad

El gestor de seguridad es un objeto de tipo SecurityManager; para obtener una referencia , invoque System.getSecurityManager. 

SecurityManager appsm = System.getSecurityManager();

Si no hay gestor de seguridad, este método devuelve null.

Una vez una aplicación tiene una referencia al objeto del gestor de seguridad, puede solicitar permiso para hacer cosas específicas. Muchas clases en las librerías estándar hacen esto. Por ejemplo, System.exit, el cual finaliza la máquina virtual de Java con un estado de salida, invoca SecurityManager.checkExit para asegurar que el hilo actual tiene permiso para apagar la aplicación.

La clase SecurityManager define muchos otros métodos usados para verificar otras clases de operaciones. Por ejemplo, SecurityManager.checkAccess verifica los accesos a hilos, y SecurityManager.checkPropertyAccess verifica el acceso a una propiedad específica. Cada operación o grupo de operaciones tiene su propio método checkXXX().

Además, el conjunto de métodos checkXXX() representan el conjunto de operaciones que ya están sujetas a la protección del gestor de seguridad. Típicamente, una aplicación no tiene que invocar directamente ninguno de los métodos checkXXX().

Reconocimiento de una Infracción de Seguridad

Muchas acciones que son rutinarias sin un gestor de seguridad pueden lanzar una SecurityException cuando se ejecutan con un gestor de seguridad. Esto es cierto incluso cuando se invoca un método que no está documentado como lanzar una SecurityException. Por ejemplo, considere el siguiente código usado para leer un fichero: 

reader = new FileReader("xanadu.txt");

En ausencia de un gestor de seguridad, esta sentencia se ejecuta sin error, siempre que xanadu.txt exista y sea legible. Pero suponga que esta sentencia está insertada en un applet de web, el cual típicamente se ejecuta bajo un gestor de seguridad que no permite la entrada desde ficheros. Los siguientes mensajes de error podrían resultar: 

appletviewer fileApplet.html
Exception in thread "AWT-EventQueue-1" java.security.AccessControlException: access denied (java.io.FilePermission characteroutput.txt write)
        at java.security.AccessControlContext.checkPermission(AccessControlContext.java:323)
        at java.security.AccessController.checkPermission(AccessController.java:546)
        at java.lang.SecurityManager.checkPermission(SecurityManager.java:532)
        at java.lang.SecurityManager.checkWrite(SecurityManager.java:962)
        at java.io.FileOutputStream.<init>(FileOutputStream.java:169)
        at java.io.FileOutputStream.<init>(FileOutputStream.java:70)
        at java.io.FileWriter.<init>(FileWriter.java:46)
...

Note que la excepción específica lanzada en este caso, java.security.AccessControlException, es una subclase de SecurityException.
