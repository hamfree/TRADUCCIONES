Excepciones No Comprobadas — La Controversia

Debido a que el lenguaje de programación Jva no requiere métodos para capturar o para especificar excepciones no controladas (RuntimeException, Error, y sus subclases), los programaores pueden estar tentados a escribir código que lance sólo excepciones no comprobadas o hacer que todas sus subclases excepciones hereden de RuntimeException. Ambos atajos permiten a los programadores escribir código sin preocuparse con los errores de compilación y sin preocuparse si especificar o capturar cualquier excepción. Aunque esto podría parecer conveniente al programador, evitar la intención del requerimiento de capturar o especificar puede causar problemas a otros que usen sus clases.

¿Porqué los diseñadores decidieron forzar a un método a especificar todas las excepciones comprobadas no capturadas que pueden ser lanzadas en su alcance? Cualquier Exception que puede ser lanzada por un método es parte de la interfaz pública de programación del método. Aquellos que llaman un método deben saber sobre las excepciones que un método puede lanzar de forma que puedan decidir que hacer con ellas. Estas excepciones forman parte del interfaz de programación del método al igual que sus parámetros y el valor de return.

La siguiente cuestión podría ser: "Si es tan bueno documentar la IPA de un método, incluyendo las excepciones que puede lanzar, ¿porqué no especificar las excepciones en tiempo de ejecución también?" Las excepciones en tiempo de ejecución representan problemas que son el resultado de un problema de programación, y como tal, el código cliente del IPA no puede razonablemente esperar el recuperarse de ellas o manejarlas en cualquier forma. Tales problemas incluyen excepciones aritméticas, tales como dividir por cero; las excepciones de punteros, tales como intentar acceder a un objeto a través de una referencia nula; y las excepciones de indexado, tales como intentar acceder a un elemento de una matriz a través de un índice que es demasiado grande o demasiado pequeño.

Las excepciones en tiempo de ejecución pueden ocurrir en cualquier parte en un programa, y en uno típico pueden ser muy numerosas. Tener que añadir las excepciones en tiempo de ejecución en cada declaración de método reduciría la claridad de un programa. Así, el compilador no requiere que capture o especifique las excepciones en tiempo de ejecución (aunque usted puede).

Un caso donde es una práctica común lanzar una RuntimeException es cuando el usuario llama a un método incorrectamente. Por ejemplo, un método puede comprobar si uno de sus argumentos es incorrectamente null. Si un argumento es null, el método podría lanzar una NullPointerException, la cuales una excepción no comprobada.

Generalmente hablando, no lance una RuntimeException o cree una subclase de RuntimeException simplemente porque no quiere preocuparse con especificar las excepciones que sus métodos puedan lanzar.

Aquí está la última directriz: si un cliente puede razonablemente esperarse recuperar de una excepción, hágala un excepción comprobada. Si un cliente no puede hacer nada para recuperarse de la excepción, hágala una excepción no comprobada.
