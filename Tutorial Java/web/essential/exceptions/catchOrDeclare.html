La Captura o la Especificación del Requisito

El código válido del lenguaje de programación Java debe cumplir la Captura o Especificación del requisito. Esto significa que el código que puede lanzar ciertas excepciones debe estar encerrado por cualquiera de los siguientes:

  • Una sentencia try que captura la excepción. El try debe establecer un manejador para la excepción, como está descrito en Capturando y Manejando Excepciones.
  • Un método que especifica que puede lanzar la excepción. El método debe establecer una cláusula throws que lista la excepción, como está descrito en Especificando las Excepciones Lanzadas por un Método .

El código que no cumple la Captura o Especificación del Requisito no compilará.

No todas las excepciones están sujetas a la Captura o Especificación del requisito. Para entender porqué, necesitamos ver las tres categorías básicas de excepciones, sólo una de las cuales está sujeta al Requerimiento.

Las Tres Clases de Excepciones

La primera clase de excepción es la excepción comprobada. Estas son condiciones excepcionales que una aplicación bien escrita anticipará y se recuperará de ellas. Por ejemplo, suponga que una aplicación le pregunta a un usuario por el nombre de un fichero de entrada, entonces abre el fichero pasándole el nombre al constructor de java.io.FileReader. Normalmente, el usuario indica el nombre de un fichero existente y legible, así que la construcción del objeto FileReader tiene éxito, y la ejecución de la aplicación procede normalmente. Pero algunas veces el usuario proporciona el nombre de un fichero no existente, y el constructor lanza java.io.FileNotFoundException. Un programa bien escrito capturará esta excepción y notifica al usuario del error, posiblemente preguntando por un nombre de fichero correcto.

Las excepciones comprobadas están sujetas a la Captura o Especificación del Requisito. Todas las excepciones son excepciones comprobadas, excepto para aquellas indicadas por Error, RuntimeException, y sus subclases.

La segunda clase de excepciones es el error. Estas son condiciones excepcionales que son externas a la aplicación, y que la aplicación usualmente no puede anticipar o recuperarse de ellas. Por ejemplo, suponga que un aplicación abre satisfactoriamente un fichero para la entrada, pero es incapaz de leer el fichero porque el hardware o el sistema no funciona bien. La lectura no exitosa lanzará java.io.IOError. Una aplicación podría elegir capturar esta excepción, con el fin de notificar al usuario el problema — pero también podría tener sentido para el programa imprimir una traza de la pila y terminar.

Los errores no están sujetos a la Captura o Especificación del Requisito. Los errores son esas excepciones indicadas por Error y sus subclases.

La tercera clase de excepción en la excepción en tiempo de ejecución. Estas son condiciones excepcionales que son internas a la aplicación, y que la aplicación usualmente no puede anticipar o recuperarse de ellas. Estas usualmente indican errores del programa, tales como errores lógicos o uso inapropiado de un IPA. Por ejemplo, considere la aplicación descrita anteriormente que pasa un nombre de fichero al constructor de FileReader. Si un error lógico causa que se pase un null al constructor, el constructor lanzará una NullPointerException. La aplicación puede capturar esta excepción, pero probablemente tiene más sentido eliminar el error que causó que la excepción ocurriera.

Las excepciones en tiempo de ejecución no están sujetas a la Captura o Especificación del Requisito. Las excepciones en tiempo de ejecución son aquellas indicadas por RuntimeException y sus subclases.

Las excepciones en tiempo de ejecución y los errores son colectivamente conocidas como excepciones no comprobadas.

Anulación de Captura o Especificación

Algunos programadores consideran la Captura o Especificación del Requisito una falla seria en el mecanismo de excepción y lo anulan usando excepciones no comprobadas en lugar de las excepciones comprobadas. En general, esto no se recomienda. La sección Excepciones No Comprobadas — La Controversia habla sobre cuándo es apropiado usar excepciones no comprobadas.
