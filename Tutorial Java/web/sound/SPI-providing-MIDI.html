Proporcionando Servicios MIDI

Introducción a los Interfaces de los Proveedores de Servicio explicó que los paquetes javax.sound.sampled.spi y javax.sound.midi.spi definen clases abstractas para ser utilizadas por desarrolladores de servicios de sonido. Al implementar una subclase de una de estas clases abstractas, un proveedor de servicios puede crear un nuevo servicio que amplíe la funcionalidad del sistema en tiempo de ejecución. La sección anterior cubrió el uso del paquete javax.sound.sampled.spi. Esta sección explica cómo usar el paquete javax.sound.midi.spi para proporcionar nuevos servicios para manejar dispositivos y archivos MIDI.

Hay cuatro clases abstractas en el paquete javax.sound.midi.spi, que representan cuatro tipos diferentes de servicios que puede proporcionar al sistema MIDI:

  • MidiFileWriter proporciona servicios de escritura de archivos MIDI. Estos servicios hacen posible que un programa de aplicación guarde, en un archivo MIDI, una Sequence MIDI que ha generado o procesado.
  • MidiFileReader proporciona servicios de lectura de archivos que devuelven una Sequence MIDI desde un archivo MIDI para su uso en un programa de aplicación.
  • MidiDeviceProvider proporciona instancias de uno o más tipos específicos de dispositivo MIDI, posiblemente incluidos los dispositivos de hardware.
  • SoundbankReader suministra servicios de lectura de archivos de banco de sonido. Subclases concretas de SoundbankReader analizan un archivo de banco de sonidos dado, produciendo un objeto Soundbank que se puede cargar en un Synthesizer.

Un programa de aplicación no creará directamente una instancia de un objeto de servicio, ya sea un objeto de proveedor, como un MidiDeviceProvider, o un objeto, como un Synthesizer, que se proporciona por el objeto proveedor. El programa tampoco se referirá directamente a las clases de SPI. En cambio, el programa de aplicación realiza solicitudes al objeto MidiSystem en el paquete javax.sound.midi, y MidiSystem a su vez usa subclases concretas de las clases de javax.sound.midi.spi para procesar estas solicitudes.

Proporcionar Servicios de Escritura de Ficheros MIDI

Hay tres formatos de archivo MIDI estándar, todos los cuales puede admitir una implementación de la IPA de Java de Sonido : Tipo 0, Tipo 1 y Tipo 2. Estos formatos de archivo difieren en su representación interna de los datos de secuencia MIDI en el archivo, y son apropiado para diferentes tipos de secuencias. Si una implementación no es compatible con los tres tipos, un proveedor de servicios puede proporcionar el soporte para los no implementados. También existen variantes de los formatos de archivo MIDI estándar, algunos de ellos propietarios, que de manera similar podrían ser compatibles con un proveedor externo.

La capacidad de escribir archivos MIDI la proporcionan subclases concretas de MidiFileWriter. Esta clase abstracta es directamente análoga a javax.sampled.spi.AudioFileWriter. Nuevamente, los métodos se agrupan en métodos de consulta para aprender qué tipos de archivos se pueden escribir y métodos para escribir realmente un archivo. Al igual que con AudioFileWriter, dos de los métodos de consulta son concretos: 

boolean isFileTypeSupported(int fileType)
boolean isFileTypeSupported(int fileType, Sequence sequence)

El primero de ellos proporciona información general sobre si el escritor de archivos puede escribir el tipo especificado de tipo de archivo MIDI. El segundo método es más específico: pregunta si una secuencia particular se puede escribir en el tipo especificado de archivo MIDI. Generalmente, no necesita anular ninguno de estos dos métodos concretos. En la implementación predeterminada, cada uno invoca uno de los otros dos métodos de consulta correspondientes e itera sobre los resultados devueltos. Al ser abstractos, estos otros dos métodos de consulta deben implementarse en la subclase: 

abstract int[] getMidiFileTypes() 
abstract int[] getMidiFileTypes(Sequence sequence)

El primero de estos devuelve una matriz de todos los tipos de archivos que se admiten en general. Una implementación típica podría inicializar la matriz en el constructor del escritor de archivos y devolver la matriz de este método. A partir de ese conjunto de tipos de archivos, el segundo método encuentra el subconjunto en el que el escritor de archivos puede escribir la secuencia dada. De acuerdo con la especificación MIDI, no todos los tipos de secuencias se pueden escribir en todos los tipos de archivos MIDI.

Los métodos write de una subclase MidiFileWriter realizan la codificación de los datos en una Sequence dada en el formato de datos correcto para el tipo de archivo MIDI solicitado, escribiendo la secuencia codificada en un archivo o una secuencia de salida: 

abstract int write(Sequence in, int fileType, 
                   java.io.File out) 
abstract int write(Sequence in, int fileType, 
                   java.io.OutputStream out)

Para hacer esto, el método write debe analizar la Sequence iterando sobre sus pistas, construir un encabezado de archivo apropiado y escribir el encabezado y las pistas en la salida. El formato de encabezado del archivo MIDI, por supuesto, está definido por la especificación MIDI. Incluye información como un "número mágico" que lo identifica como un archivo MIDI, la longitud del encabezado, el número de pistas y la información de tiempo de la secuencia (tipo de división y resolución). El resto del archivo MIDI consta de los datos de la pista, en el formato definido por la especificación MIDI.

Veamos brevemente cómo el programa de aplicación, el sistema MIDI y el proveedor de servicios cooperan para escribir un archivo MIDI. En una situación típica, un programa de aplicación tiene una Sequence MIDI particular para guardar en un archivo. El programa consulta el objeto MidiSystem para ver qué formatos de archivo MIDI, si los hay, son compatibles con la Sequence en particular, antes de intentar escribir el archivo. El método MidiSystem.getMidiFileTypes(Sequence) devuelve una matriz de todos los tipos de archivos MIDI en los que el sistema puede escribir una secuencia en particular. Para ello, invoca el método getMidiFileTypes correspondiente para cada uno de los servicios MidiFileWriter instalados, y recopila y devuelve los resultados en una matriz de enteros que se pueden considerar como una lista maestra de todos los tipos de archivos compatibles con la Sequence dada. Cuando se trata de escribir la Sequence en un archivo, a la llamada a MidiSystem.write se le pasa un número entero que representa un tipo de archivo, junto con la Sequence por escribir y el archivo de salida; MidiSystem utiliza el tipo proporcionado para decidir qué MidiFileWriter instalado debe manejar la solicitud de escritura, y envía un write correspondiente al MidiFileWriter.

Proporcionar Servicios de Lectura de Ficheros MIDI

La clase abstracta MidiFileReader es directamente análoga a la clase javax.sampled.spi.AudioFileReader. Ambos consisten en dos métodos sobrecargados, cada uno de los cuales puede tomar un argumento File, URL o InputStream. El primero de los métodos sobrecargados devuelve el formato de archivo de un archivo especificado. En el caso de MidiFileReader, el IPA es: 

abstract MidiFileFormat getMidiFileFormat(java.io.File file) 
abstract MidiFileFormat getMidiFileFormat(
    java.io.InputStream stream) 
abstract MidiFileFormat getMidiFileFormat(java.net.URL url)

Las subclases concretas deben implementar estos métodos para devolver un objeto MidiFileFormat completo que describa el formato del archivo MIDI especificado (o flujo o URL), asumiendo que el archivo es de un tipo admitido por el lector de archivos y que contiene información de encabezado válida. De lo contrario, debería lanzarse una InvalidMidiDataException.

El otro método sobrecargado devuelve una Sequence MIDI de un archivo, flujo o URL determinados: 

abstract Sequence getSequence(java.io.File file) 
abstract Sequence getSequence(java.io.InputStream stream) 
abstract Sequence getSequence(java.net.URL url)

El método getSequence realiza el trabajo real de analizar los bytes en el archivo de entrada MIDI y construir un objeto Sequence correspondiente. Este es esencialmente el proceso inverso al utilizado por MidiFileWriter.write. Debido a que existe una correspondencia uno a uno entre el contenido de un archivo MIDI según lo definido por la especificación MIDI y un objeto Sequence definido por la IPA de Sonido de Java, los detalles del análisis son sencillos. Si el archivo pasado a getSequence contiene datos que el lector de archivos no puede analizar (por ejemplo, porque el archivo está dañado o no cumple con la especificación MIDI), una InvalidMidiDataException debería lanzarse.

Proporcionar Dispositivos MIDI Particulares

Un MidiDeviceProvider puede considerarse una fábrica que suministra uno o más tipos particulares de dispositivos MIDI. La clase consta de un método que devuelve una instancia de un dispositivo MIDI, así como métodos de consulta para saber qué tipos de dispositivos puede suministrar este proveedor.

Al igual que con los otros servicios de javax.sound.midi.spi, los desarrolladores de aplicaciones obtienen acceso indirecto a un servicio MidiDeviceProvider a través de una llamada a los métodos de MidiSystem, en este caso MidiSystem.getMidiDevice y MidiSystem.getMidiDeviceInfo. El propósito de subclasificar MidiDeviceProvider es proporcionar un nuevo tipo de dispositivo, por lo que el desarrollador del servicio también debe crear una clase de acompañamiento para el dispositivo que se devuelve, tal como vimos con MixerProvider en el paquete javax.sound.sampled.spi. Allí, la clase del dispositivo devuelto implementó la interfaz javax.sound.sampled.Mixer; aquí implementa la interfaz javax.sound.midi.MidiDevice. También podría implementar una subinterfaz de MidiDevice, como Synthesizer o Sequencer.

Debido a que una sola subclase de MidiDeviceProvider puede proporcionar más de un tipo de MidiDevice, el método getDeviceInfo de la clase devuelve una matriz de objetos MidiDevice.Info que enumeran los diferentes MidiDevices disponibles: 

abstract MidiDevice.Info[] getDeviceInfo()

La matriz devuelta puede contener un solo elemento, por supuesto. Una implementación típica del proveedor podría inicializar una matriz en su constructor y devolverla aquí. Esto permite que MidiSystem itere sobre todos los MidiDeviceProviders instalados para construir una lista de todos los dispositivos instalados. MidiSystem puede devolver esta lista (matriz MidiDevice.Info[]) a un programa de aplicación.

MidiDeviceProvider también incluye un método de consulta concreto: 

boolean isDeviceSupported(MidiDevice.Info info) 
                a

Este método permite al sistema consultar al proveedor sobre un tipo específico de dispositivo. Generalmente, no es necesario anular este método de conveniencia. La implementación predeterminada itera sobre la matriz devuelta por getDeviceInfo y compara el argumento con cada elemento.

El tercer y último método MidiDeviceProvider devuelve el dispositivo solicitado: 

abstract MidiDevice getDevice(MidiDevice.Info info)

Este método primero debe probar el argumento para asegurarse de que describe un dispositivo que este proveedor puede suministrar. Si no es así, debería lanzar una IllegalArgumentException. De lo contrario, devuelve el dispositivo.

Proporcionar Servicios de Lectura de Ficheros de Bancos de Sonido

Un SoundBank es un conjunto de Instruments que se pueden cargar en un Synthesizer. Un Instrument es una implementación de un algoritmo de síntesis de sonido que produce un tipo particular de sonido e incluye cadenas de nombre e información que lo acompañan. Un SoundBank corresponde aproximadamente a un banco en la especificación MIDI, pero es una colección más extensa y direccionable; Quizás sea mejor pensar en él como una colección de bancos MIDI.

SoundbankReader consta de un único método sobrecargado, que el sistema invoca para leer un objeto Soundbank de un archivo de banco de sonidos: 

abstract Soundbank getSoundbank(java.io.File file) 
abstract Soundbank getSoundbank(java.io.InputStream stream) 
abstract Soundbank getSoundbank(java.net.URL url)

Las subclases concretas de SoundbankReader funcionarán en conjunto con implementaciones particulares definidas por el proveedor de SoundBank, Instrument y Synthesizer para permitir que el sistema cargue un SoundBank desde un archivo en una instancia de una clase particular de Synthesizer. Las técnicas de síntesis pueden diferir enormemente de un Synthesizer a otro y, como consecuencia, los datos almacenados en un Instrument o SoundBank proporcionan control o especificación. Los datos para el proceso de síntesis de un Synthesizer pueden tomar una variedad de formas. Una técnica de síntesis puede requerir sólo unos pocos bytes de datos de parámetros; otro puede basarse en amplias muestras de sonido. Los recursos presentes en un SoundBank dependerán de la naturaleza del Synthesizer en el que se cargan y, por lo tanto, de la implementación del método getSoundbank de una subclase de SoundbankReader tiene acceso al conocimiento de un tipo particular de SoundBank. Además, una subclase particular de SoundbankReader comprende un formato de archivo particular para almacenar los datos de SoundBank. Ese formato de archivo puede ser propietario y específico del proveedor.

SoundBank es solo una interfaz, con solo restricciones débiles en el contenido de un objeto SoundBank. Los métodos que un objeto debe admitir para implementar esta interfaz (getResources, getInstruments, getVendor, getName, etc.) impone requisitos laxos sobre los datos que contiene el objeto. Por ejemplo, getResources y getInstruments pueden devolver matrices vacías. El contenido real de un objeto SoundBank subclasificado, en particular sus instrumentos y sus recursos que no son instrumentos, son definidos por el proveedor de servicios. Por lo tanto, el mecanismo de análisis de un archivo de banco de sonidos depende completamente de la especificación de ese tipo particular de archivo de banco de sonidos.

Los archivos de banco de sonidos se crean fuera de la IPA de Sonido de Java, generalmente por el proveedor del sintetizador que puede cargar ese tipo de banco de sonidos. Algunos proveedores pueden proporcionar herramientas para el usuario final para crear dichos archivos.
