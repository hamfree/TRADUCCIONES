Descripción del Paquete Sampled

El paquete javax.sound.sampled se ocupa fundamentalmente del transporte de audio — en otras palabras, la IPA de Sonido de Java se centra en la reproducción y captura. La tarea central que la IPA de Sonido de Java aborda es cómo mover bytes de datos de audio formateados dentro y fuera del sistema. Esta tarea implica abrir dispositivos de entrada y salida de audio y administrar búferes que se llenan con datos de sonido en tiempo real. Puede también implicar la mezcla de varios flujos de audio en un solo flujo (ya sea para entrada o salida). El transporte de sonido dentro o fuera del sistema tiene que ser correctamente manejado cuando el usuario solicita que el flujo de sonido sea iniciado, pausado, terminado, o parado.

Para soportar este enfoque en la entrada y salida básica de audio, la IPA de Sonido de Java proporciona métodos para convertir entre varios formatos de datos de audio, y para leer y escribir en tipos comunes de archivos de sonido. Sin embargo, no pretende ser un kit de herramientas completo para archivos de sonido. Una implementación particular del IPA de Sonido de Java no necesita admitir un conjunto extensivo de tipos de archivo o conversiones de formato de datos. Los proveedores de servicio de terceras partes pueden proporcionar módulos que "se conectan" a un implementación existente para admitir tipos de archivo y conversiones adicionales.

La IPA de Sonido de Java puede manejar el transporte de audio tanto en forma de transmisión, en búfer y en memoria, sin búfer. "Transmisión" se usa aquí en el sentido general de referirse al manejo en tiempo real de bytes de audio; no se refierea al específico y bien conocido caso de enviar audio a través de Internet en cierto formato. En otras palabras, un flujo de audio es simplemente un conjunto continuo de bytes de audio que llegan más o menos con la misma velocidad y que hay que manejar (reproducir, grabar, etc.). Las operaciones sobre los bytes comienzan antes de que lleguen todos los datos. En el modo de transmisión, particularmente en el caso de la entrada de audio más que en la salida de audio, no necesariamente sabe de antemano cuanto dura el sonido y cuando terminará de llegar. Simplemente maneja un búfer de datos de audio a la vez, hasta que se pare la operación. En el caso de la salida de audio (reproducir), también necesita almacenar en búfer los datos si el sonido que quiere reproducir es demasiado grande para caber en la memoria de una vez. En otras palabras, envía sus bytes de audio al motor de sonido en fragmentos, y se cuida de reproducir cada muestreo en el tiempo correcto. Se proporcionan mecanismos que hacen fácil saber cuantos datos enviar en cada fragmento.

La IPA de Sonido de Java también permite el transporte sin búfer en el caso solo de la reproducción, asumiendo que ya tiene todos los datos de audio a mano y no son demasiado grandes para caber en memoria. En esta situación, no hay necesidad para el programa de aplicación el almacenar en búfer el audio , aunque el enfoque en tiempo real y con búfer esté aún disponible si se desea. En su lugar, el sonido entero puede ser precargado de una vez en memoria para su reproducción posterior. Ya que todos los datos de sonido se cargan por adelantado, la reproducción puede comenzar inmediatamente — por ejemplo, tan pronto cmo el usuario pulsa un botón Iniciar. Esto puede ser una ventaja comparado al modelo de búfer, donde la reproducción tiene que esperar a que se llene el primer búfer. Además, el modelo en memoria, sin búfer permite que los sonidos sean facilmente enbuclados (ciclados) o se establezcan en posiciones arbitrarias en los datos.

Para reproducir o capturar sonido usando la IPA de Sonido de Java, necesita al menos tres cosas: datos formateados de audio, un mezclador, y una línea. Lo siguiente proporciona una introducción de estos conceptos.

¿Qué son los datos de audio formateados?

Los datos de audio formateados se refieren a sonido en cualquier variedad de formatos estándar. La IPA de Sonido de Java distingue entre formatos de datos y formatos de archivo.

Formatos de Datos

Un formato de datos le indica cómo interpretar unas series de bytes de datos de audio muestreado "sin procesar", tales como los muestreos que han sido leído de un archivo de sonido, o muestreos que han sido capturados desde la entrada de micrófono. Podría necesitar saber, por ejemplo, cuántos bits constituyen un muestreo (la representación del instante más corto del sonido), y de forma similar podría necesitar conocer la frecuencia de muestreo del sonido (cómo de rápido se supone que los muestreos se siguen unos a otros). Cuando configura para reproducir o capturar, especifica el formato de datos del sonido que está capturando o reproduciendo.

En la IPA de Sonido de Java, un formato de datos se representa por un objeto AudioFormat object, que incluye los atributos siguientes:

  • Técnica de codificación, habitualmente, modulación de código de pulso (PCM)
  • Número de canales (1 para mono, 2 para estéreo, etc.)
  • Frecuencia de muestreo (número de muestreos por segundo, por canal)
  • Número de bits por muestreo (por canal)
  • Cuadros por segundo
  • Tamaño del cuador en bytes
  • Orden de los bytes(big-endian o little-endian)

PCM es una clase de codificación de la forma de onda del sonido. La IPA de Sonido de Java incluye dos codificaciones PCM encodings que usan la cuantificación lineal de amplitud y valores enteros con signo y sin signo. La cuantificación lineal significa que el número almacenado en cada muestreo es directamente proporcional (excepto por cualquier distorsión) a la presión sonora original en ese instante - e igualmente proporcional al desplazamiento de un altavoz o tímpano que está vibrando con el sonido en ese instante. Los discos compactos, por ejemplo, usan sonido codificado con PCM lineal. La codificación Mu-law y la codificación a-law son codificaciones comunes no lineales que proporcionan una versión más comprimida de los datos de audio; estas codificaciones se usan habitualmente para la telefonía o grabaciones de voz. Una codificación no lineal asoica la amplitud del sonido original al valor almacenado usando una función no lineal, que puede designarse para dar más resolución de amplitud a los sonidos silenciosos que a los sonidos fuertes.

Un cuadro contiene los datos de todos los canales en un tiempo particular. Para los datos codificados en PCM, el marco es simplemente un conjunto de muestreos simultaneos en todos los canales, para un instante dado en el tiempo, sin ninguna información adicional. En este caso, la frecuencia de cuadro es igual a la frecuencia de muestreo, y el tamaño del cuadro en bytes es el número de canales multiplicado por el tamano del muestreo en bits, dividido por el número de bits en un byte.

Para otras clases de codificaciones, un cuadro puede contener información adicional a los lados de los muestreos, y la velocidad de cuadro puede ser completamente diferente de la frecuencia de muestreo. Por ejemplo, considere la codificación MP3 (Capa de Audio MPEG-1 3), que no se menciona explícitamente en la versión actual de la IPA de Sonido de Java, pero que podría ser admitida por una implementación de la IPA de Sonido de Java o por un proveedor de servicios de terceras partes. En MP3, cada cuadro contiene un conjunto de datos comprimidos para una serie de muestreos, no simplemente un muestreo por canal. Ya que cada cuadro encapsula una serie completa de muestreos, la frecuencia de cuadro es más baja que la frecuencia de muestreo. El cuadro también contiene una cabecera. A pesar del encabezado, el tamaño del cuadro en bytes es menor que el tamaño en bytes del número equivalente de cuadros PCM. (Después de tod, el propósito de MP3 es ser más compacto que los datos PCM). Para dicha codificación, la frecuencia de muestreo y el tamaño del muestreo se refieren a los datos PCM en los que el sonido codificado será convertido eventualmente antes de ser enviados al convertidor digital a análogico (DAC).

Formatos de Fichero

Un formato de archivo especifica la estructura de un archivo de sonido, incluyendo no solo el formato del sonido sin procesar en el archivo, sino también otra información que puede ser almacenada en el archivo. Los archivos de sonido vienen en diversas variedades estándar, como un WAVE (también conocidos cmo WAV, y a menudo asociados con los PCs), AIFF (a menudo asociados con Macintoshes), y AU (a menudo asociados con sistemas Unix). Los diferentes tipos de archivo de sonido tienen diferentes estructuras. Por ejemplo, podría haber un arreglo diferente de los datos en la "cabecera" de los archivos. Una cabecera contiene información descriptiva que precede habitualmente a los muestreos de audio reales, aunque algunos formatos de archivo permiten "fragmentos" sucesivos de datos descriptivos y de audio. La cabecera incluye una especificación del formato de datos que se usó para almacenar el audio en el archivo de sonido. Cualquier de estos tipos de archivo de sonido contienes varios formatos de datos (aunque habitualmente hay solo un formato de datos dentro de un archivo dado), y el mismo formato de datos puese ser usado en archivos que tienen distintos formatos de archivo.

En la IPA de Sonido de Java, un formato de archivo se representa por un objeto AudioFileFormat, que contiene:

  • El tipo de archivo (WAVE, AIFF, etc.)
  • La longitud del archivo en bytes
  • La longitud, en cuadros/tramas, de los datos de audio contenidos en el archivo
  • Un objeto AudioFormat que especifica el formato de datos de los datos de audio contenidos en el archivo

La clase AudioSystem proporciona métodos para leer y escribir sonidos en diferentes formatos de archivo, y para convertir entre diferentes formatos de datos. Algunos de los métodos le permiten acceder a los contenidos del archivo a través de una clase de flujo llamado AudioInputStream . Un AudioInputStream es una subclase de la clase InputStream, que encapsula una serie de bytes que pueden ser leídos secuencialmente. Para su superclase, la clase AudioInputStream agrega conocimiento del formato de datos de los bytes de audio (representados por un objeto AudioFormat). Al leer un archivo de sonido como un AudioInputStream, obtiene acceso inmediato a los muestreos, sin tener que preocuparse sobre la estructura del archivo de sonido (su cabecera, fragmentos, etc.). Una única invocación de métood le da todas la información que necesita sobre el formato de datos y el tipo de archivo.

¿Qué es un Mezclador?

Muchas interfaces de programación de aplicaciones (IPA) para saondi hacen uso de la noción de un dispositivo de audio. Un dispositivo es a menudo un software o interfaz a un dispositivo físico de entrada/salida. Por ejemplo, un dispositivo de entrada de audio podría representar las capacidades de entrada de una tarjea de sonido, incluyendo una entrada de micrófono, una entrada analógica de nivel de línea, y quizás una entrada de audio digital.

En la IPA de Sonido de Java, los dispositivos están representados por objetos Mixer. El propósito de un mezclador (Mixer) es manejar uno o mas flujos de sonido de audio de entrada y uno o más flujos de audio de salida. En el caso habitual, en realida mezcla juntos varios flujos de entrada en un flujo de salida. Un objeto Mixer puede representar las capacidades de mezcla de sonido de un dispositivo físicio como una tarjeta de sonido, que podría necesitar mezclar el sonido que viene a la computadora desde varias entradas, o el sonido que viene de programas de aplicación y dirigiéndolo a salidas.

Alternativamente, un objeto Mixer puede representar las capacidades de mezcla de sonido que están implementadas enteramente por software, sin ninguna interfaz inherente a dispositivos físicos.

En la IPA de Sonido de Java, un componente tal como la entrada de micrófono en una tarjeta de sonido no es en si misma considerada un dispositivo — es decir, un mezclador — sino un puerto de entrada o salida del mezclador. Un puerto típicamente proporciona un único flujo de audio de entrada o salida del mezclador (aunque el flujo puede ser multicanal, como el estéreo). El mezclador puede tener varios de tales puertos. Por ejemplo, un mezclador que representa las capacidades de salida de una tarjeta de sonido podría mezclar varias flujos de audio juntos, y después enviar la señal mezclada a cualquier de los puertos de salida conectados al mezclador. Estos puertos de salida pueden ser (por ejemplo) un conector para auriculares, un altavoz integrado o una salida de nivel de línea.

Para entender la noción de un mezclador en la IPA de Sonido de Java, ayuda el visualizar una mesa de mezclas física, tal como aquellas que se usan en conciertos en vivo y estudios de grabación .

Una Mesa de Mezclas Física

Una mezcladora física tiene "tiras" (también llamadas "tajadas"), cada una representando un camino a través del cual un señal única de audio entra en la mezcladora para su procesado. La banda tiene perillas y otros controles mediante los cuales puede controlar el volumen y el panorama (ubicación en la imagen estéreo) de la señal en esa banda. Además, el mezclador podría tener un bus separado para efectos tales como reverberación, y este bus puede ser conectado a una unidad de reverberación interna o externa. Cada banda tiene un potenciómetro que controla cuánta de la señal de esa banda va a la mezcla reverberada. La mezcla reverberada ("húmeda") es después mezclada con las señales "secas" de las tiras. Un mezclador físico envía esta mezcla final a un bus de salida, que habitualmente va a una grabadora de cinta (o sistema de grabación basado en disco) y/o altavoces.

Imagine un concierto en vivo que está siendo grabado en estero. Los cables (o conexiones sin cable) vienen de muchos micrófonos e instrumentos eléctricos en el escenario que se conectan en las entradas de la mesa de mezclas. Cada entrada va a una tajada separada del mezclador, como se ilustra. El ingeniero de sonido decide las configuraciones de los controels de ganancia, paneo, y reverberación. La salida de todas las tajadas y la unidad de reverberación se mezclan juntas en dos canales. Estos dos canales van a dos salidas en la mezcladora, en las cuales están conectados cables que se conectan a las entradas de la grabadora de cinta. Los dos canales son quizás también enviados a través de un amplificador a altavoces en la sala, dependiendo del tipo de música y del tamaño de la sala.

Ahora imagien un estudio de grabación, en el cual cada instrumento o cantante se graba en una pista separada de una grabadora de cienta multipista. Después de que los instrumentos y cantantes han sido todos grabados, el ingeniero de grabación realiza una "mezcla" para combinar todas las pistas grabadas en una grabación de dos canales (estéreo) que puede ser distribuída en discos compactos. En este caso, la entrada a cada una de las tiras de la mezcladora no son un micrófono, sino una pista de una grabación multipista. Una vez más, el ingeniero puede usar los controles en las tiras para decidir cada cantidad de volumen, paneo y reverberación en cada pista. La salida del mezclador va una vez más a una grabadora estéreo y a altavoces estéreo, como en el ejemplo del concierto en vivo.

Estos dos ejemplos ilustran dos usos diferentes de un mezclador: para capturar varios canales de entrada, combinarlos en menos pistas, y salvar la mezcla, o reproducir varias pistas mientras las mezcla a menos pistas.

En la IPA de Sonido de Java, un mezclador puede de forma similar utlizarse para la entrada (capturando audio) o la salida (reproduciendo audio). En el caso de la entrda, la fuente de la cual el mezclador obtiene el audio para mezclar es uno o más puertos de entrada. El mezclador envía los flujos de audio capturados y mezclados a su destino, el cual es un objeto con un búfer desde el cual un programa de aplicación puede recupera los datos de audio mezclados. En el caso de la salida de audio, la situación se invierte. La fuente de audio del mezclador es uno o más objetos que contienen búferes en los que uno o más programas de aplicación escriben sus datos de sonido; y el objetivo del mezclador es uno o más puertos de salida.

¿Qué es una Línea?

La metáfora de una mesa de mezclas física es también util para entender el concepto de una línea en la IPA de Sonido de Java.

Una línea es un elemento de la "tubería" de audio digital; es decir, una ruta para mover audio dentro y fuera del sistema. Habitualmente la línea es una ruta dentro y fuera de un mezclador (aunque técnicamente el mezclador en sí mismo es también una clase de línea).

Audio input and output ports are lines. These are analogous to the microphones and speakers connected to a physical mixing console. Another kind of line is a data path through which an application program can get input audio from, or send output audio to, a mixer. These data paths are analogous to the tracks of the multitrack recorder connected to the physical mixing console.

Una diferente entre línea en la IPA de Sonido de Java y aquellas de la mesa de mezclas física es que los datos de audio fluyen a través de una línea en la IPA de Sonido de Java pueden ser mono o multicanal (por ejemplo, estéreo). Por contraste, cada una de las entradas y salidas de una mesa de mezclas física es habitualmente un único canal de sonido. Para obtener dos o más canales de salida desde la mesa de mezclas física, se usan normalmente dos o más salidas físicas (al menos en el caso de sonido analógico; un conector de salida digital es a menudo multicanal). En la IPA de Sonido de Java, el número de canales en una línea se especifica por el AudioFormat de los datos que están actualmente fluyendo a través de la línea.

Ahora examinemos algunas clases específicas de líneas y mezcladores. El diagrama siguiente muestra diferentes tipos de líneas en un sistema simple de salida de audio que podría ser parte de una implementación de la IPA de Sonido de Java:

Una Posible Configuración de Líneas de Salida de Audio

En este ejemplo, un programa de aplicación ha obtenido acceso a algunas entradas disponibles de un mezclador de entrada de audio: uno o más clips y línes de fuentes de datos. Un clip es una entrada de mezclador (una clase de línea) en la cual puede cargar datos de audio antes de la reproducción; una línea de fuente de datos es una entrada de mezcladora que acepta un flujo de datos de audio en tiempo real . El programa de aplicación precarga los datos de audio de un archivo de sondio en los clips. Luego empuja otros datos de audio a las líneas de fuentes de datos, un búfer a la vez. El mezclador lee los datos de todas estas líneas, cada una de ellas tiene sus propios controles de reverberación, ganancia y balanceo, y mezcla las señales de audio seco con la mezcla húmeda (reverberada). El mezclador envia su salida final a uno o más puertos de salida, tales como un altavoz, un conector de auriculares, y un conector de línea de salida.

Aunque las diversas líneas se representa como rectángulos separados en el diagrama, todas ellas son "propiedad" del mezclador, y pueden ser consideradas partes integrales del mezclador. Los rectángulos de reverberación, ganancia y balanceo representa los controles de procesamiento (en lugar de líneas) que el mezclador puede aplicar a los datos que fluyen a través de las líneas.

Tenga en cuenta que esto es simplemente un ejemplo de los mezcladores posibles que son admitidos por la IPA. No todas las configuraciones de audio tendrán las características ilustradoas. Una línea de fuente de datos podría no soportar el balaceo, un mezclador podría no implementar la reverberación, etc.

Aquí, los datos fluyen dentro del mezclador desde una o más puertos de entrada, comunmente el micrófono o el conector de la línea de entrada. Se aplican ganancia y balanceo, y el mezclador envía los datos capturados a un programa de aplicación a través de la línea de datos destino del mezclador. Una línea de datos destino es una salida del mezclador, que contiene la mezcla de los sonidos de entrada transmitidos. El mezclador más simple tiene simplemente un línea de datos de destino, pero algunos mezcladores pueden enviar los datos capturados a varias líneas de datos de destino simultáneamente.

La Jerarquía de la Interfaz Línea

Ahora que hemos visto algunas imágenes funcionales de qué son las líneas y los mezcladores, analicémoslas desde una perspectiva ligeramente más programática. Varios tipos de líneas se define por subinterfaces de la interfaz básica Line. La jerarquía de interfaces se muestra abajo.

La Jerarquía de la Interfaz de Line

La interfaz base, Line, describe la funcionalidad mínima común a todas las líneas:

  • Controles – Las líneas de datos y puertos a menudo tienen un conjunto de controles que afectan a la señal de audiuo que pasa a través de la línea. La IPA de Sonido de Java especifica clases de control que se pueden usar para manipular aspectos del sonido tales como: ganancia (que afecta al volumen de la señal en decibelios), balanceo (que afecta al posicionamiento derecha-izquierda del sonido), reverberación (que agrega reverberación al sonido para emular diferentes clases de acústicas de sala), y frecuencia de muestreo (que afecta a la velocidad de reproducción como también al tono del sonido ).
  • Estado abierto o cerrado – la apertura con éxito de una línea garantiza que los recursos han sido asignados a la línea. Un mezclador tiene un número finito de líneas, así que en algún punto varios programas de aplicaciones (o el mismo) pueden competir por el uso de las líneas del mezclador. Cerrar una línea indica que cualquier recurso usado por la línea puede ahora ser liberado.
  • Eventos – Una línea genera eventox cuando se abre o cierra. Las subinterfaces de Line pueden introducir otros tipos de eventos. Cuando una línea genera un evento, el evento se envía a todos los objeto que se han registrado para "monitorizar" eventos en esa línea. Un programa de aplicación puede crear estos objetos, registrarlos para monitorizar los eventos de línea, y reaccionar a los eventos como se desee.

Ahora examinaremos las subinterfaces de la interfaz Line.

Ports son líneas simples para la entrada o salida de audio a o desde dispositivos de audio. Como se mencionón anteriormente, algunos tipos comunes de puertos son el micrófono, la entrada de línea, la unidad de CD-ROM, el altavoz, los auriculars, y la salida de línea.

El interfaz Mixer representa un mezclador, por supuesto, el cual hemos visto representa un dispositivo de hardware o un software. La interfaz Mixer proporciona métodos para obtener las líneas de un mezclador, Esto incluye líneas fuente, que alimentan de audio al mezclador, y líneas destino, a las cuales el mezclador envía su audio mezclado. Para un mezclador de entrada de audio, las líneas fuente son puertos de entrada como la entrada de micrófono, y las líneas destino son TargetDataLines (descritas abajo), que envían audio al programa de aplicación. Para un mezclador de salida de audio, por el otro lado, las líneas fuente son Clips o SourceDataLines (descritas abajo), a las cuales el programa de aplicación alimenta datos de audio, y las líneas de destino son puertos de salida como el altavoz.

Un Mixer se define como teniendo una o más líneas de origen y una o más líneas de destino. Tenga en cuenta que esta definición siginifica que un mezclador no necesita en realidad mezclar datos; podría tener una sola línea de origen. La IPA de Mixer está destinada a abarcar una variedad de dispositivos, pero el caso típico admite la mezcla.

La interfaz Mixer adminte la sincronización; es decir, puede especificar que dos o más de las líneas de un mezclador sean tratadas como un grupo sincronizado. Después puede iniciar, parar, o cerrar todas esas líneas de datos mandando un único mensaje a cualquier línea en el grupo, en vez de tener que controlar cada línea individualmente. Con un mezclador que admita esta característica, puede obtener una sincronización precisa de muestras entre líneas.

La interfaz genérica de Line no proporciona una forma de iniciar o parar una reproducción o grabación. Para eso necesita una línea de datos. La interfaz DataLine proporciona las siguientes características adicionales relacionadas con la media más alla de aquellas de una Line:

  • Formato de audio – Cada línea de datos tiene un formato de audio asociado con su flujo de datos.
  • Posición de Media – Una línea de datos puede informar de su posición actual en la media, expresada en cuadros de muestreo. Esto representa el número de cuadros de muestreo capturados o reproducidos desde la línea de datos desde que fue abierta.
  • Tamaño del búfer – Este es el tamapo del búfer interno de la línea de datos en bytes. Para una línea de datos de origen, el búfer intero es uno cuyos datos pueden ser escritos, y para una línea de datos destino es uno del cual los datos pueden ser leídos.
  • Nivel (la amplitud actual de la señal de audio)
  • Iniciar y detener la reproducción o captura
  • Pausar y reanudar la reproducción o captura
  • Vaciar (descartar los datos no procesados de la cola
  • Drain (bloquear hasta que todos los datos no procesados se hayan drenado de la cola y el búfer de la línea de datos se hayan vaciado)
  • Estado activo – Una línea de datos se considera activa si está involucrada en una presentación o captura activa de datos de audio hacia o desde un mezclador.
  • Eventos – Los eventos START y STOP se producen cuando la presentación activa o la captura de datos desde o a la línea de datos se inicia o se para.

Una TargetDataLine recibe datos de audio de un mezclador. Comunmente, el mezclador ha capturado datos de audio desde un puerto tal como un micrófono; podría procesar o mezclar este audio capturado antes de colocar los datos en el búfer de la línea de datos de destino. La interfaz TargetDataLine proporciona métodos para leer los datos del búfer de línea de datos de destino y para determinar cuántos datos están disponibles actualmente para leer.

Una SourceDataLine recibe datos de audio pare reproducir. Proporciona métodos para escribir datos al búfer de la linea de datos origen para reproducir, y para determinar cuántos datos está preparada la línea para recibir sin bloquearse.

Un Clip es una línea de datos en la cual los datos de audio pueden cargarse antes de reproducirse. Ya que los datos están precargadso en vez de transmitidos, la duración del clip se conoce antes de la reproducción, y puede elegir cualquier posición de inicio en la media. Los clips se pueden reproducir en bucle, lo que significa que al reproducirlos, todos los datos entre dos puntos de bucle especificados se repetirán un número específico de veces o de forma indefinida.
