Lección: Introducción a las Colecciones

Una colección — algunas veces llamada un contenedor — es simplemente un objeto que agrupa múltipes elementos en una unidad única. Las colecciones son usadas para almacenar, recuperar, manipular, y comunicar datos agregados. Típicamente, representan elementos de datos que forman un grupo natural, como una mano de póker (una colección de cartas), una carpeta de correo (una colección de letras), o un directorio de teléfono (un mapeo de nombres a números de teléfonos). Si ha usado el lenguaje de programación Java — o casi cualquier otro lenguaje de programación — ya está familiarizado con las colecciones.

¿Qué es un Marco de Trabajo de Colecciones?

Un marco de trabajo de colecciones es una arquitectura unificada para representar y manipular colecciones. Todos los marcos de trabajo de colecciones contienen lo siguiente:

  • Interfaces: Estos son tipos de datos abstractos que representan colecciones. Los interfaces permiten a las colecciones ser manipuladas independientemente de los detalles de su representación. En los lenguajes orientados a objetos, los interfaces generalmente forman una jerarquía.
  • Implementaciones: Estas son las implementaciones concretas de los interfaces de colección. En esencia, estas son estructuras de datos reutilizables.
  • Algoritmos: Estos son los métodos que realizan cálculos útiles, tales como buscar y ordenar, sobre objetos que implementan las interfaces de colección. Se dice que los algoritmos son polimórficos: esto es, el mismo método puede ser utilizado en muchas implementaciones diferentes del interfaz de colección apropiado. En esencia, los algoritmos son funcionalidades reutilizables.

Además del Marco de Trabajo de Colecciones de Java, los ejemplos más conocidos de marcos de colecciones son la Librería Estándar de Plantillas de C ++ (STL o Standard Template Library, en inglés) y la jerarquía de colección de Smalltalk. Históricamente, los marcos de trabajo de colecciones han sido bastantes complejos, lo que les dió una reputación de tener una curva de aprendizaje empinada. Creemos que el Marco de Trabajo de Colecciones de Java rompe con esta tradición, como aprenderá por usted mismo en este capítulo.

Beneficios del Marco de Trabajo de Colecciones de Java

El Marco de Trabajo de Colecciones de Java ofrece los beneficios siguientes:

  • Reduce el esfuerzo de programación: Al proporcionar estructuras de datos y algoritmos útiles, el Marco de Trabajo de Colecciones lo libera para concentrarse en las partes importantes de su programa en lugar de enredarse con los entresijos " de bajo nivel " requeridos para hacerlo funcionar. Al facilitar la interoperabilidad entre IPAs no relacionadas, el Marco de Trabajo de Colecciones de Java le libera de escribir objetos adaptadores o código de conversión para conectar las IPAs.
  • Aumenta la velocidad y la calidad del programa: Este Marco de Trabajo de Colecciones ofrece alto rendimiento, implementaciones de alta calidad de estructuras de datos útiles y algoritmos. Las diversas implementaciones de cada interfaz son intercambiables, así que los programas pueden ser fácilmente ajustados cambiando las implementaciones de colección. Al estár liberado de la monotonía de escribir sus propias estructuras de datos, tendrá más tiempo que dedicar a mejorar la calidad y rendimiento de los programas.
  • Permite la interoperabilidad entre IPA no relacionadas: Las interfaces de colección son la lengua vernácula por la cual las IPA pasan colecciones de un lado a otro. Si mi IPA de administración de red proporciona una colección de nombres de nodo y si su kit de herramientas IGU espera una colección de cabeceras de columna, nuestras IPAS interactuarán sin problemas, a pesar de que fueron escritas de forma independiente.
  • Reduce el esfuerzo para aprender y usar nuevas IPA: Muchas IPAs naturalmente toman colecciones como entrada y las suministran como salida. En el pasado, cada una de esas IPA tenía una pequeña IPA secundaria dedicada a manipular sus colecciones. Había poca consistencia entre estas sub-IPAs de colecciones a medida, de forma que tenía que aprender cada una desde cero, y era fácil cometer errores cuando las usaba. Con la llegada de las interfaces de colección estándar, el problema desapareció.
  • Reduce el esfuerzo para diseñar nuevas IPA: Esta es la otra cara de la ventaja anterior. Los diseñadores e implementadores no tienen que reinventar la rueda cada vez que crean una IPA que se basa en colecciones; sino que pueden usar interfaces estándar de colección.
  • Fomenta la reutilización del software: Las nuevas estructuras de datos que se ajustan a las interfaces de colección estándar son, por naturaleza, reutilizables. Lo mismo ocurre con los nuevos algoritmos que operan en objetos que implementan estas interfaces.
