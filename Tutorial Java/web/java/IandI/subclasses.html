Herencia

En las lecciones precedentes, usted ya ha visto la herencia mencionada varias veces. En el lenguaje Java, las clases pueden ser derivadas de otras clases, de este modo heredar campos y métodos desde estas clases.

Definiciones: una clase que es derivada de otra clase es llamada una subclase (también una clase derivada, clase extendida, o clase hija). La clase desde la cual la subclase es derivada es llamada superclase (también una clase base o una clase padre).

Excepto Object, que no tiene superclase, todas las clases tienen una y sólo una superclase directa (herencia simple). En ausencia de cualquier otra superclase explícita, cada clase es implícitamente una subclase de Object.

Las clases pueden ser derivadas desde clases que están derivadas desde clases que son derivadas de clases, y así, y últimamente derivadas de la clase más alta, Object. Dicha clase se dice que será descendida desde todas las clases en la cadena de herencia que se remonta a Object.

La idea de la herencia es simple pero poderosa: Cuando quiere crear una nueva clase y hay ya una clase que incluye algo del código que usted quiere, puede derivar su nueva clase de la clase existente. Haciendo esto, puede reusar los campos y métodos de la clase existente sin tener que escribirlas (¡y depurarlas!) por sí mismo.

Una subclase hereda todos los miembros (campos, métodos, y clases anidadas) de su superclase. Los constructores no son miembros, asi que no son heredados por las subclases, pero el contructor de la superclase puede ser invocado desde la subclase.

La Jerarquía de Clases de la Plataforma Java

La clase Object , definida en el paquete java.lang, define e implementa comportamiento común para todas las clases — incluyendo las que usted escriba. En la plataforma Java, muchas clases derivan directamente de Object, otras clases derivan de algunas de estas clases, y así, formando una jerarquía de clases.

Todas las Clases en la la Plataforma Java son Descendientes de Object

En el tope de la jerarquía, Object es la más general de todas las clases. Las clases cerca de la parte inferior de la jerarquía ofrecen un comportamiento más especializado.

Un Ejemplo de Herencia

Aquí hay un código de ejemplo para una posible implementación de una clase Bicicleta que fue presentada en la lección Clases y Objetos: 

public class Bicicleta {

    // La clase Bicicleta tiene tres campos
    public int cadencia;
    public int marcha;
    public int velocidad;

    public Bicicleta() {
        super();
    }

    public Bicicleta(int cadenciaInicial, int velocidadInicial, int marchaInicial) {
        marcha = marchaInicial;
        cadencia = cadenciaInicial;
        velocidad = velocidadInicial;
    }

    public void setCadencia(int nuevoValor) {
        cadencia = nuevoValor;
    }

    public void setMarcha(int nuevoValor) {
        marcha = nuevoValor;
    }

    public void frenar(int decremento) {
        velocidad -= decremento;
    }

    public void acelerar(int incremento) {
        velocidad += incremento;
    }

}

Una declaración de clase para una clase BicicletaMontaña que es una subclase de Bicicleta podría parecerse a esto: 

public class BicicletaMontaña extends Bicicleta {
        
    // la subclase BicicletaMontaña añade un campo
    public int alturaSillin;

    // la subclase BicicletaMontaña tienen un constructor
    public BicicletaMontaña(int alturaSillinInicial,
                        int cadenciaInicial,
                        int velocidadInicial,
                        int marchaInicial) {
        super(cadenciaInicial, velocidadInicial, marchaInicial);
        alturaSillin = alturaSillinInicial;
    }   
        
    // la subclase BicicletaMontaña añade un método
    public void setAltura(int nuevoValor) {
        alturaSillin = nuevoValor;
    }   
}

BicicletaMontaña hereda todos los campos y métodos de Bicicleta y añade el campo alturaSillin y un método para establecerlo. Excepto por el constructor, es como si hubiera escrito una nueva clase BicicletaMontaña desde cero, con los cuatro campos y cinco métodos. Sin embargo, usted no ha hecho todo el trabajo. Esto será especialmente valioso si los métodos en la clase Bicicleta fueran complejos y hubieran tomado un tiempo substancial en depurarse.

Qué Puede Hacer en una Subclase

Una subclase hereda todos los miembros públicos y protegidos de su padre, no importa en qué paquete esté la subclase. Si la subclase están en el mismo paquete que su padre, también hereda los miembros privados-para-el-paquete del padre. Puede usar los miembros heredados como tales, reemplazarlos, ocultarlos, o suplementarlos con nuevos miembros:

  • Los campos heredados pueden ser usados directamente, justo como cualquier otro campo.
  • Puede declarar un campo en la subclase con el mismo nombre que el que hay en la superclase, así ocultándolo (no recomendado).
  • Puede declarar nuevos cmapos en la subclase que no están en la superclase.
  • Los métodos heredados pueden ser usados directamente como tales.
  • Puede escribir un nuevo método de instancia en la subclase que tenga la misma firma que el de la superclase, y sobreescribirlo.
  • Puede escribir un nuevo método estático en la subclase que tiene la misma firma que el de la superclase, y ocultarlo.
  • Puede declarar nuevos métodos en la subclase que no están en la superclase.
  • Puede escribir un constructor de la subclase que invoque al constructor de la superclse, o implícitamente o usando la palabra clave super.

Las siguientes secciones en esta lección expandirñan estos temas.

Miembros Privados en una Superclase

Una subclase no hereda los miembros privados de su clase padre. Sin embargo, si la superclase tiene métodos públicos o protegidos para acceder a sus campos privados, éstos pueden ser también usados por la subclase.

Una clase anidada tiene acceso a todos los miembros privados de su clase envolvente — tanto campos como métodos. Por lo tatno, una clase anidada pública o protegida heredada por una subclase tiene acceso indirecto a todos los miembros privados de la superclase.

Moldeando Objetos

Hemos visto que un objeto es del tipo de datos de la clase desde la cual fue instanciado. Por ejemplo, si escribimos 

public BicicletaMontaña miBici = new BicicletaMontaña();

entonces myBici es del tipo BicicletaMontaña.

BicicletaMontaña es descendente de Bicicleta y Object. Por lo tanto, una BicicletaMontaña es una Bicicleta y es también un Object, y puede ser usada en cualquier lugar donde los objetos Bicicleta o Object son llamados.

Lo inverso no es necesariamente verdad: una Bicicleta puede ser una BicicletaMontaña, pero no lo es necesariamente.. Similarmente, un Object puede ser una Bicicleta o una BicicletaMontaña, pero no lo es necesariamente.

El Moldeo muestra el uso de un objeto de un tipo en lugar de otro tipo, entre los objetos permitidos por la herencia e implementaciones. Por ejemplo, si escribimos 

Object obj = new BicicletaMontaña();

entonces obj es a la vez Object y una BicicletaMontaña (hasta que a obj sea asignado a otro objeto que no sea una BicicletaMontaña). A esto se le llama moldeo implícito.

Si, por el contrario, se escribe 

BicicletaMontaña myBike = obj;

obtendremos un error en tiempo de compilación porque obj no es conocido al compilador para ser una BicicletaMontaña. Sin embargo, podemos decirle al compilador que prometeemos asignar una BicicletaMontaña a obj por moldeo explícito: 

BicicletaMontaña myBici = (BicicletaMontaña)obj;

Este moldeo inserta una comprobación en tiempo de ejecución de que obj es asignado a BicicletaMontaña de forma que el compilador puede de forma segura asumir que obj es una BicicletaMontaña. Si obj no es una BicicletaMontaña en tiempo de ejecución, se lanzará una excepción.

Nota:  Puede hacer un test lógico en tanto el tipo de un objeto particular use el operador instanceof. Esto puede salvarle de un error de ejecución debido a un moldeo inadecuado. Por ejemplo: 
if (obj instanceof BicicletaMontaña) {
    BicicletaMontaña miBici = (BicicletaMontaña)obj;
}

Aquí el operador instanceof verifica que obj referencia a BicicletaMontaña así que podemos hacer el moldeo con conocimiento de que no habrá un lanzamiento de excepción en tiempo de ejecución.
