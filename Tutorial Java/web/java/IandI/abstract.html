Métodos y Clases Abstractos

Una clase abstracta es una clase que es declarada abstracta — puede o no incluir métodos abstractos. Las clases abstractas no pueden ser instanciadas, pero pueden generarse subclases a partir de ellas.

Un método abstracto es un método que es declarado sin una implementación (sin llaves, y seguidos por un punto y coma), como este: 

abstract void moveTo(double deltaX, double deltaY);

Si una clase incluye método abstractos, entonces la clase en sí misma debe ser declarada abstracta, como en: 

public abstract class ObjetoGrafico {
   // declara campos
   // declara métodos abstractos
   abstract void draw();
}

Cuando una clase abstracta es derivada, la subclase usualmente establece implementaciones para todos los métodos abstractos en su clase padre. Sin embargo, si no lo hace, entonces la subclase debe ser también declarada abstracta.

Nota: Los métods en un interfaz (vea la sección Interfaces ) que no son declarados como por defecto o estáticos son implícitamente abstractos, así que el modificador abstract no es usado con los métodos de interfaz. (Pueden ser usados, poer no es necesario).

Clases Abstractas Comparadas con Interfaces

Las clases abstractas son similares a los interfaces. No puede instanciarlas, y pueden contener una mezcla de métodos declarados con o sin implementación. Sin embargo, con las clases abstractas, puede declarar campos que no son estáticos y finales, y definir métodos concretos que sean públicos, protegidos o privados. Con las interfaces, todos los campos son automáticamente públicos, estáticos y finales, y todos los métodos que declare o defina (como los métodos por defecto) son públicos. Además, puede extender solo una clase, sea o no abstracta, mientras que se pueden implementar cualquier número de interfaces.

¿Qué debería usar, clases abstractas o interfaces?

  • Considere usar clases abstractas si alguna de estas sentencias se aplican a su situación:
    • Quiere compartir código entre varias clases fuertemente relacionadas.
    • Espera que las clases que extienden su clase abstracta tengan muchos métodos comunes o campos, o que requieren modificadores de acceso distintos a public (tales como protected y private).
    • Quiere declarar campos no estáticos o no finales. Esto le habilita a definir métodos que puedan acceder y modificar el estado del objeto al cual ellos pertenecen.
  • Considere usar interfaces si alguna de estas sentencias aplican a su situación:
    • Espera que clases sin relación implementarán su interfaz. Por ejemplo, las interfaces Comparable y Cloneable son implementadas por muchas clases sin relación.
    • Quiere especificar el comportamiento de un tipo de dato particular, pero no está preocupado sobre quién implementa su comportamiento.
    • Quiere tomar ventaja de la herencia múltiple de tipo.

Un ejemplo de una clase abstracta en el KDJ es AbstractMap , que es parte del Marco de Trabajo Colecciones. Sus subclases (que incluyen HashMap, TreeMap, y ConcurrentHashMap) comparten muchso métodos (incluyendo get, put, isEmpty, containsKey, y containsValue) que AbstractMap define.

Un ejemplo de una clase en el KDJ que implementa varias interfaces es HashMap , la cual implementa las interfaces Serializable, Cloneable, y Map<K, V>. Leyendo esta lista de interfaces, puede inferir que una instancia de HashMap (independientemente del desarrollador o compañía que implemente la clase) puede ser clonada, es serializable (lo que significa que puede ser convertida en un flujo de bytes; vea la sección Objetos Serializables ), y tiene la funcionalidad de un mapa. Además, el interfaz Map<K, V> ha sido mejorado con muchos métodos por defecto tales como merge y forEach que las clases más viejas que implementaron esta interfaz no tienen que definir.

Note que muchas librerías de software usan tanto clases abstractas como interfaces; la clase HashMap implementa varias interfaces y también extiende la clase abstracta AbstractMap.

Un Ejemplo de Clase Abstracta

En una aplicación de dibujo orientada-a-objeto, puede dibujar círculos, rectángulos, líneas, curvas de Bezier, y muchos otros objetos gráficos. Estos objetos tienen todos ciertos estados (por ejemplo: posición, orientación, color de línea, color de relleno) y comportamientos (por ejemplo: mueveA, rota, cambia el tamaño, dibuja) en común. Algunos de estos estados y comportamientos son los mismos para todos los objetos gráficos (por ejemplo: posición, color de relleno, y mueveA). Otros requieren diferentes implementaciones (por ejemplo, cambia el tamaño o dibuja). Todo los ObjetoGraficos deben ser capaces de dibujarse o cambiarse de tamaño; sólo difieren en como lo hacen. Ésta es una perfecta situación para una superclase abstracta. Puede tomar ventaja de las similaridades y declarar que todos los objetos gráficos hereden del mismo objeto padre abstracto (por ejemplo, ObjetoGrafico) como es mostrado en la siguiente figura.

Las Clases Rectangulo, Linea, Bezier, y Circulo Heredan de ObjetoGrafico

Primero, declara una clase abstracta, ObjetoGrafico, para establecer variables miembro y métodos que son completamente compartidos por todas las subclases, tales como la posición actual y el método moverA. ObjetoGrafico también declara métodos abstractos para métodos, tales como dibuja o cambiaTamaño, que necesitan ser implementados por todas las subclases pero ser implementados de diferentes maneras. La clase ObjetoGrafico puede parecer algo como esto: 

abstract class ObjetoGrafico {
    int x, y;
    ...
    void mueveA(int newX, int newY) {
        ...
    }
    abstract void dibuja();
    abstract void cambiaTamaño();
}

Cada subclase no abstracta de ObjetoGrafico, tal como Circulo y Rectangulo, deben establecer implementaciones para los métodos dibuja y cambiaTamaño: 

class Circulo extends ObjetoGrafico {
    void dibuja() {
        ...
    }
    void cambiaTamaño() {
        ...
    }
}
class Rectangulo extends ObjetoGrafico {
    void dibuja() {
        ...
    }
    void cambiaTamaño() {
        ...
    }
}

Cuando una Clase Abstracta Implementa un Interfaz

En la sección sobre Interfaces , se hizo notar que una clase que implementa un interfaz debe implementar todos los métodos del interfaz. Es posible, sin embargo, definir una clase que no implemente todos los métodos de la interfaz, siempre que la clase sea declarada abstracta. Por ejemplo, 

abstract class X implements Y {
  // implementa todo menos un método de Y
}

class XX extends X {
  // implementa el método que queda en Y
}

En este caso, la clase X debe ser abstracta porque no implementa completamente Y, pero la clase XX lo hace, en realidad, implementa Y.

Miembros de Clase

Una clase abstracta puede tener campos estáticos y métodos estáticos. Puede usar estos miembros estáticos con una referencia de clase (por ejemplo, ClaseAbstracta.metodoEstatico()) como si fuera con cualquier otra clase.
