Captura de Comodines y Métodos de Ayuda

En algunos casos, el compilador infiere el tipo de un comodin. Por ejemplo, una lissta puede ser definida como List<?> pero, cuando evalúa una expresión, el compilador infiere un tipo particular del código. Este escenario es conocido como captura de comodín.

En su mayor parte, no necesita preocuparse por la captura de comodín, excepto que vea un mensaje de error que contemga la frase "capture of".

El ejemplo ComodinError produce un error de captura cuando se compiló: 

import java.util.List;


public class ComodinError {

    void foo(List<?> i) {
        i.set(0, i.get(0));
    }
}

En este ejemplo, el compilador procesa el parámetro de entrada i para ser del tipo Object. Cuando se invoca el método foo List.set(int, E) , el compilador no es capaz de confirmar el tipo de objeto que está siendo insertado en la lista, y se produce un error. Cuando este tipo de error ocurre típicamente significa que el compilador cree que usted está asignando el tipo equivocado a una variable. Los genéricos fueron añadidos al lenguaje Java por ésta razón — para hacer cumplir la seguridad de tipos en tiempo de compilación.

El ejemplo ComodinError genera el siguiente error cuando es compilado con la inmplementación javac del KDJ 7 de Oracle: 

ComodinError.java:6: error: method set in interface List<E> cannot be applied to given types;
    i.set(0, i.get(0));
     ^
  required: int,CAP#1
  found: int,Object
  reason: actual argument Object cannot be converted to CAP#1 by method invocation conversion
  where E is a type-variable:
    E extends Object declared in interface List
  where CAP#1 is a fresh type-variable:
    CAP#1 extends Object from capture of ?
1 error

En este ejemplo, el código está intentando realizar una operación segura, asi que, ¿cómo se puede evitar el error del compilador? Puede arreglarlo escribiendo un método de ayuda privado que captura el comodín. En este caso, puede solucionar el problema creando el método privado de ayuda, fooHelper, como es mostrado en ComodinArreglado : 

public class ComodinArreglado {

    void foo(List<?> i) {
        fooHelper(i);
    }


    // Método de ayuda creado para que el comodín pueda ser capturado
    // a través de la inferencia de tipos.
    private <T> void fooHelper(List<T> l) {
        l.set(0, l.get(0));
    }

}

Gracias al método de ayuda, el compilador usa la inferencia para determinar que T es CAP#1, la variable de captura, en la invocación. El ejemplo ahora compila con éxito.

Por convención, los métodos de ayuda son generalmente nombrados metodoNombreOriginalHelper.

Ahora considere un ejemplo más complejo, ComodinErrorMalo : 

import java.util.List;

public class ComodinErrorMalo {

    void swapFirst(List<? extends Number> l1, List<? extends Number> l2) {
      Number temp = l1.get(0);
      l1.set(0, l2.get(0)); // se espero un CAP#1 extends Number,
                            // se obtiene CAP#2 extends Number;
                            // mismo limite, pero diferentes tipos
      l2.set(0, temp);	    // se espero un CAP#1 extends Number,
                            // se obtiene un Number
    }
}

En este ejemplo, el código está intentando una operación no segura. Por ejemplo, considere la siguiente invocación del método swapFirst: 

List<Integer> li = Arrays.asList(1, 2, 3);
List<Double>  ld = Arrays.asList(10.10, 20.20, 30.30);
swapFirst(li, ld);

Mientras ambas List<Integer> y List<Double> cumplen los criterios de List<? extends Number>, es claramente incorrecto tomar un item de una lista de valores Integer e intentar colocarlos en una lista de valores Double.
