Directrices para el Uso de Comodines

Uno de los aspectos más confusos cuando aprende a programar con genéricos es determinar cuándo usar un comodín acotado superiormente y cuándo usar un comodín acotado inferiormente. Esta página establece algunas directrices para seguir cuando diseña su código.

Para los propósitos de esta discusión, es útil pensar de las variables estableciendo una de dos funciones:

Una Variable "In"
Una variable "in" sirve datos al código. Imagine un método de copia con dos argumentos: copy(src, dest). El argumento src establece los datos a ser copiados, así que es un parámetro "in".
Una Variable "Out"
Una variable "out" contiene datos para usar en otros lugares. En el ejemplo de copy, copy(src, dest), el argumento dest acepta datos, así que es el parámetro "out".

Por supuesto, algunas variables son usadas tanto para objetivos "in" como "out" — este escenario también se aborda con las directrices.

Puede usar el principio "in" y "out" cuando decide si usar un comodín y qué tipo de comodín es apropiado. La siguiente lista establece las directrices a seguir:

Directrices para comodines: 
  • Una variable "in" es definida con un comodín limitado superiormente, usando la palabra clave extends.
  • Una variable "out" es definida con un comodín limitado inferiormente, usando la palabra clave super.
  • En el caso donde la variable "in" puede ser accedida usando métodos definidos en la clase Object, use un comodín no acotado.
  • En el caso donde el código necesita acceder a la variable tanto como una variable "in" como una variable "out, no use un comodín.

Estas directrices no se aplican al tipo de retorno del método. Usar un comodín como tipo de retorno debería ser evitado porque fuerza a los programadores que usan el código a hacer frente a los comodines.

Una lista definida por List<? extends ...> puede ser informalmente considera como de sólo-lectura, pero eso no es una garantía estricta. Suponga que tiene las dos clases siguientes: 

class NumeroNatural {

    private int i;

    public NumeroNatural(int i) { this.i = i; }
    // ...
}

class NumeroPar extends NumeroNatural {

    public NumeroPar(int i) { super(i); }
    // ...
}

Considere el siguiente código: 

List<NumeroPar> le = new ArrayList<>();
List<? extends NumeroNatural> ln = le;
ln.add(new NumeroNatural(35));  // error en tiempo de compilación

Porque List<EvenNumber> es un subtipo de List<? extends NaturalNumber>, puede asignar le a ln. Pero no puede usar ln para añadir un número natural a la lista de los números paraes. Las siguientes operacioens sobre la lista son posibles:

  • Puede añadir null.
  • Puede invocar clear.
  • Puede obtener el iterador e invocar remove.
  • Puede capturar el comodín y escribir elementos que ha leído de la lista.

Puede ver que la lista definida por List<? extends NaturalNumber> no es de sólo-lectura en el sentido estricto de la palabra, pero podría pensar de ello de esa forma porque no puede almacenar un nuevo elemento o cambiar un elemento existente en la lita.
