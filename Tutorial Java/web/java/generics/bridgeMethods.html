Efectos del Borrado de Tipos y Métodos Puente

Algunas veces el borrado de tipos causa una situación que usted no pudo haber anticipado. El siguiente ejemplo muestra cómo esto puede ocurrir. El ejemplo (descrito en Métodos Puente ) muestra como un compilador algunas veces crea un método sintético, llamado un método puente, como parte del proceso de borrado de tipos.

Dadas las dos siguientes clases: 

public class Node<T> {

    public T data;

    public Node(T data) { this.data = data; }

    public void setData(T data) {
        System.out.println("Node.setData");
        this.data = data;
    }
}

public class MyNode extends Node<Integer> {
    public MyNode(Integer data) { super(data); }

    public void setData(Integer data) {
        System.out.println("MyNode.setData");
        super.setData(data);
    }
}

Considere el siguiente código: 

MyNode mn = new MyNode(5);
Node n = mn;            // Un tipo crudo - el compilador lanzará una advertencia de no comprobado
n.setData("Hello");     
Integer x = mn.data;    // Causa que se lance una ClassCastException.

Después del borrado del tipo, este código se convierte en : 

MyNode mn = new MyNode(5);
Node n = (MyNode)mn;         //  Un tipo crudo - el compilador lanzará una advertencia de no comprobado
n.setData("Hello");
Integer x = (String)mn.data; // Causa que se lance una ClassCastException.

Aquí está lo que ocurre cuando el código es ejecutado:

  • n.setData("Hello"); causa que el método setData(Object) sea ejecutado sobre el objeto de la clase MyNode. (La clase MyNode heredó setData(Object) de Node.)
  • En el cuerpo de setData(Object), el campo de datos del objeto referenciado por n es asignado a String.
  • El campo de datos del mismo objeto, referenciado vía mn, puede ser accedido y se espera que sea un integer (desde que mn es un MyNode el cual es un Node<Integer>.
  • Intentar asignar una String a un Integer causa una ClassCastException desde un moldeo insertado en la asignación por un compilador de Java.

Métodos puente

Cuando se compila una clase o interfaz que extiende una clase parametrizada o implementa un interfaz parametrizado, el compilador puede necesitar crear un método sintético, llamado método puente, como parte del proceso de borrado de tipos. Normalmente no necesita preoucparse sobre los métodos puente, pero podría confundirse si uno aparece en un seguimiento de pila.

Después del borrado de tipos, las clases Node y MyNode se convierten en: 

public class Node {

    public Object data;

    public Node(Object data) { this.data = data; }

    public void setData(Object data) {
        System.out.println("Node.setData");
        this.data = data;
    }
}

public class MyNode extends Node {

    public MyNode(Integer data) { super(data); }

    public void setData(Integer data) {
        System.out.println("MyNode.setData");
        super.setData(data);
    }
}

Después del borrado de tipos, las signaturas de los métodos no coinciden. El método Node se convierte en setData(Object) y el método MyNode se convierte en setData(Integer). Por lo tanto, el método setData de MyNode no sobreescribe el método setData de Node.

Para resolver este problema y preservar el polimorfismo de los tipos genéricos después de un borrado de tipos, un compilador de Java genera un método puente para asegurar que la derivación (subtipado) funciona como se esperaba. Para la clase MyNode, el compilador general el siguiente método puente para setData: 

class MyNode extends Node {

    // Método puente generado por el compilador
    //
    public void setData(Object data) {
        setData((Integer) data);
    }

    public void setData(Integer data) {
        System.out.println("MyNode.setData");
        super.setData(data);
    }

    // ...
}

Como puede ver, el método puente, el cual tiene la misma firma de método como el método setData de la clase Node después del borrado de tipo, delega al método original setData.
